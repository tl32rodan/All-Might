## 2. Panoramic snapshot — Agent Skills open spec (agentskills.io)

- **File shape**: a directory named `<skill-name>/` holding `SKILL.md`,
  optionally with `scripts/`, `references/`, `assets/` subdirs.
- **Required frontmatter** (YAML): exactly two fields.
  - `name` — max 64 chars, lowercase letters / numbers / hyphens.
    **Must match the parent directory name** or the host won't load it.
  - `description` — the discovery field. The host loads only this
    plus the name at startup (~100 tokens). The agent decides whether
    to activate the skill based on this string alone.
- **Body**: free-form markdown. Loaded **only on activation**.
  Recommended cap ~5000 tokens.
- **Progressive disclosure (3-tier)**: metadata-at-startup →
  body-on-activation → `scripts/`/`references/`/`assets/`-on-demand.
- **Adopters (2026)**: Claude Code, Codex CLI, Gemini CLI, GitHub
  Copilot (via VS Code agent-skills), Cursor (manual placement),
  OpenCode, Cline, Windsurf.

## 3. Panoramic snapshot — `kepano/obsidian-skills`

- **Pitch**: "Teach your agent to use Markdown, Bases, JSON Canvas,
  and the CLI." A reference *collection* — five skills, no framework
  layer.
- **Skills shipped** (each one directory + `SKILL.md`):
  | Skill | Teaches the agent to |
  |---|---|
  | `obsidian-markdown` | Edit `.md` with wikilinks, embeds, callouts, properties |
  | `obsidian-bases` | Author `.base` files (views, filters, formulas) |
  | `json-canvas` | Edit `.canvas` files (nodes, edges, groups) |
  | `obsidian-cli` | Drive the Obsidian CLI for plugin/theme dev |
  | `defuddle` | Strip page chrome → clean markdown |
- **Installation**: three independent paths.
  - Marketplace: `npx skills add <name>` (project scope).
  - Claude Code: copy to `<vault>/.claude/skills/`.
  - Codex / OpenCode: copy to `~/.codex/skills/` or
    `~/.opencode/skills/obsidian-skills/` (user-global).
- **Design thesis (implicit, derived from skill choice)**: an
  agent should manipulate *user-owned data in user-owned formats*
  (Markdown, JSON Canvas, the Obsidian Base file format). No skill
  teaches the agent to manage an internal SQLite or vector store.
- **No memory, no role, no hook, no lifecycle.** Pure capability
  collection.

## 4. Panoramic snapshot — `open-gsd/get-shit-done-redux`

- **Pitch**: "light-weight meta-prompting, context engineering, and
  spec-driven development system for Claude Code, OpenCode, Gemini
  CLI, Kilo, Codex, Copilot, Cursor, Windsurf, and more."
- **Fixed artifact paths** at project root (not per-personality):
  - `PROJECT.md` — vision
  - `REQUIREMENTS.md` — scope
  - `ROADMAP.md` — phase breakdown
  - `STATE.md` — current position
  - `CONTEXT.md` — per-phase technical decisions
  - `REVIEW.md` — verification results
  - `.planning/config.json` + `.planning/phases/<phase>/FALLOW.json`
- **Six-command lifecycle** (one project-shaped state machine):
  1. `/gsd-new-project` — questions → research → requirements → roadmap
  2. `/gsd-discuss-phase [N]` — capture decisions before planning
  3. `/gsd-plan-phase [N]` — research + plan + verify
  4. `/gsd-execute-phase <N>` — parallel waves
  5. `/gsd-verify-work [N]` — manual acceptance
  6. `/gsd-ship [N]` — open PR
- **Cross-tool installer**: `bin/install.js` translates skill
  frontmatter per host (e.g.
  `convertClaudeToOpencodeFrontmatter`) — the **opposite** of
  All-Might's "one source, both editors read it" symlink strategy.
- **Real implementation signal** (from the archived original):
  hook crashes (#3878), CI flakes (#3869), deprecated-event
  residue in `hooks/gsd-session-state.sh` (#3873) — the bugs of a
  system actually run. Not a doc repo.
- **No role / personality / memory abstraction.** State *is* the
  artifact files; identity is the project. Mono-project,
  mono-purpose.

## 5. Reference scan — `shareAI-lab/learn-claude-code`

Not a competitor — a 20-lesson teaching repo that rebuilds a
Claude-Code-like harness from scratch. Useful as the *consensus
checklist* of mechanisms in a 2026 harness:

> agent loop · tools + permissions · planning (TodoWrite) ·
> subagents · skill loading · context compaction (multi-layer:
> snip / micro / toolResultBudget) · memory subsystems
> (selection / extraction / consolidation) · task graph with
> dependencies · background tasks · cron · agent teams (mailbox
> coordination) · worktree isolation · MCP · hooks (PreToolUse /
> PostToolUse).

What All-Might owns from this list, by deliberate scope:
**memory + skill loading + role identity** (plus optional
`schedule` capability). Everything else — agent loop, tools,
planning, subagents, compaction, MCP, task graph, worktrees — is
**host-owned** (OpenCode / Claude Code). Restating that boundary
explicitly is the value of this scan; it's an anti-scope-creep
artifact, not a feature catalog.

## 6. Four-axis comparison

### a) Design philosophy

| | All-Might | agentskills.io | obsidian-skills | gsd-redux |
|---|---|---|---|---|
| Central abstraction | **role × capability** | `SKILL.md` (capability unit) | bare skill collection | **fixed project-lifecycle state machine** |
| State model | per-personality memory + git mirror | none (format only) | none | fixed artifact files at project root |
| Scope | project | format-level only | project or user-global | project, mono-purpose |
| Identity | user-defined personality | n/a | n/a | the project itself |
| Adoption mode | scaffold + agent-driven `/onboard` | spec adopted by ~8 hosts | manual copy / `npx skills add` | scaffold + scripted phase commands |

**Two findings, both All-Might-internal-state preserving:**
1. **All-Might's `SKILL.md` is already on the open spec.** Verified:
   `core/skill_io.py::write_skill_md` writes `name:` and
   `description:` frontmatter and the helper passes the skill's
   directory name straight into the `name:` field — so the
   "frontmatter `name` must match parent dir" rule is satisfied by
   construction. The "lowercase letters / numbers / hyphens" rule
   holds for every current skill (`onboard`, `one-for-all`,
   `all-for-one`, `sync`, `recover`, `scheduling`, `split`). No
   migration needed; the conformance is structural.
2. **GSD's "fixed artifact paths" pattern is the opposite of
   All-Might's "per-personality data dir" pattern.** GSD encodes one
   shape (greenfield-project workflow) into the filesystem
   permanently. All-Might encodes *user-defined* shapes (one per
   personality). They solve different problems; mixing them would
   surrender All-Might's per-role differentiation for a single
   built-in workflow that few All-Might projects need.

|---|---|---|---|---|

| Per-role identity | ✓ | ✗ | ✗ | ✗ |

### c) Mechanism design — the three orthogonal axes

1. **`SKILL.md` is now a load-bearing format with a published spec.**
   All-Might inherited the shape from OpenCode's plugin sample. The
   spec hardened around the same two-field frontmatter. Implication:
   any host that strictly validates `name` (matches dir, ≤64 chars,
   lowercase/hyphens) will silently drop a skill whose body we
   emit. The structural guarantee in `skill_io.py` covers the
   `name == dir` axis but **does not** assert the character-class
   constraint. A skill named `One-For-All` (capitalised) or
   `share_publish` (underscore) would fail validation today. No
   current skill does, but nothing pins it. See §7 Q3.

2. **GSD's fixed-shape workflow is a *different problem*, not a
   competitor.** Spec-driven dev assumes one greenfield project,
   one author, one shape. All-Might assumes a long-running project
   with multiple specialists ("personalities"). Adopting GSD-style
   `PROJECT.md`/`REQUIREMENTS.md`/etc. at the All-Might root would
   inflate the surface and have no place to put per-personality
   knowledge.

3. **obsidian-skills is a *capability set*, not a framework.** It
   inverts All-Might's "framework provides capabilities" arrow:
   here, capabilities ship as third-party skills the user installs.
   For domain-content (Markdown, Canvas, Bases) this is the right
   shape — those formats aren't anything All-Might's framework
   should know about. The implication is **complementary, not
   competing**: an All-Might user editing an Obsidian vault can
   install `obsidian-skills` *alongside* All-Might without conflict
   (different `.opencode/skills/` entries, no shared state).

### d) Implementation details

- **The skill-frontmatter validator is now an externalised contract.**
  Before agentskills.io, "what's a valid SKILL.md" was tribal
  knowledge per host. Now the rules are written down and hosts will
  diverge on strictness. All-Might's `name == dir` guarantee covers
  the *correctness* axis but not the *character-class* axis — a
  one-line assertion in `skill_io.py` would close that gap
  permanently.
- **GSD ships a per-host frontmatter translator** (`bin/install.js`
  with `convertClaudeToOpencodeFrontmatter`). All-Might solves the
  same problem differently: one canonical `.opencode/` source and
  a `.claude/` mirror via dir-symlinks + hand-mirrored hooks. GSD's
  translator approach is necessary when *each host has its own
  frontmatter dialect*. With agentskills.io stabilising the dialect,
  the translator approach loses its motivation — All-Might's
  symlink-mirror strategy ages better.
- **obsidian-skills exposes three install paths** (npx, manual CC,
  manual codex/OC). The marketplace path (`npx skills add`) is the
  only one All-Might lacks an equivalent for. Today, an All-Might
  personality bundle moves via `allmight share publish/pull` over a
  git remote. A marketplace registry is bigger than "publish a
  bundle" — but if a public registry of personality bundles ever
  emerges, `share pull <name>` resolving against it would slot in
  cleanly. Not a today-job.

## 9. Feature rating — Q2 (the one candidate worth rating)

| Dimension | Assessment |
|---|---|
| Problem reality | **Real but latent.** No skill today violates the rule, but the only thing keeping that true is reviewer vigilance. A skill with a stray underscore or capital would silently drop on a strict host (and silently *not* drop on a lenient one), reproducing the "behaviour depends on which editor I open the project with" bug class CLAUDE.md already warns about. |
| Simplicity cost | **Negligible.** ~5 lines in `core/skill_io.py::write_skill_md` (an `assert re.fullmatch(...)`), plus a one-line negative test (passing an invalid name raises). No surface change, no doc change. |
| Upside if it triggers | Catches the bug at *install time* on the author's machine, where the cost is one error message — instead of *load time* on a downstream user's strict host, where the cost is a silent missing skill. |
| Reversibility | Trivial — single assertion, one commit. |
| Risk of mechanising into noise | None — the assertion fires only on invalid names, never during normal flow. |
| **Verdict** | **Build now. Rating 5/5 for cost-adjusted value.** This is the kind of defensive pin that *prior* passes have repeatedly flagged in retrospect as cheap and missed (cf. 2026-05-22's regression on `.claude/settings.json` hook-script existence). Do the cheap pin before the bug. |

## 10. Lessons captured for next reconnaissance

- **An open standard ratifying your choice changes the question from
  "should we adopt it" to "do we conform to it."** Conformance is
  cheaper than adoption — pin it as soon as the spec hardens. The
  agentskills.io spec was a watch-item for "what's the canonical
  `SKILL.md` shape?" and the answer is: the shape we already write.
  Now pin it (Q2).
- **"Different problem" disposes of more candidate features than
  "bad design" does.** GSD is a well-built system solving a
  *different* problem (greenfield project planning vs. long-running
  multi-specialist work). Calling it out as off-axis is faster than
  arguing it down on merits.
- **Capability collections (obsidian-skills) and capability
  frameworks (All-Might) are not the same animal.** The first
  inverts the dependency arrow: the user installs the skill, the
  framework knows nothing about it. All-Might *should* welcome
  obsidian-skills-style collections — they're cheap third-party
  capability, not surface to absorb. The scope-boundary paragraph
  (Q7) is how to signal this without coding.
- **Reference-only "consensus checklist" repos are an
  anti-scope-creep tool.** `learn-claude-code` doesn't compete; it
  catalogs. The catalog is what lets All-Might say *"we own role +
  memory + skills; the host owns everything else"* without
  hand-waving. Maintain the list explicitly — next time someone
  proposes adding `/plan` or `/compact` or a task graph, the
  catalog answer is "that lives in the host."
