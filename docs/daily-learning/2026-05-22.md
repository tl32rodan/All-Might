## 2. Panoramic snapshot ‚Äî volcengine/OpenViking

- **Pitch**: "abandons the fragmented vector storage model of
  traditional RAG and innovatively adopts a 'file system paradigm'
  to unify the structured organization of memories, resources, and
  skills."
- **Virtual filesystem** (`viking://` URI scheme):
  ```
  viking://
  ‚îú‚îÄ‚îÄ resources/        ‚Üê project docs, repos, web pages
  ‚îú‚îÄ‚îÄ user/             ‚Üê personal preferences, habits
  ‚îÇ   ‚îî‚îÄ‚îÄ memories/
  ‚îÇ       ‚îî‚îÄ‚îÄ preferences/
  ‚îî‚îÄ‚îÄ agent/            ‚Üê skills + memories + instructions
      ‚îú‚îÄ‚îÄ skills/
      ‚îú‚îÄ‚îÄ memories/
      ‚îî‚îÄ‚îÄ instructions/
  ```
- **Three-tier abstraction generated *per directory* on write**:
  - **L0 `.abstract`** ‚Äî one-sentence summary (~100 tokens), for
    retrieval & identification.
  - **L1 `.overview`** ‚Äî core info + usage scenarios (~2k tokens),
    for planning-phase decisions.
  - **L2 full content** ‚Äî original data, loaded only when "absolutely
    necessary."
  - Generation: **at write time** (README does not specify whether
    by LLM call or rule-based; the bug reports about embedder
    behaviour imply an LLM is in the loop somewhere on the write
    path).
- **Recursive directory retrieval** (the README's distinctive claim):
  1. Intent analysis ‚Üí multiple retrieval conditions
  2. Vector search ‚Üí locate high-score *directory* (not file)
  3. Secondary search within that directory
  4. Recurse into subdirectories
  5. Aggregate
- **No personality / role concept.** Context partitioned by *type*
  (resources / user / agent), not by *identity*. Single-agent shape.
- **No per-project isolation today** (issue #1942: all memories
  consolidated when run under OpenCode).
- **"Visualized retrieval trajectories"** ‚Äî preserves the directory
  drill-down path for debuggability.

## 3. Panoramic snapshot ‚Äî thedotmack/claude-mem (spot)

- Agent-level (not project-level) persistent context across sessions.
- Storage: **SQLite + Chroma vector DB + FTS5 full-text index** ‚Äî
  exactly the heavy stack All-Might's CLAUDE.md non-goals explicitly
  reject ("No SQLite memory store").
- Hooks: SessionStart, UserPromptSubmit, PostToolUse, Stop,
  SessionEnd ‚Äî same hook surface All-Might uses.
- Runtime: Bun-managed worker on port 37777 + web viewer UI.
- "Modes" (e.g. `code--zh`) are language / workflow flavours, not
  identity partitioning. **No personality concept.**

## 4. Four-axis comparison

### a) Design philosophy

| | All-Might | OpenViking | claude-mem |
|---|---|---|---|
| Primary organization axis | **Role** (personality) | **Type** (resources / user / agent) | Session timeline |
| Multi-agent shape | First-class (`personalities/<name>/...`) | None ‚Äî single-agent | None |
| Layered context | L1 MEMORY.md / L2 understanding TOC / L3 journal+SMAK RAG | L0 `.abstract` / L1 `.overview` / L2 full ‚Äî **per-directory** | Index ‚Üí timeline ‚Üí details (progressive disclosure) |
| Summary generation | L1+L2 hand-written / agent-authored ; L3 indexed by SMAK | **Auto-generated per directory on write** | Auto-compressed by AI on Stop |
| Address scheme | filesystem path + `ROUTING_PREAMBLE` | `viking://` URI | SQLite rows + Chroma vectors |

**Insight**: OpenViking and claude-mem both converge on
**auto-generated summaries on the write path**. All-Might keeps L1 +
L2 human-/agent-authored and only auto-indexes L3. Two different
defensible positions ‚Äî see ¬ß6 Q1.

### b) Feature coverage

| Concern | All-Might | OpenViking | claude-mem |
|---|---|---|---|
| Per-project isolation | ‚úì (project root) | **‚úó (open bug #1942)** | ‚úó (agent-level) |
| Per-role isolation | ‚úì | ‚úó | ‚úó |
| Cross-project share | ‚úì (`/one-for-all` + `/all-for-one` + git bundles, PII review) | ‚úó (no public mechanism) | ‚úó |
| Local-git memory mirror & recovery | ‚úì (`.allmight/memory-history/.git`) | ‚úó (SQL row history at best) | SQLite history |
| Heavy infra | None (YAML + git + filesystem) | Vector DB + LLM-generated summaries on hot path | SQLite + Chroma + FTS5 + HTTP server |
| Multi-IDE | OpenCode + Claude Code | OpenCode + Claude Code plugin examples | Claude Code + OpenClaw + Codex + others |
| Hierarchical retrieval | ‚úó (L3 flat vector) | ‚úì (recursive directory drill-down) | ‚úó (flat hybrid) |

### c) Mechanism design

- **Per-directory `.overview`** (OpenViking) is the genuinely novel
  mechanism. All-Might's `understanding/_index.md` is the same
  *intent* (a TOC over a personality's L2), but it's one file per
  personality and hand-curated. OpenViking writes a `.overview` for
  *every* directory and auto-generates it. Scales without manual
  curation ‚Äî at the cost of an LLM call on the write path.
- **Recursive directory retrieval** depends on having a `.overview`
  per directory. Without that artefact the algorithm degrades to
  flat vector search. It is **not** something All-Might can adopt
  alone; it's pair-shaped with the `.overview` mechanism above.
- **No per-project isolation** in OpenViking is reported by its own
  users as a bug. All-Might's project-scoped layout (`personalities/`
  lives inside the project) is **vindicated** rather than
  challenged.
- **claude-mem's SQLite+Chroma+FTS5+HTTP-server** stack is exactly
  the kind of infrastructure All-Might's CLAUDE.md non-goals
  explicitly reject. It does work ‚Äî 77k stars say so ‚Äî but it
  trades human-readable, diff-able, recoverable memory for
  query speed. All-Might's choice (YAML + git mirror) is
  honest about the trade-off.

### d) Implementation details

- **OpenViking's "automatic processing on write"** has a real cost,
  visible in their bug tracker: #2132 (untruncated content ‚Üí 413),
  #2154 (embedder param ignored), #2118 (index never persisted),
  #1972 (large patch stalls). Every write-path summarization carries
  a long failure-mode tail. All-Might's "indexer = SMAK out-of-band,
  triggered async by marker file" design avoids this entire failure
  class.
- The 3-tier shape (L0/L1/L2) is **isomorphic to All-Might's L1/L2/L3**
  but at different scopes:
  - OpenViking's L0/L1/L2 is *per directory* and *contentwise*.
  - All-Might's L1/L2/L3 is *per personality* and *by access pattern*.
  Convergent evolution toward 3 tiers.

## 5. Carry-over follow-up ‚Äî `agents.md` alignment (Q4 from 2026-05-18)

**Spec status (as of May 2026)**:

- Single-file standard at repo root, "a README for agents". Plain
  markdown, no enforced frontmatter schema.
- Donated to **Agentic AI Foundation** (Linux Foundation directed
  fund) in Dec 2025, alongside Anthropic's MCP and Block's Goose.
- **Claude Code still does NOT natively read AGENTS.md** as of April
  2026 (active feature request with thousands of upvotes).
- Recommended cross-tool pattern: write `AGENTS.md`, symlink
  `CLAUDE.md ‚Üí AGENTS.md`. "One file on disk, two filenames, zero
  drift."
- Research caveat (Gloaguen et al. 2026, 138 repos): LLM-generated
  context files **reduce** agent task success rate and add >20%
  inference cost. Developer-written context files give only +4%
  improvement, and only when minimal and precise.

**All-Might status**:

- Root `AGENTS.md` exists (composed from per-personality `ROLE.md`
  by `compose_agents_md`).
- Root `CLAUDE.md` is an `@`-import shim that points at both
  `AGENTS.md` *and* `MEMORY.md`. This is **stronger than the
  recommended symlink** ‚Äî Claude Code receives the project map +
  default-personality hint, not just the agent brief.
- **Verdict**: aligned with intent of the standard, with a deliberate
  improvement (shim over symlink). No code change required. Action:
  add a one-paragraph rationale to `README.md` so the divergence is
  on record. (Estimated cost: 1 small README edit, no code change.)

The Gloaguen caveat is a watch item for the `compose_agents_md`
mechanism. Composed AGENTS.md stitches per-personality ROLE.md (some
of which is agent-authored). If we ever observe success-rate
regressions correlated with AGENTS.md length, the response is to
trim ‚Äî not to add structure.

## 7. Investment verdict

**No new feature meets the simplicity bar today.** Reasoning:

1. The only genuinely novel mechanism in today's reconnaissance ‚Äî
   per-directory auto-generated `.overview` (OpenViking) ‚Äî is
   already covered by an existing All-Might non-goal *and* shows
   its own failure-mode tail in OpenViking's bug tracker. The honest
   action is to wait for the threshold to be crossed.
2. The competing memory products (`claude-mem`, `mem0`, `letta`,
   `honcho`) all converge on heavy infrastructure (SQLite, vector
   DBs, HTTP services). All-Might's lighter, human-readable choice
   remains distinctive ‚Äî and validated by the trade-offs the
   heavyweights make.
3. Per-project + per-role isolation, which All-Might has from day
   one, is now an *open bug* for OpenViking (#1942). The design
   gap is widening in All-Might's favour, not narrowing.
4. The `agents.md` carry-over closes with a 1-paragraph README edit;
   no compose-mechanism change required.

## 8. Lessons captured for next reconnaissance

- When a competitor's bug tracker shows a failure-mode tail tied to
  a feature you don't have, that *absence* is design strength ‚Äî
  log it explicitly and stop questioning the absence.
- Convergent evolution toward 3-tier context (All-Might L1/L2/L3,
  OpenViking L0/L1/L2, claude-mem index/timeline/details) is real.
  The axis matters: *per-role + by access pattern* (All-Might) vs
  *per-directory + by content* (OpenViking) vs *per-session +
  chronological* (claude-mem). All three solve different problems;
  none is universally right.
- Heavy auto-summarization on the write path has a long failure
  tail. All-Might's marker-file-async-spawn closure (for L3 SMAK
  ingest) is the right primitive ‚Äî keep it.
- When the recommended cross-tool pattern (`symlink CLAUDE.md ‚Üí
  AGENTS.md`) loses information that an `@`-import shim preserves
  (MEMORY.md too), the small-divergence-with-stronger-property is
  worth the documentation cost.
