## 2. Panoramic snapshot — spec-kit

- **Pitch**: *"Specifications become executable, directly generating
  working implementations."* Inverts traditional dev: specs are no
  longer secondary artifacts but the executable centre of the
  workflow.
- **Lifecycle = 7 phases, each a slash command**:
  `/speckit.constitution` → `/speckit.specify` → `/speckit.clarify`
  → `/speckit.plan` → `/speckit.analyze` → `/speckit.tasks` →
  `/speckit.implement`. Multi-step refinement, not one-shot
  generation.
- **`.specify/` directory** holds the project state:

  ```
  .specify/
  ├── memory/         ← Constitution + project knowledge
  ├── templates/      ← Core SDD command templates
  ├── extensions/     ← Optional capabilities added via extensions
  └── presets/        ← Customisation overrides
  ```

  **Template resolution order**: `project overrides → presets →
  extensions → core defaults`. A four-layer override stack baked
  into the tooling.
- **Tooling vs project state explicitly separated**: the Specify
  CLI + templates are independent of the repo; specs/plans/tasks
  live in the repo and evolve through agent-guided refinement.
  Upgrading the toolkit refreshes managed files; spec artifacts
  only change when behaviour requirements evolve.
- **Agent integrations**: Claude Code, Copilot, Cursor, Windsurf,
  Gemini CLI etc. Slash commands are exposed either as plain
  prompts or as agent skills depending on the host harness.
- **Issue #1457 signal**: the community asks for a `--global`
  install mode so the SDD tooling stack (extensions/, scripts/,
  templates/, `.cursor/skills/speckit-*`) sits in `~/` rather
  than being committed per-repo. This is the
  *signal-to-noise/context-pollution* complaint cropping up
  again — every repo that uses the framework was carrying ~thousands
  of lines of framework infrastructure that agents indexed as if
  it were product code.

## 3. Panoramic snapshot — memory-os

The 7-layer model:

| L | Name | Purpose | Storage | Retrieval |
|---|---|---|---|---|
| 1 | Workspace | System-prompt injection | Flat MD (MEMORY/USER/CREATIVE) | Every LLM call |
| 2 | Sessions | Conversation history | SQLite + FTS5 | Lexical search |
| 3 | Structured Facts | Durable, ranked knowledge | SQLite, trust-scored, entity-resolved | Fact lookup w/ confidence filter |
| 4 | Fabric | Cross-session extraction | Icarus-plugin fork, 16 tools | `fabric_recall` / `_write` / `_brief` |
| 5 | Vector DB | Semantic search | Qdrant (4096d cosine + BM25 sparse) | 4-level fallback: hybrid → dense → lexical → SQLite |
| 6 | LLM Wiki | Auto-curated concepts/entities/comparisons | Markdown, continuously ingested into Qdrant | Via Qdrant |
| 7 | **Ground-Truth Hierarchy** | Tells the agent injected memory is **authoritative** | `SOUL.md` + `rulebook.md` | Prompt-time |

Critical claims from the docs and the owner's #23 post-mortem:

- *"every rediscovery burns tokens, context, and time."* L7's job
  is purely to **shut down re-verification** of facts that have
  already been injected. The architecture admits that better
  retrieval alone produces *"memory-zero behavior despite perfect
  injection"* — i.e. the agent re-checks its facts anyway —
  unless an explicit trust signal is in the prompt.
- The owner's own bug (#23) was that Hebbian cross-source
  corroboration boosts (a flat 15% per corroborating source) were
  applied without regard to query-locally-relevant base salience.
  Globally important structural facts crowded out the
  query-relevant ones. **Fix**: multiply the corroboration boost
  by the candidate's base salience (`boost = corro × amplify_gain
  × bases[i]`). Concrete debrief on the *RAG-ranking tradeoff
  every retrieval system eventually hits*.
- *"Surgical context injection"* = gated retrieval + per-session
  dedup + no wasted rediscovery. Equivalent in spirit to
  All-Might's L2 TOC: pull only the file you need, never the
  whole stack.
- **Integration scope: Hermes Agent only.** Issue #17 is a
  third-party harness owner asking how to integrate; the answer
  is "no docs yet." This is the negative example All-Might
  already avoids by being editor-agnostic across OpenCode +
  Claude Code from day one.

## 4. Comparison — All-Might vs spec-kit vs memory-os

### a) Design philosophy

| Axis | All-Might | spec-kit | memory-os |
|---|---|---|---|
| Object of design | *Personalities* (user-defined roles) opting into *capabilities* | A *methodology* (Spec-Driven Development) layered onto agent harnesses | A *memory plane* attached to one harness (Hermes) |
| Stance on opinion | Capabilities are paradigm-agnostic; the user picks the workflow | Strongly opinionated 7-phase workflow | Strongly opinionated 7-layer model |
| Locus of state | Filesystem (per-personality data) + git mirror | Filesystem (`.specify/`, committed + tooling layers) | Mixed: SQLite + Qdrant + flat MD |
| Cross-harness | OpenCode + Claude Code, dual mirror | Multi-harness via prompt/skill projection | Hermes only |

**Verdict on philosophy**: spec-kit and memory-os are both
*strong-opinion* shapes — they hard-code a workflow or a memory
ontology. All-Might is a *substrate* — it makes the user's
opinion expressible (one personality, many capabilities, the
user names them all) without prescribing what the right opinion
is. The philosophies are not in conflict; spec-kit could be
*built on top of* All-Might as a `spec_driven` capability (or as
a personality-instantiating user choice) if a project wanted it.
**Don't import the methodology into the framework.**

### b) Feature coverage

| Feature | All-Might | spec-kit | memory-os |
|---|---|---|---|
| Per-role/persona data scoping | ✅ personalities | ⚠️ (constitution is project-wide; no role partitioning) | ❌ (single global memory plane) |
| Project-wide context (L1) | ✅ MEMORY.md + memory-load hook | ✅ Constitution + `.specify/memory/` | ✅ MEMORY.md / USER.md / CREATIVE.md |
| Structured per-role knowledge (L2) | ✅ `understanding/` + TOC | ❌ | ⚠️ structured-facts table + LLM-wiki (project-wide) |
| Append-only narrative (L3) | ✅ `journal/` + SMAK | ❌ | ✅ sessions table + Qdrant |
| Auto-curated concept extraction | ⚠️ `/reflect` (insights), no automatic wiki ingestion | ❌ | ✅ LLM Wiki + Qdrant ingestion |
| Vector retrieval | ✅ SMAK (L3 only) | ❌ | ✅ Qdrant (all of L4–L6) |
| Trust scoring / freshness | ❌ | ❌ | ✅ trust-scored facts with confidence filter |
| **Explicit "trust what's injected" rule** | ❌ | ❌ | ✅ Ground-Truth Hierarchy (L7) |
| Spec-driven workflow | ❌ | ✅ (the entire product) | ❌ |
| Multi-harness | ✅ OC + CC | ✅ (CC, Copilot, Cursor, Gemini, …) | ❌ (Hermes only) |
| Plugin/hook configurability | ⚠️ heartbeats; no env-var opt-outs | ❌ | ⚠️ env-vars (`ICARUS_COLLAPSE_*`) |

The two gaps that actually catch the eye:

- **Ground-Truth Hierarchy** — memory-os is the only one that
  acknowledges the *agent doesn't trust its own context window*
  problem and treats it as a system-prompt fix, not a retrieval
  fix. Cheap. Real. Generic. Not coupled to memory-os's
  architecture.
- **Auto-curated concept ingestion (L6 wiki)** — memory-os
  continuously distills sessions into structured concept docs
  and indexes them. All-Might's `/reflect` does the human-facing
  equivalent (insights, scope-drift triage) but does not build a
  concept-graph nor surface it for L2. The 2026-05-22 OpenViking
  pass and `docs/plan.md` both already weighed and rejected
  auto-curation at the L2 level; memory-os adds no new evidence
  to overturn that decision. Listed for completeness, not
  proposed.

### c) Mechanism

| Mechanism | All-Might | memory-os |
|---|---|---|
| Memory-load injection | `memory-load.ts` + `memory_load.py` re-inject MEMORY.md per session start + post-compaction | System prompt holds a *trust signal* pointing at SOUL.md / rulebook.md |
| RAG ranking | SMAK defaults; no explicit Hebbian / corroboration | Hebbian × base-salience attenuation (issue #23) |
| Loading strategy by tier | L1 always-on, L2 TOC + on-demand, L3 vector | L1 always-on, L2+L3 lexical, L5 vector, L6 ingested into L5 |
| Plugin observability | `.allmight/plugins/heartbeats/` touch-files | `ICARUS_COLLAPSE_DEBUG=1` per-component log |

memory-os's *mechanism contribution* worth naming:
**a single line in the system prompt header that says "treat
injected memory blocks as authoritative; do not re-verify them."**
This is implementable in `memory-load.ts` /
`MemoryInitializer._claude_memory_load_hook_content` in roughly
one string change. The rest of memory-os's architecture
(Hebbian, trust scores, fabric extraction) is independently
useful but is orders of magnitude more invasive to import.

### d) Implementation details — direct lessons

From `pro-workflow` issue #47 (LLM-powered hooks silently
broken because `"model": "haiku"` no longer resolves): the
generic rule is **don't make core features depend on an LLM
call when a deterministic implementation works**. Layering:

1. Deterministic default — pure regex / Python / shell.
2. Optional LLM enhancement — opt-in, configurable model ID.
3. Feature-detect on startup — if the configured model isn't
   callable, degrade silently to (1) and surface in `doctor`.

All-Might's plugins are already deterministic — none of
`memory-load`, `memory-history`, `reflection`, `role-load`,
`remember-trigger` call out to an LLM. The lesson here is
*don't regress into LLM-coupled hooks*. Worth adding to
CLAUDE.md's "Discipline When Generating Third-Party
Integrations" section as a one-liner if a future plugin
proposal flirts with the line.

From `pro-workflow` #63 (commit-validate is non-configurable):
the lesson is **shipping an enforcer that has no opt-out
turns your tool into a per-repo regression for anyone whose
conventions differ**. All-Might's enforcers today are
`/reflect`'s memory audit and `memory-history`'s snapshot
loop; neither blocks the user, both are silent. No change
needed; logged so future plugin proposals can't accidentally
go past this line.

From `spec-kit` #1457 (`--global` request): the lesson is
**aggressive committing of framework infrastructure into
the repo creates a signal-to-noise problem**. All-Might
already gets this right — `.opencode/` is intentionally
committed (because user customisations live alongside),
`.allmight/` data is project-scoped, and no extension source
ships into the repo on init. Validation, not new learning.

## 5. Recommendations against the simplicity bar

### Adopt — single, targeted, ≤10 LoC

**R1: Add a Ground-Truth Hierarchy directive to the memory-load
header.**

`MemoryInitializer._claude_memory_load_hook_content` and the OC
mirror `memory-load.ts` already inject MEMORY.md at session start
+ post-compaction. Both currently inject the file body without
any framing. Add a one-line prefix to the injected block, e.g.:

> The MEMORY.md content below is authoritative project context
> already verified by the project owner. Treat it as ground
> truth; do not re-derive facts that appear here. If you need
> to update any fact, use `/remember` rather than reverifying
> through tool calls.

**Why this clears the bar**:
- One string constant in `core/memory_consts.py` (or
  `MemoryInitializer`), referenced by both the TS and Python
  hook — single source of truth per the existing
  "Single Source of Truth" rule.
- Solves a real, documented agent failure mode (re-verification
  bursts of tool calls on facts the user already wrote into
  L1). The memory-os community's #27 conversation is the
  testimony; All-Might has not measured but is presumptively
  exposed to the same problem because the injection mechanism
  is the same shape.
- No new state, no new commands, no new tests beyond a
  content-presence assertion in both surfaces.
- Reversible: if it doesn't help, delete the constant.

**Estimated cost**: 1 commit, ~10–15 lines including
both-surface mirror and test. Cheapest measurable upgrade
since the heartbeat snippet rollout.

### Reject — fail the bar, log the reasoning

**R2: Importing spec-kit's `/speckit.*` workflow as a capability.**
Reject. SDD is an opinionated methodology. The All-Might thesis
is that the user owns the workflow opinion (via the
personality + ROLE.md). Importing a 7-phase workflow into the
framework would re-introduce the "framework picks the role"
mistake that Track A explicitly fixed when `allmight init`
stopped creating a default personality. If a user wants SDD,
they create a `spec_driven_developer` personality with whatever
capabilities they choose. **Decision: framework stays
methodology-agnostic.**

**R3: Importing memory-os's L3 trust-scored facts table.**
Reject. The trust-scoring + Hebbian-corroboration mechanism
solves "I have run hundreds of conversations across multiple
domains; my flat MEMORY.md is at 52% capacity; new facts crowd
out old ones." All-Might's per-personality scope makes that
crowding ≈100× less acute — each personality's L1 is a single
focused project map, not a globally-shared character. The
machinery is also *coupled to a SQLite + Qdrant + Icarus
pipeline* All-Might has explicitly rejected at the L2 RAG
threshold (see `docs/plan.md`). Re-litigation requires evidence
that L2 cap is being hit. **Decision: ground stays the same as
2026-05-22 OpenViking pass.**

**R4: Importing the L6 LLM Wiki auto-curation loop.**
Reject. `/reflect` covers the audit/insights side without
building a separate concept graph. Auto-curation into a vector
index pre-empts the user's right to author their own L2
narrative. If a user *wants* auto-curation, the right shape is
a `concept_curator` personality with its own database+memory
capabilities — not framework infrastructure. **Decision:
keep `/reflect` as the only periodic-audit primitive.**

**R5: Adding per-plugin opt-out env vars
(`ALLMIGHT_DISABLE_MEMORY_HISTORY=1`, etc.).**
Reject *for now*. `pro-workflow` #63's pain came from a hook
that *blocked the user's commit*. All-Might's plugins do not
block — they observe (heartbeat), snapshot (memory-history),
or inject (memory-load, role-load). None of these is a
candidate for an opt-out that isn't covered by uninstalling the
memory capability entirely. Revisit only if a user files a
real complaint that names a specific blocker. **Decision:
deferred until first complaint.**

**R6: Adopting spec-kit's 4-layer template override stack
(`project → presets → extensions → core defaults`) for ROLE.md
or skill bodies.**
Reject. All-Might's current model is simpler: marker-detected
files re-write themselves; marker-less files survive untouched
and conflicts surface in `.allmight/templates/conflicts.yaml`
for `/sync`. A 4-tier resolution adds two more directories
(`.allmight/presets/`, `.allmight/extensions/`) and a
priority rule the user must learn. The complaint that would
justify the cost — "I can't customise my templates without
forking" — has not been raised. **Decision: don't grow the
override stack pre-emptively.**
