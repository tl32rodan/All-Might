## 2. Panoramic snapshot — opencode-mem

- **Pitch**: "A persistent memory system for AI coding agents that
  enables long-term context retention across sessions using local
  vector database technology." Local-first; piggybacks on existing
  OpenCode auth so no separate API key for embedding providers.
- **Storage**: SQLite (source of truth) + USearch (in-memory vector
  index, falls back to ExactScan if USearch is unavailable). All
  local — no cloud round-trip.
- **Scoping**: two modes — `scope: "project"` (default, filters by
  canonical project ID = git remote URL or root-path hash) and
  `scope: "all-projects"` (cross-project search across user shards).
  No third "per-role" scope.
- **Write path**: auto-capture learns user profiles from
  conversation, plus explicit `memory({mode: "add"})` API. Auto-
  capture means the write happens on the hot path.
- **Surface**: not slash-commands — a `memory({mode, ...})`
  JS/TS function callable from any plugin or tool, plus a local
  web UI at `127.0.0.1:4747`.
- **Maturity**: v2.17.4 (2026-06-28); 27+ releases; 25 open issues
  (not retrievable from session — repo not in MCP scope).

## 3. Panoramic snapshot — agnix

- **Pitch**: "Lint agent configurations before they break your
  workflow." Solves the problem that misconfigured CLAUDE.md /
  AGENTS.md / SKILL.md / hooks / MCP entries fail *silently* — the
  agent ignores the bad shape instead of erroring.
- **Rule set**: 432 rules across 9 tool families
  (Claude Code, agent-skills, Kiro, MCP, AGENTS.md, Copilot,
  Cursor, Cline, Gemini CLI). Example rule: `CC-SK-*` rejects

## 4. Four-axis comparison

### a) Design philosophy

| | All-Might | opencode-mem | agnix |
|---|---|---|---|
| Central abstraction | **role × capability** | flat tagged-memory store | tool-config validator |
| Memory store | markdown/YAML + git mirror | **SQLite + USearch vector index** | n/a (lints, doesn't store) |
| Scope unit | per-personality (per-role) | per-project / cross-projects | per-file |
| Write trigger | agent-explicit (`/remember`) | **auto-capture on chat events** | n/a |
| Auditability | git mirror + `allmight memory log/restore` | SQLite + web UI | rule-traceable lint errors |

Two observations:

1. **opencode-mem is the third independent vote for "heavyweight DB
   memory" on OpenCode** (after `claude-mem`'s SQLite+Chroma stack
   and the broader `mem0`/`letta`/`honcho`/`cognee` set). Combined
   with Serena (markdown) + All-Might (markdown), the ecosystem has
   split decisively into two camps: opaque-DB vs human-readable. The
   2026-05-25 pass called the markdown camp the validated direction;
   today's data point doesn't change the call, but it does say the
   opaque-DB camp is still recruiting. The deciding criterion remains
   **diff-ability and recovery**: when an auto-capture goes wrong on
   the hot path, opencode-mem's user has SQLite columns to inspect;
   All-Might's user has a git log of every memory file change and
   `allmight memory restore <file>`. The markdown camp wins on
   recoverability per session-cost.

2. **agnix is a *category* All-Might doesn't have a peer in yet** —
   third-party static validation of agent-config files. The closest
   we have internally is `test_capability_manifest.py` (validates
   our own contract). agnix validates against the *broader
   ecosystem's* expectations. See §5 Q3.

### c) Mechanism design — what's actually different

1. **opencode-mem's auto-capture is the design tax for the
   convenience.** Auto-capture sounds great until it captures the
   wrong thing and you have to untangle it from a SQLite blob. All-
   Might's `/remember` requires the agent to make an explicit call,
   which costs one slash-command per write but means every entry has
   a traceable origin and lives in a diff-friendly file. The 2026-05-
   22 OpenViking review and the 2026-05-25 Q5 ReflexionMemory
   rejection both pointed at the same failure mode: write-path
   automation amplifies bad signal. opencode-mem confirms the failure
   mode is real *and being shipped anyway* — which means the
   recoverability angle becomes more important, not less, as a
   selling point of the markdown camp.

2. **agnix's rule density is its dual-edged sword.** 432 rules
   across 9 tools is a strong assertion of "here's how the ecosystem
   should be configured." For someone who hand-writes AGENTS.md
   files, that's a productivity tool. For All-Might — which
   *generates* every artifact deterministically with a marker — it's
   both: (a) free CI insurance against our generators emitting
   shapes the ecosystem rejects, AND (b) a possible source of
   *false positives* where agnix's opinion of the canonical shape
   disagrees with All-Might's deliberate divergence (e.g. nested
   `personalities/*/commands/` slots that don't match the assumed
   flat `.claude/commands/` layout). The right move is to *try it
   on a generated demo dir first* and see whether the failure
   ratio is "useful catch" or "noise to silence."

3. **OpenCode V2 plugin API is additive, not breaking.** The
   v1.17.10 release notes say *Added* the V2 API, not *Replaced*.
   A live community plugin guide (rstacruz gist) still documents
   the V1 surface — top-level keys `chat.message`,
   `chat.params`, `permission.ask`, `tool.execute.before/after`,
   `config`, `event` — with no V1-deprecation notice. All-Might's
   generated `memory-load.ts` / `memory-history.ts` / `role-load.ts`
   / `feedback-check.ts` / `remember-trigger.ts` / `todo-curator.ts`
   / `offline-reference.ts` use the V1 surface and should keep
   working without code change. **But that's an *expectation*, not
   a verified fact** — the CLAUDE.md "Discipline When Generating
   Third-Party Integrations" rule explicitly forbids inferring
   shape from release notes. The right closure is in §5 Q4.

### d) Implementation details

- **`plugin status` heartbeats catch this class of regression
  exactly.** If a future OpenCode release silently demotes a V1
  hook key, `allmight plugin status` will show `oc/<plugin>:
  never fired` after a user-driven session — precisely the
  failure shape this telemetry was designed to surface. That
  vindicates the 2026-05-22-era investment in heartbeats: the
  T1/T2 columns turn an invisible API drift into a visible
  per-session diagnostic. No new mechanism needed.
- **Per-role memory isolation gap is now three projects wide.**
  SuperClaude+Serena (last pass), and now opencode-mem (this pass)
  all converge on project-or-global scope. None express per-role.
  The decomposition cannot represent it — opencode-mem could
  *bolt on* a `role:` tag in its SQLite schema, but that's a
  retrofit, not a primitive, and would interact poorly with the
  existing project/all-projects scope axis.

| Q3 | Run `agnix` in CI against the output of `allmight init <demo>`? | **Defer — single-run trial first.** | Real candidate but not a code change yet. Action: one-shot run of `npx agnix .` on a fresh `allmight init` output, capture the lint report, categorise findings as (a) real issues to fix in our generators, (b) divergence we accept and would need to suppress, (c) noise. Decision to wire it into pytest is downstream of the trial. Estimated cost ≤30 min for the trial; the wiring itself is a 1-line `subprocess.run(["npx", "-y", "agnix", str(demo)])` test if it passes the trial. See §7 for rating. |
| Q4 | Automate the existing manual `tsc --noEmit --skipLibCheck` step against generated plugins? | **Yes — small, high-leverage.** | The CLAUDE.md *After Code Changes* step is already in our muscle memory but runs only when someone remembers. A new test `tests/test_generated_plugins_typecheck.py` that runs `allmight init <tmpdir>`, installs `@opencode-ai/plugin@latest`, runs `tsc --noEmit --skipLibCheck`, and asserts exit 0 would catch the V2-API regression class *as soon as upstream ships a deprecation* — exactly the failure shape Q1 defers on. This is the simplicity-preserving way to make Q1's deferral safe. See §7 for rating. |
| Q5 | Bolt opencode-mem's project/all-projects search axis on top of `/recall`? | **Reject.** | `/recall` is per-personality by design (L3 = personality's journal). ## 8. Lessons captured for next reconnaissance

- **Additive upstream releases (V2 alongside V1) are not always
  free.** The V1 contract continues to work *until* upstream
  silently demotes a hook. The way to make additivity safe is to
  pin the contract you depend on (Q4: typecheck against the actual
  installed types). "It still works on my machine" is a present-
  tense statement.
- **Third independent vote is the threshold for declaring a camp.**
  After two votes for a contrarian call (Serena+All-Might on
  markdown memory), the third datapoint determines whether you've
  found a stable minority position or a moment. opencode-mem
  joining the auto-capture / opaque-DB camp confirms *that* camp
  too has converged — the split is now permanent, not transitional.
  When the ecosystem splits into stable camps, the moat argument
  becomes "which camp's failure mode is your user's biggest pain"
  rather than "which is mainstream."
- **A third-party linter is a new category of dependency.** It's
  not a runtime dep, not a build dep — it's an *opinion* dep. The
  cost is the suppression list you have to maintain when its
  opinion diverges from yours. Trial before adoption; pin
  aggressively if adopted.
- **The plugin observability investment is paying compound returns.**
  Heartbeats catch silent hook demotion (this pass), missed
  ingests (prior pass), and the "fired but never injected" class
  (design rationale doc). Every cycle the ecosystem ships a new
  way for plugins to fail silently, and the heartbeat surface
  catches it without bespoke instrumentation.
