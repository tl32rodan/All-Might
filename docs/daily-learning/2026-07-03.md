## 2. Panoramic snapshot — anthropics/claude-plugins-official

- **Pitch**: Anthropic's curated, official plugin registry for Claude
  Code. Two-tier structure: **internal** plugins (`/plugins`) that
  Anthropic maintains + **external** plugins (`/external_plugins`)
  vetted through a submission form (`clau.de/plugin-directory-submission`).
- **Installation**: `/plugin install <name>@claude-plugins-official`
  (namespace-scoped, npm-like), or via the `/plugin > Discover` UI.
  Not GitHub-URL-based.
- **Marketplace schema**: `marketplace.json` entries carry
  `name`, `description`, `author`, `category`, `source`, and a
  `strict: false` flag that lets an entry advertise skills without
  a `.claude-plugin/plugin.json` manifest. `source` uses the
  `git-subdir` type with `url`, `path`, `ref`, and `sha` — SHA-
  pinned distribution with automatic update flow.
- **What a plugin exposes**: slash commands (`/commands`), agents
  (`/agents`), skills (`/skills`). **No personality/role notion**
  in the schema — plugins are functional bundles, not identities.
- **Scale**: 55+ curated plugins + 72+ community entries by early
  July 2026 (Groundy overview article). Growth is real.
- **Trust posture**: README explicitly disclaims —
  "Anthropic does not control what MCP servers, files, or other
  software are included." So the "official" badge covers curation,
  not runtime safety.

## 3. Panoramic snapshot — ClawMem

- **Pitch**: "On-device memory layer for AI agents. Claude Code,
  Hermes and OpenClaw. Hooks + MCP server + hybrid RAG search."
  Design goal: "~90% automatic (hooks) + ~10% agent-initiated
  (MCP tools)."
- **Storage**: single SQLite file at `~/.cache/clawmem/index.sqlite`
  with **FTS5** (BM25 keyword) + **sqlite-vec** (vector similarity)
  extensions. Fragment-level chunking with independent embeddings
  per fragment. Transaction-safe with atomic rollback on crash.
- **Auto-capture mechanism**: `decision-extractor` hook uses a
  **local GGUF observer model** to extract structured observations
  (decisions / preferences / milestones / problems / bugfixes) from
  session transcripts at session end. This is new territory — the
  auto-capture camp so far shipped LLM-provider-based extractors
  (opencode-mem uses OpenCode's auth) or rule-based ones (claude-mem);
  ClawMem runs an on-device model.
- **Hook set (7 default)**: `context-surfacing`, `postcompact-inject`,
  `curator-nudge`, `precompact-extract`, `decision-extractor`,
  `handoff-generator`, `feedback-loop`. This is dense — All-Might
  emits 7 plugins today (`memory-load`, `memory-history`, `role-load`,
  `feedback-check`, `offline-reference`, `remember-trigger`,
  `todo-curator`), so the surface counts are similar; the ClawMem
  ones lean more toward memory-write, All-Might's lean more toward
  context-injection.
- **MCP surface**: 31 tools (`memory_retrieve`, `search`, `vsearch`,
  `query`, `intent_search`, `find_causal_links`, `timeline`, …).
  This is a much larger MCP surface than All-Might exposes — All-
  Might's L3 search is a single `/recall` skill; ClawMem gives the
  agent a query language with multiple entry points.
- **Scope model**: session-level + optional named vaults ("work"
  vs "personal"), *no* per-role scope. Multi-vault is opt-in via
  config; retrieval tools default to a shared vault.
- **Recovery**: `lifecycle_restore(query)` resurrects auto-archived
  docs; A-MEM metadata tracks evolution version numbers with
  reasoning. **No git-diff-style comparison.** Users inspect
  document versions manually or use `timeline` for temporal context.
- **Maturity**: v0.10.4 latest visible; patch cadence biweekly-to-
  monthly; active early 2026.

## 4. Four-axis comparison

### a) Design philosophy

| | All-Might | claude-plugins-official | ClawMem |
|---|---|---|---|
| Central abstraction | **role × capability** | plugin as functional bundle | memory as on-device DB |
| Distribution unit | personality bundle (git remote) | plugin (marketplace ref + sha) | vault (per-machine) |
| Discovery | user-supplied git URL | curated registry | n/a (local install) |
| Runtime trust | user-owned files | Anthropic-curated + user disclaimer | user-owned files |
| Extraction | agent-explicit (`/remember`) | n/a | **local GGUF model** on hot path |
| Recovery | git mirror + `allmight memory restore` | no-op (plugin is code, not state) | lifecycle-restore, no diff |

Three observations:

1. **The upstream distribution model has moved.** When
   `allmight share publish/pull` shipped, "git remote" was the
   only working option — Anthropic hadn't yet published a
   marketplace, and OpenCode's plugin story was ad-hoc `.opencode/
   plugins/*.ts` sourcing. Today, `/plugin install <name>@<market>`
   is first-party in Claude Code, and the ecosystem has ~4-5
   competing marketplaces (Anthropic official, tonsofskills.com,
   xiaolai's, ananddtyagi's cc-marketplace, quemsah's tracker).
   The distribution-model gap between All-Might's git-URL flow and
   the marketplace flow is now visible. See §5 Q1.

2. **The opaque-DB memory camp now has three distinct engineering
   theses.** claude-mem = SQLite+Chroma (heavy cloud-embedding
   optional); opencode-mem = SQLite+USearch (OpenCode-auth-piggy-
   back); ClawMem = SQLite+FTS5+sqlite-vec (local GGUF extractor).
   All three are auto-capture-first. The consistent absence in
   every one of them is **per-role scoping** — none expose it,
   and ClawMem explicitly documents "no RBAC or capability-scoped
   retrieval." Same finding as 2026-06-29 — the moat holds — but
   now the sample size makes it a *category truth*, not just a
   trend datapoint. Three engineering theses converge on the same
   scope model.

3. **ClawMem's local GGUF observer is the first genuinely novel
   move on the extraction axis in two months of surveying.** It
   changes the failure-mode conversation: opaque-DB failures used
   to be "SQLite blob to untangle"; ClawMem adds "on-device model's
   opinion of what mattered." The user now has two write-path
   opacities (the DB shape and the observer's judgment). This
   *strengthens* the recoverability argument: git-diff-able YAML
   files with explicit `/remember` invocations remain the only
   camp where the user can trace every entry to a specific,
   readable decision. But it *weakens* the "auto-capture is a
   design tax" argument — ClawMem shows the auto-capture side has
   room to keep innovating without adopting cloud embeddings.

### c) Mechanism design — what's actually different

1. **Marketplace = solved installation, not solved sharing.** The
   marketplace mechanism handles *plugin distribution* well —
   SHA pinning + git-subdir + a `strict: false` escape hatch for
   pre-existing skill repos is a clean pattern. But an All-Might
   personality bundle is *per-project state* (a personality's
   memory data + database config), not a piece of reusable tooling.
   The marketplace shape doesn't match: there's no `--target-project`
   parameter, no merge/dialog semantics, no PII-review checkpoint.
   The correct read is that the marketplace validates the **tooling
   distribution** shape (what All-Might's `.opencode/` globals *look
   like*), not the **state distribution** shape (what `share
   publish/pull` does). See §5 Q1.

2. **Local GGUF observer is a real engineering path — but not for
   us.** ClawMem's `decision-extractor` is impressive because it
   sidesteps the "which cloud embedding API" question opencode-mem
   answers by piggy-backing on OpenCode auth. For All-Might, this
   design would require: (a) shipping or fetching a GGUF at init;
   (b) an inference runtime dependency (llama.cpp or equivalent);
   (c) an extraction schema. Every step is a step toward opaque
   auto-capture and *away* from git-diff-able entries with explicit
   provenance. The right response is not to copy — it is to
   **document** why we do not, so a future contributor doesn't
   propose it in the next reconnaissance cycle.

3. **agentskills.io as a hedge worth watching.** If the standard
   really is in ~40 clients by mid-2026, then All-Might's dual OC/CC
   bridge could become an "N-1" strategy: everything but the
   standard. That's *fine* for now — we cover the two editors our
   users use — but the moment a bug report is "why doesn't All-
   Might work in Cursor?", the answer changes shape. Verification
   plan: WebFetch agentskills.io and inspect what a compliant
   skill actually declares. If it's a superset of `.opencode/
   skills/<name>/SKILL.md` frontmatter, our mirror plan is a
   near-term win; if it introduces a new manifest with fields we
   don't emit, it's a new asymmetry to close. See §5 Q4.

### d) Implementation details

- **The plugin-status heartbeat surface is unaffected by any of
  these.** Both new competitors are in domains our observability
  doesn't sit on top of: claude-plugins-official is about install-
  time distribution, ClawMem is about a foreign SQLite process.
  T1/T2/T3 columns keep working exactly as designed.
- **The share/publish CLI does *not* need a marketplace pivot yet.**
  Point-of-fact evidence: as of this pass, All-Might has zero
  observed users asking for marketplace-style distribution. Two
  personalities have been shared in the wild both times over
  git-remote (no source, but internal record). Until there is a
  *user*-driven reason to add a marketplace publisher, this is
  premature architecture. See §5 Q1.
- **Local extraction models would violate CLAUDE.md's model-
  independence stance.** All-Might's L2 storage is human-authored,
  L3 storage is vector-indexed via SMAK. Introducing a GGUF at
  init means a runtime model choice bakes into the framework —
  and every user's project state depends on which observer version
  we shipped. That's a model-lock-in vector we've explicitly
  avoided. Not a new axis; a restatement of an existing rule against
  a new temptation.

## 8. Lessons captured for next reconnaissance

- **Category truth from three data points.** After 2026-06-29 the
  case for "per-role scope is our moat" was two-vote. Adding
  ClawMem makes it three-vote and — more importantly — three
  independent engineering theses (SQLite+USearch, SQLite+Chroma,
  SQLite+FTS5+sqlite-vec) all converging on the same
  session-or-vault scope. The pattern is now a **category truth**:
  auto-capture memory *cannot* naturally express per-role scope,
  because the write-path signal comes from the session, not from
  the role. Documenting this in `docs/plan.md` would fossilise a
  point the next contributor should not re-derive.
- **New engineering paths can validate old design decisions.** The
  local-GGUF observer is a legitimately clever move on ClawMem's
  side, and the temptation to think "well, the auto-capture criticism
  no longer applies" is real. It doesn't — the criticism was never
  about *which* extractor, it was about *any* extractor between the
  user and the memory-state. Restating that in Q6 turns "the last
  criticism no longer applies" into an easy-to-cite paragraph.
- **Upstream infrastructure shifts change the *shape of the question*,
  not always the answer.** The Anthropic marketplace launch made
  us look at `allmight share`, and the result was "not applicable"
  — but the *look* was correct. When an upstream ships a distribution
  primitive, ask "does our thing fit the new shape?" every time
  before dismissing. Once was enough this pass; it will keep being
  worth asking.
- **Cross-editor open standards are the trend to *watch*, not to
  *chase*.** agentskills.io claims ~40-client adoption. If true,
  it's a supply-side change that turns our editor-bridge story
  (OC + CC) into a legitimate simplification target *later*. The
  play now is verification, not migration. If we're already
  compliant, README wins the day; if not, one small commit.
- **Marketplaces distribute tooling; frameworks distribute state.**
  Naming this distinction crisply is what let this pass reject Q1
  without a long design discussion. Marketplace = code, git-remote
  = state — the split is clean and worth remembering.
