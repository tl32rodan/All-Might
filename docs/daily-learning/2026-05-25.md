## 2. Panoramic snapshot ‚Äî SuperClaude_Framework

- **Pitch**: "a **meta-programming configuration framework** that
  transforms Claude Code ‚Ä¶ through behavioral instruction injection
  and component orchestration." Architecture is "intentionally
  simple: a well-organized collection of context files that Claude
  Code reads to modify its behavior ‚Ä¶ not from any executing code."
- **Personas = 16‚Äì20 predefined domain-specialist `.md` agents**
  (architect, security, performance, QA, frontend, backend,
  python-expert ‚Ä¶). **Generic software-engineering roles**, shipped
  by the framework.
- **Agent file format**: frontmatter (`name`, `description`,
  `category`) + sections `Triggers` / `Behavioral Mindset` /
  `Focus Areas` / `Key Actions`. The **`Triggers` section is
  explicit**: "Keywords that activate this agent, File types that
  trigger activation, Complexity thresholds."
- **Activation ‚Äî three paths**: manual (`@agent-python-expert "task"`),
  automatic (keyword match), contextual (file-type/pattern). Mostly
  model-driven pattern-matching.
- **Stateless.** Personas are "just text" behaviour modifiers ‚Äî **no
  per-persona memory**. For persistence it **recommends Serena MCP**
  ("session persistence & memory") plus a built-in "ReflexionMemory"
  for error learning.
- **Install scope = user-global**: files land in
  `~/.claude/agents/[name].md`, not in the project.
- 30 slash commands (`/sc:implement`, `/brainstorm`, `/design`, ‚Ä¶);
  claims ~70% token reduction via context compression.

## 3. Panoramic snapshot ‚Äî Serena

- **Pitch**: "The IDE for your coding agent." Agent-first tool
  design, "robust high-level abstractions" that operate "at the
  symbol level" rather than "line numbers or regex."
- **Retrieval mechanism = LSP (language servers), not vectors.**
  "Serena incorporates a powerful abstraction layer for the
  integration of language servers (LSP)." Tools: find_symbol,
  find_references, type_hierarchy, find_implementations, plus
  symbolic edits (rename, replace_symbol_body, insert_before/after).
  **Explicitly NOT vector embeddings / RAG.**
- **Memory = plain markdown files** (confirmed via docs):
  `.serena/memories/*.md` (project-scoped) and
  `~/.serena/memories/global/` (global). Human-readable; agent + user
  both read/write. Topics via `/` in name ‚Üí subdirectories. `mem:`
  reference prefix, but files are usable outside Serena.
- **Onboarding-to-memory flow**: on first encounter with a project,
  Serena runs an onboarding pass (structure, build, test setup) and
  **stores the result as memories** for future sessions.
- **Integration**: MCP server; lists Claude Code and OpenCode as
  clients.

## 4. Four-axis comparison

### a) Design philosophy

| | All-Might | SuperClaude | Serena |
|---|---|---|---|
| Central abstraction | **role** (user-defined personality) √ó capability | **persona** (predefined generic SWE specialist) | symbol-level code tool + optional memory |
| Who defines the roles | the user (`stdcell_owner`, `pll_owner`) | the framework (architect/security/QA) | n/a (no role concept) |
| Role state | **memory-bearing** (per-personality L2/L3) | **stateless** (delegates memory to Serena MCP) | n/a |
| Scope | **project** (`personalities/` in-tree) | **user-global** (`~/.claude/agents/`) | project + global memory dirs |
| Memory store | markdown/YAML + git mirror | none (recommends Serena) | **plain markdown files** |

**Two convergences, both vindicating All-Might:**
1. **Role/persona is mainstream.** A 20k-star framework's central
   abstraction is "specialized personas." All-Might is on the right
   conceptual axis.
2. **Human-readable markdown memory is independently re-derived.**
   Serena (24.5k‚òÖ, mature) made the *exact* call All-Might's CLAUDE.md
   non-goal makes ‚Äî "No SQLite memory store; markdown is
   human-readable, diff-able." Two independent projects converging on
   markdown-not-database is the strongest validation of that non-goal
   to date (claude-mem's SQLite+Chroma stack is now the outlier, not
   the norm).

### c) Mechanism design ‚Äî the two genuinely distinct ideas

1. **Serena's LSP-symbolic retrieval vs All-Might's SMAK vector
   retrieval solve *different problems*.** Serena answers "where is
   symbol `X` defined / used, rename it safely" ‚Äî *precise code
   navigation/editing*. SMAK answers "find setup-time violations in
   the stdcell rtl" ‚Äî *fuzzy semantic domain-knowledge retrieval*.
   LSP can't do the second (no semantic match); vectors can't do the
   first reliably (no exact symbol graph). **They are complementary,
   not competing.** And the second observation matters: the host
   agent (Claude Code / OpenCode) *already ships* code-navigation
   (Grep + editor LSP), so All-Might building an LSP backend would
   duplicate the host. See ¬ß6 Q2.

2. **SuperClaude's explicit `Triggers` section is a routing
   mechanism All-Might lacks.** All-Might routes via MEMORY.md
   project map + `Default personality` hint + conversation context.
   SuperClaude adds per-persona keyword/file-type/complexity
   triggers. For *capable* models this is redundant (CLAUDE.md's
   anti-pattern list says so). For the *weak/air-gap* models
   All-Might explicitly targets, explicit triggers *could* improve
   multi-personality routing. This is the only idea this pass with a
   non-zero case. See ¬ß6 Q3.

### d) Implementation details

- **The ecosystem is decomposing role + memory + code-nav into
  separate composable pieces.** SuperClaude (role layer, stateless)
  explicitly recommends Serena MCP (memory + code-nav). The de-facto
  mainstream stack is becoming *"persona framework + memory MCP."*
  All-Might bundles all three with **per-role isolation** ‚Äî which the
  composable stack *structurally cannot express*: SuperClaude personas
  are stateless+global, and Serena memory is project/topic-scoped, not
  identity-scoped. There is no way in the SuperClaude+Serena stack to
  say "stdcell_owner remembers X, pll_owner remembers Y, and they
  don't bleed." That gap is All-Might's widening moat ‚Äî same shape as
  OpenViking's per-project-isolation bug (#1942) vindicating
  All-Might in the 2026-05-22 pass.
- **SuperClaude's user-global install** (`~/.claude/agents/`) makes
  personas reusable across every project but loses project-scoped
  isolation. All-Might's project-scoped layout + `/one-for-all` share
  bundles get cross-project reuse *without* surrendering isolation.

## 6. Investment verdict

**No new feature clears the simplicity bar today.** The two
deep-dive competitors *validate* All-Might's two riskiest design
bets rather than challenge them:

1. SuperClaude proves the **role/persona abstraction** is mainstream
   and in demand (20k stars) ‚Äî and its stateless+global variant
   throws All-Might's *memory-bearing, project-scoped, per-role*
   variant into sharper relief as the differentiated design.
2. Serena independently re-derived the **human-readable markdown
   memory** choice (24.5k stars) ‚Äî directly vindicating the
   "No SQLite memory store" non-goal. claude-mem's heavy stack is now
   the outlier.
3. The widening moat is **per-role memory isolation**: the
   mainstream "persona framework + memory MCP" decomposition
   (SuperClaude+Serena) *cannot structurally express* it.

## 8. Lessons captured for next reconnaissance

- **Two independent projects converging on your contrarian call is
  stronger evidence than one.** Serena (markdown memory) + All-Might
  (markdown memory) vs claude-mem (SQLite) ‚Äî the "human-readable not
  database" non-goal is now the majority position among serious
  tools, not a quirky bet. Log convergence explicitly.
- **When the ecosystem decomposes a thing into composable parts
  (persona framework + memory MCP), check what the decomposition
  *cannot* express.** Here it's per-role memory isolation. The
  property that survives only in the vertically-integrated design is
  the moat ‚Äî name it, don't dilute it.
- **"Different mechanism" ‚âÝ "competing mechanism."** Serena's LSP and
  SMAK's vectors look like rivals but answer different questions
  (exact symbol graph vs fuzzy semantic intent). Before proposing to
  adopt a competitor's mechanism, confirm it solves *your* problem,
  not an adjacent one ‚Äî and confirm the host agent isn't already
  doing it.
- **Renames are silent doc-rot.** A dependency you cite by name
  (oh-my-opencode) can be renamed out from under your docs. Worth a
  periodic name-check on every third-party tool referenced in README/
  CLAUDE.md.
