## 2. Panoramic snapshot — `tickernelz/opencode-mem`

- **Pitch**: "A persistent memory system for AI coding agents that
  enables long-term context retention across sessions using local
  vector database technology."
- **Storage**: SQLite + USearch vector (ExactScan fallback). Data at
  `~/.opencode-mem/data` — **user-global**, not project-local.
- **Write-path = LLM-driven auto-capture.** When `autoCaptureEnabled:
  true`, the plugin issues structured-output requests to OpenCode's
  session API (Anthropic / OpenAI / Copilot provider) to extract
  memory from conversations. Also exposes a manual
  `memory({mode:"add", content:"..."})` tool. *Hot path runs an LLM
  call on relevant turns.*
- **Read-path**: vector search + chat injection (max 3 memories per
  first call by default). Web UI at `http://127.0.0.1:4747`.
- **Identity scoping**: user-level only — canonical email-derived tag
  `opencode_user_<sha16>`. *No per-role / per-persona scoping.*
- **Per-project isolation**: yes — `project` and `all-projects` scope
  modes; project tag from git remote URL or root path.
- **Deduplication**: canonical container tags
  (`opencode_project_<sha16>`, `opencode_user_<sha16>`).

## 3. Panoramic snapshot — `omnigent-ai/omnigent`

- **Pitch**: "A common orchestration layer over Claude Code, Codex,
  Cursor, Pi, and the agents you write yourself." A **meta-harness**.
- **Layer**: sits *above* the agent CLIs. Example agent ("Polly"):
  "a multi-agent coding orchestrator who writes no code herself …
  she plans, delegates the work to coding sub-agents (Claude Code,
  Codex, or Pi) in parallel."
- **Harness-swap = config + CLI flag.** Agent YAML declares
  `executor: {harness: claude-sdk}` (or `codex`, `cursor`, `pi`,
  `openai-agents`); runtime `--harness pi` overrides; `/model`
  switches in-chat. **No deep abstraction layer — each harness is a
  distinct integration; "swap" works because executors are
  pluggable backends.**
- **Policy stack** (server → agent → session, strictest wins):
  `ask_on_os_tools` (approval before shell/file writes),
  `max_tool_calls_per_session`, `cost_budget` (soft/hard), execution
  sandboxing (Modal/Daytona/Islo cloud + macOS seatbelt + Linux
  bubblewrap).
- **No persistent memory.** State is the session transcript;
  sessions sync across devices.
- **No persona/role concept.** `prompt:` field carries personality
  prose; multi-user accounts exist for team collaboration but agents
  have no identity model.

## 4. Four-axis comparison

### a) Design philosophy

| | All-Might | opencode-mem | omnigent |
|---|---|---|---|
| Layer | Project scaffold (inside one harness) | Plugin (inside OpenCode) | Meta-harness (above N CLIs) |
| Central abstraction | role × capability | session-spanning facts | executor backend + policy |
| Write trigger | **Explicit `/remember`** (agent-driven) + per-turn marker for L3 ingest | **LLM-call per relevant turn** (auto-capture) | n/a (no memory) |
| Storage | Markdown / YAML + git mirror | SQLite + USearch vector | Session transcript only |
| Identity scoping | **Per-personality** (project-local) | User-global + project-scoped; **no per-role** | User accounts; no per-agent identity |
| Per-project isolation | ✓ | ✓ | n/a |

### b) Feature coverage

| Concern | All-Might | opencode-mem | omnigent |
|---|---|---|---|
| Per-role memory isolation | ✓ | ✗ | ✗ |
| Markdown / human-readable | ✓ | ✗ (SQLite) | n/a |
| Git-mirror recovery | ✓ | ✗ | ✗ |
| Web UI browser | ✗ | ✓ (`:4747`) | ✓ (collab web UI) |
| LLM-driven auto-capture | ✗ (deliberate) | ✓ | n/a |
| Cross-harness execution | partial (`AGENTS.md` markdown + OC/CC hook mirrors only) | ✗ (OpenCode-only) | ✓ |
| Policy / sandbox stack | ✗ (out of scope) | ✗ | ✓ |
| Cross-project share with PII review | ✓ (`/one-for-all` + `/all-for-one`) | partial (`all-projects` scope mode, no PII gate) | ✗ |

### c) Mechanism design — the genuinely distinct ideas

1. **Write-path divergence: explicit vs LLM-driven.**
   opencode-mem's *auto-capture extracts memory by issuing a
   structured-output LLM call on each relevant turn*. This is the
   ReflexionMemory shape rejected on 2026-05-25 (Q5), now shipped
   for real in 945★-worth of users. The trade-off plays out exactly
   as predicted in CLAUDE.md's `/reflect`-vs-auto-capture rationale:
   opencode-mem buys "no agent discipline needed" at the cost of an
   LLM call on the hot path. All-Might's `/remember` + the periodic
   `/reflect` audit keeps the embedder/extractor off the per-turn
   path and pays its cost only when the user (or `remember-trigger`
   plugin) calls it. The convergence finding from 2026-05-25 holds:
   **claude-mem (SQLite + Chroma, port :37777) + opencode-mem
   (SQLite + USearch, port :4747) define the "heavy memory stack
   with web UI" pattern; All-Might + Serena define the
   "markdown memory" counter-pattern.** Two pairs, two stable camps.

2. **opencode-mem has per-project isolation already — the OpenViking
   #1942 gap is not universal.** This was an open `[carry-over]`
   from the 2026-05-22 pass. Resolution: per-project isolation is
   *mainstream*, not differentiating. All-Might's moat narrows to
   the *per-role* axis (per-personality isolation inside one
   project), which neither opencode-mem nor SuperClaude+Serena
   expresses.

3. **omnigent's harness-swap is config-driven, not abstraction-
   driven.** No common "agent IR" that all harnesses implement —
   each executor is a distinct integration; swapping just means
   selecting which one to call. This *vindicates* All-Might's
   dual-platform invariant (CLAUDE.md): if no published meta-harness
   has cracked a unified abstraction over Claude Code + OpenCode +
   Codex + Cursor + Pi, attempting one inside All-Might would be
   reinventing the swap-by-config wheel without the surrounding
   policy/sandbox infrastructure that justifies the layer.

### d) Implementation details

- **opencode-mem's storage at `~/.opencode-mem/data` is user-global**
  — same scope choice as SuperClaude's `~/.claude/agents/`. User-
  global makes installation simpler but kills the per-project
  baseline All-Might gets for free from project-tree placement.
  Note: opencode-mem then *re-adds* project-scoping through tag
  filtering — i.e. *project-scoping in a user-global store*, which
  is strictly more code than a per-project directory.
- **omnigent's policy stack is the only genuinely novel mechanism
  on display today** (cost budgets + sandbox tiers + per-tool
  gating). It lives at the right layer (orchestrator), not inside
  a project scaffold. All-Might already routes risk-bearing actions
  through the host harness's permission prompts — borrowing the
  *concept* would mean owning the *runtime*, which it doesn't.

## 5. The "OpenCode memory plugin tail" — a finding

14 OpenCode-memory-plugin repos created in 6 months, all with
SQLite + vector storage, all single-author, all under 50★ except
the leader. The pattern signals:

- **The OpenCode plugin ecosystem treats memory as a first-class
  problem** — there is real, broad demand.
- **The "ship-a-memory-plugin" template has converged on
  SQLite + vector + LLM-extracted writes.** That is the easy
  default for a solo plugin author. None of the tail attempts the
  markdown/git-mirror shape — that requires the surrounding
  capability framework All-Might already has.
- **All-Might isn't in this tail's competitive set.** A user
  shopping for "an OpenCode memory plugin" will compare the 14
  tail entries by feature checklists; All-Might competes for users
  who want *project scaffold + capability framework*, not
  *plug-and-play memory*. Different shelf.

The tail is the empirical answer to "is there demand for the
plug-and-play vector-memory plugin?" — yes — and to "is there
demand for *role-isolated, markdown, project-scoped* memory?" — the
tail is silent on this, which means it's still All-Might's slot to
hold (or to lose to a future entrant).

## 7. Investment verdict

**No new feature clears the simplicity bar today.** Both deep-dive
competitors reinforce — not challenge — All-Might's positioning:

1. opencode-mem's 945★ proves there is real demand for
   memory-for-coding-agents, *and* its design choices (SQLite +
   vector + auto-capture + user-global) are the **opposite** of
   All-Might's deliberate ones. The two products serve different
   users on different shelves.
2. omnigent is at a different *layer* (orchestrator above harnesses)
   and treats memory + persona as out of scope. The two are
   structurally complementary — an omnigent user could run an
   All-Might project as one of its executor targets.
3. The 14-plugin OpenCode memory tail is empirical confirmation
   that "SQLite + vector + LLM-extract" is the *default* solo-author
   shape. All-Might's role-isolated markdown shape is structurally
   harder to drop into a single plugin file, which is *why* the tail
   doesn't contain it — and why All-Might's slot is still
   defensible.

Feature ratings for the only two ideas with even a small case:

| Idea | Rating | Verdict |
|---|---|---|
| Web UI for memory browsing (Q2) | **1/5** | Markdown is already navigable in any editor; SMAK CLI + `allmight memory log/diff` cover versioning. The UI value-add over markdown is low. |
| Codex/Cursor mirror (Q3) | **2/5** | Real demand is hypothetical; cost (drift surface) is real. Re-open on first user request. |

Neither rating crosses the investment bar. Today's deliverable is
documentation (this log + the skip-list update + the Q7 resolution).

## 8. Lessons captured

- **An ecosystem tail is a signal too.** 14 memory-plugin repos in
  6 months, all converging on the same storage shape, is a strong
  statement about *which* axis is "easy to ship" and which axis is
  structurally harder. The shape that no one in the tail ships
  (role-isolated markdown + git mirror) is the one with structural
  defence — the very property that makes it hard to ship as a
  single plugin file is the property that makes it hard to copy.
- **Per-project isolation has become table stakes.** Even
  opencode-mem (a memory plugin, not a framework) has
  `project` and `all-projects` scope modes. The `OpenViking #1942`
  gap dispositioned in 2026-05-22 was specific to OpenViking, not
  ecosystem-wide. Stop citing it as a moat; cite per-*role*
  isolation instead.
- **A rename you cite by name in your docs is a doc-rot fuse.**
  Q7 was worth chasing — and the answer was "still safe, because
  legacy was kept" — but next time a dependency renames, the
  legacy-window might not be there. The mitigation is *minimize
  what we cite by name*; CLAUDE.md's OMO block could be one line
  shorter (drop the specific filename, keep the principle), but
  that's a cleanup, not a regression fix. Park it.
- **"Different layer" is now the most common comparison verdict.**
  Three consecutive passes (claude-mem, Serena, omnigent) have
  ended with "complementary, not competing." That's worth naming as
  a *positive* finding about All-Might's scope: the project
  scaffold layer is well-bounded — adjacent layers exist and don't
  bleed into it.
