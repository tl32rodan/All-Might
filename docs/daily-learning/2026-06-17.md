## 2. Panoramic snapshot — `zilliztech/memsearch`

- **Pitch**: "A persistent, unified memory layer for all your AI
  agents (Claude Code, Codex, OpenCode), backed by Markdown and
  Milvus."
- **Storage tiers** (in `.memsearch/memory/`):
  - `YYYY-MM-DD.md` — daily journal markdown
  - `PROJECT.md` — durable project state
  - `USER.md` — user preferences
  - `.memsearch/skill-candidates/` — distilled patterns staged for
    promotion to real skills
  - Milvus / Milvus-Lite as **shadow index**, explicitly
    *rebuildable, not authoritative*
- **Retrieval is progressive**: ranked chunks → expand to full
  sections → original transcript replay.
- **Hooks**: Claude Code `Stop` hook + OpenCode plugin both
  auto-capture turn deltas. Capture is mandatory; the user does not
  invoke a `/remember` analogue.

## 3. Panoramic snapshot — `SethGammon/Citadel`

- **Pitch**: "Operating layer for Claude Code + Codex: persistent
  project memory, intent routing, safety hooks, cost telemetry,
  parallel agent fleets."
- **Layout**:
  - `.planning/` — campaigns + state + telemetry (markdown)
  - `.citadel/` — runtime config
  - `.claude-plugin/` + `.codex-plugin/` — **enforced parity**
  - `hooks/` — 35 Node hook scripts across 29 lifecycle events on
    both surfaces
  - `skills/` — 46 built-in skills
- **Four hard-coded execution "personas"** — Skill / Marshal /
  Archon / Fleet. These are *workflow tiers* (single skill, planned
  multi-step, autonomous, parallel fleet), not user-defined roles.
- **Intent routing is a 4-tier cascade**: regex → state machine →
  keyword table → LLM classifier. Known intents short-circuit
  before any LLM call.

## 4. Comparison vs All-Might

Ordered by importance: **design philosophy → feature coverage →
mechanism → implementation**.

### 4.1 Design philosophy

- **memsearch validates the markdown-source / vector-shadow call.**
  That's now **three** independent projects on the same line:
  Serena (markdown memories), `claude-mem-reverse-call`
  (markdown), memsearch (markdown daily + Milvus rebuildable
  shadow). The 2026-05-25 entry flagged Serena+All-Might as a
  two-project convergence against the SQLite cohort
  (claude-mem / mem0 / letta / honcho / cognee). Memsearch makes
  it three-against-five; among **self-hostable** projects (i.e.
  excluding managed-cloud memory products) the markdown camp is
  now the **majority position**. The contrarian call has matured
  into the consensus call for serious tools, exactly the trajectory
  worth logging.
- **Citadel validates the dual-surface discipline.** Their
  `.claude-plugin/` + `.codex-plugin/` parity, with a 35-script hook
  manifest enforced on both sides, is the closest analogue to All-
  Might's `PLUGIN_MANIFEST` + mirror-coherence test. Independent
  arrival at the same headache is evidence the headache is real and
  worth paying.
- **Citadel diverges on the role abstraction.** Their four hard-
  coded "personas" are *workflow tiers*, not domains. All-Might's
  user-defined personalities (`stdcell_owner`, `pll_owner`) survive
  this comparison cleanly: workflow tiers and domain roles are
  orthogonal. Citadel cannot express a domain personality;
  All-Might cannot natively express a "parallel fleet" workflow.
  Neither is wrong, but the All-Might axis is the one with a
  scarce competitor.
- **memsearch is missing per-personality isolation.** Memory is
  flat per project — no `personalities/<p>/` partition. For a
  multi-domain user (the All-Might target) this is a real gap, not
  a stylistic difference. Reinforces the value of naming this in
  the README as a moat (open follow-up from 2026-05-25 #Q2 — still
  not written).

### 4.2 Feature coverage

- memsearch covers a feature All-Might does not have a direct
  analogue to: **explicit `skill-candidates/` staging directory**.
  All-Might's `/reflect` writes distilled insights directly into
  `understanding/`; memsearch stages a candidate as a visible file
  the user can `ls`, review, and explicitly promote.
- Citadel covers **zero-LLM-cost intent routing** (regex / keyword
  table tier). All-Might routes via the model + ROUTING_PREAMBLE,
  paying tokens on every turn for intent decisions a keyword table
  could resolve for free.

Both are real differences. The next section is where they have to
defend themselves against the simplicity bar.

### 4.3 Mechanism design

- memsearch and All-Might land on the **same** capture mechanism
  (Stop hook + post-turn write to markdown) and the **same** index
  policy (vector layer is rebuildable). The convergence is total
  on the L3 mechanism — and All-Might's auto-ingest closure
  (`memory-history.ts` Stop + `memory-load.ts` `session.created`)
  is *more* sophisticated than memsearch's straight Stop-hook write,
  because the All-Might mirror also protects against accidental
  deletion via the local git mirror. No adoption needed.
- Citadel's hook manifest enforces dual-surface parity via a
  scripted check; All-Might's `PLUGIN_MANIFEST` does the same via
  `test_capability_manifest.py::TestMirrorCoherence`. Same shape,
  different runtime — All-Might's lives in the test suite, which
  is the right place (you cannot ship a regression past CI).

### 4.4 Implementation details

- memsearch's progressive retrieval (chunks → sections →
  transcript) is a single product feature; All-Might's
  understanding `_index.md` TOC + on-demand file read is the same
  pattern at a different granularity. Their chunked tier collapses
  one of the read steps. Not worth adopting — the L2 design
  rationale (narrative integrity matters, RAG fragments meaning)
  is intact; chunk-tier reintroduces fragmentation.
- Citadel's keyword table is implemented as a flat YAML file the
  user maintains. If a routing miss occurs the user adds a line.
  This pushes maintenance burden onto the user *to save tokens* —
  a bad trade-off for the All-Might target deployment shape (1–3
  user-defined personalities; ambiguity already low).

## 5. Concrete ideas considered

Two ideas survive long enough to be evaluated against the
simplicity bar; both are rejected, with reasoning logged so the
next pass does not re-evaluate them.

### 5.1 Explicit `skill-candidates/` staging directory

**Idea**: when `/reflect` finds a candidate distillation, write it
to `.allmight/skill-candidates/<name>.md` for user review before it
lands in `personalities/<p>/memory/understanding/`.

| Dimension | Assessment |
|---|---|
| Problem reality | **Marginal.** `/reflect` already produces audit-able output; users can roll back via `allmight memory log` / `restore`. The "I want to see what's about to be added" workflow is already covered by the git mirror diff. |
| Simplicity cost | **High.** Adds a new file-system convention, a promotion step, a new lifecycle command (or skill body) to handle promotion. Doubles the surface area of memory writes. |
| Upside | Visible staging — the user does not have to `git diff` to see what `/reflect` proposed. |
| Verdict | **Reject.** Rating **1/5**. The git mirror is the staging directory; making it visible at the FS level duplicates a mechanism that already works. |

### 5.2 Zero-LLM-cost keyword routing table

**Idea**: Citadel-style flat YAML mapping known intents to commands,
short-circuiting the LLM intent decision.

| Dimension | Assessment |
|---|---|
| Problem reality | **Unproven.** All-Might routing via ROUTING_PREAMBLE + a small personality set has *no logged misroute incident*. The token-cost argument is theoretical. |
| Simplicity cost | **Very high.** Directly trips the *Skill / Command Body Style Guide* anti-pattern: "Trust the model — can step-by-step prose collapse to 'do X'?" A keyword table is the *opposite* answer. It also competes with ROUTING_PREAMBLE as a second source of routing truth. |
| Upside if real | Token savings on routing decisions on hot paths. |
| Verdict | **Reject.** Rating **1/5**. Mechanism creep; competes with single-source-of-truth ROUTING_PREAMBLE. Same fate as the explicit-Triggers idea from 2026-05-25 (Q3) — rejected for the same reason. |

## 6. Lessons captured for next reconnaissance

- **Three independent projects on the same line crosses from
  "convergence" to "consensus."** Markdown-source + vector-shadow
  is no longer a contrarian All-Might call; it is the prevailing
  design among serious self-hostable agent-memory tools. The
  README/CLAUDE.md framing can move from "we deliberately rejected
  SQLite" (defensive) to "the design that lets memory be diffable
  and human-recoverable" (declarative). Wording change only — no
  code work. Re-open if a clear counter-trend emerges.
- **The differentiator narrows but sharpens.** When the storage
  layer becomes commodity (markdown + shadow vector everywhere),
  the *isolation* layer is what's left. memsearch's flat-per-
  project memory has no `personalities/<p>/` partition; All-Might
  does. That property — per-role memory isolation under a single
  workspace — is now the moat. Carry-over open follow-up from
  2026-05-25 (README scope-boundary note positioning the database
  capability) should be widened to also name this memory-isolation
  property.
- **Dual-surface OC↔CC discipline has a peer now.** Citadel pays
  the same parity tax we do. No reason to back off the
  `PLUGIN_MANIFEST` + mirror-coherence test approach.
- **A routine that finds nothing worth building is doing its job.**
  Two ideas fully evaluated, both rejected with reasoning logged
  so the next pass does not re-evaluate them. That is the
  intended outcome of "若無值得學習之處或者只是小幅度改進，以
  簡潔設計為最高級指導原則".
