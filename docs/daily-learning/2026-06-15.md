## 2. Panoramic snapshot — `guard-skills`

- **Pitch**: *"let your agent do the work, then invoke the relevant
  guard on the diff before you present, commit, or merge it."*
  Guards are **reactive review passes** keyed to known
  AI-generated failure modes, not generative agents.
- **Layout** — per-skill folders, not monolithic:
  ```
  skills/clean-code-guard/
    ├── SKILL.md             # short entrypoint
    ├── agents/openai.yaml
    └── references/          # progressive-disclosure
  ```
  Same shape All-Might uses (`.opencode/skills/<name>/SKILL.md` +
  body in `capabilities/<cap>/templates/`); independent convergence.
- **The five guards** target *specific* failure modes, not generic
  review:
  - `clean-code-guard` — catch-all error swallowing, hardcoded
    "success" returns, hallucinated APIs, premature abstraction.
  - `test-guard` — mock abuse, duplicate tests, implementation-
    detail assertions.
  - `docs-guard` — hallucinated symbols, broken samples,
    docs-vs-code drift.
  - `wp-guard` — WordPress escaping, sanitization, nonces.
  - `woo-guard` — WooCommerce HPOS, order meta, checkout bypass.
- **Invocation** — explicit `$` prefix, manual:
  `Use $clean-code-guard on the diff you just produced.`
  No automatic hooks; the *user* triggers the review.

## 3. Panoramic snapshot — `vibecode-pro-max-kit`

- **Pitch**: a "meta-harness" that "turns any AI coding agent into
  a spec-driven engineering team" via the **RIPER-5 workflow**:
  *Research → Innovate → Plan → Execute → Update*. Transitions
  between phases require **explicit user approval** at each gate.
- **Phase-locking via tool restrictions (real, not advisory):**
  > "Your agent literally cannot write code during research.
  > RESEARCH is read-only, INNOVATE has no Bash, PLAN can only
  > write to process/. Actual capability removal, not suggestions."
- **Self-improving context memory** — domain-routed groups under
  `process/context/`:
  ```
  process/context/
    ├── all-context.md         # router
    ├── tests/all-tests.md
    ├── container/all-container.md
    └── {domain}/all-{domain}.md
  ```
  Auto-updates after each completed feature via a dedicated
  `vc-update-process-agent` running a "7-phase mandatory checklist
  including stale-artifact scanning". Completed plans archive to
  `process/features/{name}/completed/` and feed forward.
- **12 agents / 31 skills** — agents are *workflow phases* (one per
  RIPER step + specialists like debugger/tester/git-manager), not
  domain personas. Skills are **auto-discovered by keyword
  matching**: *"Scans 32 skills and matches keywords. Say 'add
  webhook support' and vc-security + vc-scenario surface
  automatically."*
- **Mirrors Claude → Codex** via `.codex/agents/`; `.agents/skills`
  is a symlink to `.claude/skills` for discovery. Same dual-platform
  intent as All-Might's `.opencode/` ↔ `.claude/` bridge.

## 4. Horizontal comparison

### a) Design philosophy

| Axis | All-Might | guard-skills | vibecode |
|---|---|---|---|
| Core unit | Capability (`database`/`memory`) installed into a Personality (role) | Skill (reactive review) | Workflow phase agent + auto-routed skill |
| Memory shape | YAML + markdown, per-personality, git-mirrored | None — review-only | Domain-routed markdown groups, auto-updated after feature completion |
| Routing | MEMORY.md project map + Default-personality callout + conversation context (single source: `ROUTING_PREAMBLE`) | Explicit `$<name>` user trigger | Keyword auto-discovery → intent detection → phase-locked agent |
| Isolation primitive | **Per-personality data dir** (true isolation, file-system enforced) | None | Tool restrictions per workflow phase (read-only / no-Bash / write-only-to-process/) |
| Composition story | Capabilities compose into a personality; multiple personalities coexist | Guards are independent and stateless | Phases are sequential; one phase's output is the next phase's input |
| Trigger story | Implicit (agent picks via context); explicit only via `/<command>` | Explicit `$` (user invokes) | Hybrid: auto-discovery for skills, explicit gate for phases |

### b) Feature coverage

- **What guard-skills has and All-Might doesn't**: a *named catalog*
  of common AI failure modes (catch-all error handling, mock abuse,
  hallucinated APIs, docs-vs-code drift). All-Might has `/reflect`
  but it's a generic audit — no opinionated checklist of what to
  look for. *Could* be useful, but see §5 Q1 for why packaging
  domain-specific guards into the framework is the wrong layer.
- **What vibecode has and All-Might doesn't**: (i) **tool-level
  capability restrictions per agent**, (ii) **auto-updating
  context after completed work**, (iii) **keyword-matched skill
  auto-discovery**. (i) is the most interesting — see §5 Q3.
- **What All-Might has and neither has**: (a) per-personality memory
  isolation with file-system enforcement, (b) `share publish/pull` +
  `/one-for-all`/`/all-for-one` for cross-project personality
  transfer, (c) capability-template plug-in seam (`cli_options`,
  `install`, `install_globals`), (d) dual-platform invariant
  enforced by `PLUGIN_MANIFEST` + tests.

### c) Mechanism design

1. **Convergence: per-skill folder with progressive-disclosure
   references is now mainstream.** guard-skills uses
   `<skill>/{SKILL.md, references/}`; vibecode uses
   `.claude/skills/<name>/SKILL.md`; All-Might uses
   `.opencode/skills/<name>/SKILL.md` with content lazy-loaded
   from `capabilities/<cap>/templates/`. Three independent
   reinventions of the same shape — re-validates All-Might's
   skill-IO contract.
2. **Divergence: explicit user trigger vs implicit context routing.**
   guard-skills' `$<name>` and vibecode's keyword auto-discovery
   solve the *same* problem — "which skill should fire?" — with
   opposite UX. All-Might took a third path: agent picks via
   `ROUTING_PREAMBLE` + conversation context. The vibecode keyword
   approach is the SuperClaude-style explicit-trigger idea (Q3 of
   2026-05-25) re-instantiated for skills instead of personas — and
   reinforces the same deferred verdict (see §5 Q4).
3. **Tool-level capability restrictions per agent** (vibecode's
   "RESEARCH cannot Bash") is a mechanism All-Might has no analogue
   for. All-Might personalities are advisory roles; the agent has
   the same tool surface regardless of which personality is
   "active." This is the *one* mechanism with a real, non-zero case
   for All-Might to consider — see §5 Q3.
4. **Auto-update on feature completion** (vibecode) vs **periodic
   audit** (All-Might `/reflect`) vs **per-observation manual
   write** (All-Might `/remember`). vibecode chose the hot-path
   write that All-Might explicitly rejected (see 2026-05-22 OpenViking
   failure-tail analysis). Two opposing bets in the field now; worth
   re-watching to see which one accumulates breakage.

### d) Implementation details

- guard-skills' per-skill `references/` directory is the same
  *progressive disclosure* pattern All-Might uses with its lazy-
  loaded `.md` bodies and CLI deference. Worth a one-line callout
  in CLAUDE.md's "Skill / Command Body Style Guide" that the
  pattern is no longer All-Might-specific — it's mainstream.
- vibecode's `process/context/` router (`all-context.md` listing
  every domain) is structurally what All-Might's
  `personalities/<p>/memory/understanding/_index.md` TOC does at
  the *L2* tier — but vibecode does it at the *project* tier
  without a per-personality slice. The "router at top, single-file
  read on demand" pattern is the same mechanism (and the same
  "L2 RAG not needed below ~200 files" non-goal applies for both).
- vibecode's `.codex/agents/` mirror is the same dual-platform
  problem All-Might solves with `.opencode/` ↔ `.claude/`. Worth
  noting: vibecode treats Codex as the secondary surface and Claude
  Code as canonical. The asymmetric direction matches our choice
  (`.opencode/` canonical, `.claude/` mirror) — independent
  convergence on "pick one source of truth, project to the other."

|---|---|
| Problem reality | **Plausible, not yet observed.** A `code_reviewer` personality writing code "by accident" hasn't been logged, but the failure mode is concrete enough to anticipate. Vibecode's choice to enforce this structurally is evidence that someone hit it. |

| **Verdict** | **Spike, then decide.** Rating **4/5** if the host APIs align (build it); **2/5** if they don't (defer, document the gap). The asymmetric cost — 1 hour to know — is why this clears the simplicity bar for *investigation* even before it clears for *build*. |

## 8. Lessons captured for next reconnaissance

- **A skill pack ≠ a framework.** guard-skills (729 ★) is genuinely
  useful but has no framework thesis; it's content shipped in the
  same shape All-Might already supports. When the field produces a
  popular skill pack, the question for All-Might is "does our
  skill-IO seam make this trivial to install?" — not "should we
  ship this content?" Today the answer is yes (the seam works).
- **Convergent reinvention is the strongest validation signal so
  far.** Three projects (All-Might, guard-skills, vibecode) landed
  on per-skill-folder + progressive-disclosure independently; two
  projects (vibecode, All-Might) landed on dual-platform mirror
  with one canonical surface independently. When two or more
  independent teams arrive at the same shape, the shape is the
  default — write it into CLAUDE.md as such.
- **A 1-hour spike is the right unit when the answer depends on a
  single fact you can verify.** Q3 has been deferred twice
  (2026-05-25 as routing, 2026-06-15 as tools); the deferral has
  always been "we don't know if the host supports it." That's a
  question, not an open design issue. Schedule the spike, get the
  answer, stop deferring.
- **Watch the opposite bet.** When a competing project takes the
  exact bet you rejected (vibecode → hot-path auto-update,
  All-Might → user-triggered `/reflect`), put their issue tracker
  on the watch list. Either they'll vindicate your call when the
  failure tail materialises, or they'll find a mechanism that
  removes the risk you assumed was inherent — both are useful.
