## 2. Panoramic snapshot — agentic-stack

- **Pitch**: "Keep one portable memory-and-skills layer across
  coding-agent harnesses, so switching tools doesn't reset how your
  agent works." One brain, many harnesses.
- **Abstraction unit**: `.agent/` folder. Five sub-layers:
  - `memory/working/` — immediate session context.
  - `memory/episodic/` — action logs and events.
  - `memory/semantic/` — graduated lessons (jsonl + rendered .md).
  - `memory/personal/` — user preferences read at session start.
  - `skills/`, `protocols/` (typed tool schemas + `permissions.md`),
    plus per-harness adapter shims and a local `data/` layer
    (dashboards, cron timelines, token estimates).
- **Memory promotion** is **review-gated, not agent-autonomous**:
  - `auto_dream.py` nightly clusters episodic into *candidate*
    lessons (it never writes to `semantic/` directly).
  - A host-agent review protocol enforces gates:
    `graduate.py` (accept + rationale), `reject.py` (decline + reason
    preserved), `retract_lesson.py` (append-only audit trail).
  - Graduated lessons land in `semantic/lessons.jsonl` → rendered to
    `semantic/LESSONS.md`. Rejected candidates retain churn history.
- **Skill self-rewrite**: "every skill includes a self-rewrite hook
  for degraded performance" — flags any skill failing 3+ times in
  14 days for rewrite.
- **Harness coverage**: 10+ adapters — Claude Code, Copilot CLI,
  Cursor, Gemini CLI, Windsurf, OpenCode, OpenClaw, Hermes, Pi,
  Codex. Each is declared via `adapter.json`; shared Python backend
  applies platform-specific config (`CLAUDE.md`,
  `.cursor/rules/*.mdc`, `.windsurfrules`, `gemini.md`, etc.).
- **Safe upgrade**: `agentic-stack upgrade --dry-run` previews
  `.agent/` infrastructure changes; the actual upgrade leaves
  user-owned config untouched.
- **Transfer wizard**: `./install.sh transfer` moves preferences /
  lessons / skills / working memory between projects, blocking
  secret-like content, emitting a one-line curl for the target.
- **Explicit anti-patterns**: no provider coupling, no unattended
  memory mutation, no config overwrites on upgrade, no orphaned
  adapter files, no skill bloat (progressive disclosure via
  manifest).

## 3. Panoramic snapshot — OpenGAP

- **Pitch**: "Clone a repo, get an agent." A framework-agnostic,
  git-native standard for agent identity. Spec + JSON Schema +
  adapter ecosystem, not a runtime.
- **Required schema** (only two mandatory files):
  - `agent.yaml` — manifest with `spec_version`, `name`, `version`,
    `description`, optional `model.preferred`, `compliance:`,
    `dependencies:`, `extends:`.
  - `SOUL.md` — identity, personality, values, communication style.
- **Optional directories** (constraint-minimal — add what you need):
  - `RULES.md` / `DUTIES.md` — hard constraints + segregation-of-
    duties policies.
  - `skills/`, `tools/` (MCP-compatible), `workflows/`,
    `knowledge/`, `memory/runtime/` (`dailylog.md`, `context.md`),
    `compliance/`, `agents/` (recursive sub-agent nesting).
- **Composition primitives**: `extends:` (parent agent inheritance,
  via git URL + semver), `dependencies:` (git-based agent
  composition).
- **Adapter ecosystem** — bidirectional. **Export targets** (14+):
  system-prompt, claude-code (CLAUDE.md), openai (Agents SDK),
  crewai, lyzr, cursor (`.cursor/rules/`), github, gemini (GEMINI.md
  + settings.json), copilot, codex, kiro, gitclaw, nanobot.
  **Import sources** (reverse-port): claude, cursor, crewai, opencode.
- **CLI**: `init`, `validate`, `export --format <adapter>`,
  `import --from <fmt>`, `run <source> --adapter <adapter>`, `audit`
  (compliance report), `skills`, `info`. `gitagent` and `opengap`
  are aliases.
- **Compliance-first**: `compliance:` block in `agent.yaml`
  declares FINRA 3110 / 4511, Fed SR 11-7, SEC 17a-4 policies +
  segregation-of-duties role definitions + conflict matrices.
  `opengap audit` generates checklists; `opengap validate
  --compliance` blocks deployment on violations.
- **Spec versioning**: `spec_version` is required and validated
  against `spec/schemas/` JSON Schemas on every CLI invocation —
  a stale manifest fails fast rather than half-applies.

## 4. Horizontal comparison vs. All-Might

Ranked in the priority the problem statement specifies:
**a) design philosophy → b) feature coverage → c) mechanism →
d) implementation**.

### 4.a Design philosophy

| Dimension | agentic-stack | OpenGAP | All-Might |
|---|---|---|---|
| Primary cut | One **brain**, many harnesses | Open **standard** for identity, runtime-agnostic | Many **personalities** per project, dual-harness, opinionated runtime |
| Composition | Single `.agent/` folder, shared by all adapters | `extends:` inheritance + `dependencies:` graph | `Personality(capabilities=[...])`; no inheritance between personalities |
| Mutation policy | Human-in-the-loop gate on memory promotion (`graduate/reject/retract`) | Spec-versioned + schema-validated; runtime mutation is out of scope | Markers + `/sync` reconciliation; `memory-history.ts` git-mirror as recovery layer |
| Stance on harnesses | Pluralism (10+) | Pluralism (14+, plus reverse-port) | Discipline (CC + OpenCode only, single shared `.opencode/`, mirror via `.claude/`) |

**Read**: All-Might intentionally trades breadth (harness count)
and inheritance (deep composition) for *invariants* — one
`.opencode/` per project, one `AGENTS.md`, one personality registry,
markers as the contract for "this file is ours". agentic-stack
trades the invariants for harness pluralism and gets the dual-fire
/ orphan problem that the All-Might CLAUDE.md already warns about
(the OMO interaction, §Editor Compatibility). OpenGAP trades the
runtime for spec-strictness and pays for it with `extends:` cascade
complexity that the simplicity bar would reject for All-Might.

**Verdict**: All-Might's design thesis is **stricter and more
defensible**. Neither competitor pushes it to relax.

### 4.b Feature coverage

| Feature | agentic-stack | OpenGAP | All-Might |
|---|---|---|---|
| Persona / role | `personal/` preferences only (no first-class role) | `SOUL.md` per agent; `agents/` recursive sub-agents | First-class `Personality` with capability subset |
| Memory tiers | working / episodic / semantic / personal (cognition-axis) | `runtime/dailylog.md` + `context.md` (single tier) | L1 / L2 / L3 by access-pattern (storage-axis) |
| Vector / search | jsonl + clusters; no vector layer | None (delegates to runtime) | SMAK at L3 only; L1/L2 stay loadable |
| Reflect / promote | Nightly clustering + human review verbs | `validate` + `audit` (spec-side, not memory-side) | `/reflect` agent-driven; `memory-history` git mirror |
| Multi-platform | 10 adapters | 14 export + 4 import | 2 mirrored (CC + OpenCode) |
| Lifecycle CLI | `upgrade --dry-run`, `dashboard`, `transfer`, `sync-manifest` | `init`, `validate`, `audit`, `info` | `init`, `add`, `list`, `clone`, `migrate`, `memory`, `share` |
| Skill body discipline | Manifest-driven progressive disclosure | Constraint-minimal, no size policy | Hard size budget (`test_skill_body_size.py`) + style guide |

**Read**: All-Might has **deeper** memory primitives (L1/L2/L3
chosen by access pattern) and **stricter** skill discipline, but
**shallower** lifecycle CLI than agentic-stack (no dry-run, no
dashboard, no health check) and **shallower** spec discipline than
OpenGAP (no `spec_version` integrity check).

### 4.c Mechanism

- **agentic-stack `auto_dream + graduate/reject/retract`** is an
  explicit gate against unattended memory mutation. All-Might's
  closest equivalent: `memory-history.ts` snapshots every memory
  write to a separate `.git`, and `allmight memory log/restore`
  exposes it. That gives **recovery** but not a **gate** — the
  agent's edit has already landed before the user sees it. The
  pro-workflow tradeoff (the simplicity bar): the gate's three
  verbs + a jsonl candidate-pool add real surface; recovery without
  a gate has been judged sufficient (see `docs/plan.md` non-goal:
  "No SQLite memory store").

- **agentic-stack `upgrade --dry-run`** plus the "safe upgrade
  policy" formalises a preview step before re-init touches the
  workspace. All-Might already preserves user-authored files via
  markers + stages conflicts at `.allmight/templates/conflicts.yaml`
  for `/sync`. The missing piece is the **preview** — today the
  agent has to commit to the write and inspect the staged conflicts
  *after*. A dry-run would emit the same conflict report without
  the write.

- **OpenGAP `spec_version`** enforces a one-field integrity check
  at every CLI invocation. All-Might's `.allmight/personalities.yaml`
  carries per-personality `versions:` but no **framework-level**
  schema version. After a Part-D → Part-E layout change, `allmight`
  on an old project would either silently mis-write or fail in a
  hard-to-diagnose place. The marker system catches *file* drift,
  not *layout* drift.

- **OpenGAP `extends:` + `dependencies:`** is interesting on paper
  but recreates the inheritance taxes the personality platform was
  designed against. `compose()` is intentionally **downward only**;
  upward symlink (commit 5) and a Composer pattern (rejected in
  Part-D notes) are the closest All-Might came to inheritance and
  both were reverted. The `/all-for-one` skill covers the only real
  use case (merge N → 1) without an inheritance taxonomy.

- **agentic-stack harness pluralism** is achieved with per-adapter
  config-read + a shared `.agent/` backend. The cost is that the
  adapter shims read **platform-native** config (`CLAUDE.md`,
  `.cursor/rules/`, `.windsurfrules`, `gemini.md`) and the user
  must not author conflicting state in either place. All-Might
  rejects this premise: `.claude/` is a **generated mirror** of
  `.opencode/`, not a parallel-edited source. The dual-platform
  invariant test (`test_capability_manifest.py`) is the
  enforcement; OMO's hook-bridge is the cautionary tale (CLAUDE.md
  §"Interaction with `oh-my-opencode`").

### 4.d Implementation

- agentic-stack uses **Python** for shared backend + **`adapter.json`**
  manifests; All-Might already centralises plugin/adapter shape in
  `PLUGIN_MANIFEST` (`core/plugin_telemetry.py`) — the same role,
  with stricter coherence tests. Implementation-wise All-Might is
  ahead.
- OpenGAP uses **JSON Schema** under `spec/schemas/`. All-Might has
  no JSON Schema for `personalities.yaml`. Cost to add: small.
  Value: catches the same class of manifest-drift bugs OpenGAP
  catches.
- agentic-stack's `auto_dream.py` writes a candidate-pool jsonl;
  OpenGAP writes audit reports. Both add an artifact lifecycle
  (clean-up, retention) that becomes its own maintenance burden.

### Pitch for the two "Propose" items

I owe a sell on these — the brief says *"convince me to invest in
new feature development."* Both are sub-half-day items, both pay
for themselves on the first failure, both expand zero new surface
area:

- **#1 (`init --dry-run`)**: the current re-init path is
  forward-only — the agent commits to writing, then inspects
  `conflicts.yaml`, then has to undo if it was wrong. Adding a
  no-op mode is the diff equivalent of `git diff` before `git add`
  — and most users already expect `--dry-run` on any
  filesystem-mutating CLI. The risk profile is tiny because the
  dry-run branch only *skips* writes; it doesn't add new write
  paths.

- **#2 (`framework_version`)**: a `Part-D → Part-E` migration is
  inevitable. When it happens, the binary will need to know
  whether a workspace is on the old layout. Today the only signal
  is "the `migrate` command happens to do the right thing or it
  doesn't" — there is no fail-fast. A single integer in the
  manifest is the cheapest possible integrity check; without it,
  the first user to hit a Part-E layout on a Part-D workspace gets
  a half-applied write with file-by-file marker checks unable to
  spot the layout-level mismatch.

Anything past #2 in the table fails the simplicity bar or has no
real evidence of pain. Today's deliberate non-action is itself the
answer to the brief: "if there is no clear gain or it violates
simplicity, don't do it."
