## 2. Panoramic snapshot — moai-adk

- **Pitch / thesis**: "**Harness Engineering** — design the
  environment for AI agents rather than writing code directly.
  Human steers, agents execute." Codifies *how to start work*,
  not *who is doing it*.
- **Agent composition**: 8 **framework-shipped** specialists —
  4 Manager (spec, develop, docs, git) + 2 Evaluator
  (plan-auditor, sync-auditor) + 1 Builder (builder-harness) + 1
  Anthropic Explore. **Dynamic teammates** (researcher, analyst,
  architect, implementer, tester, designer, reviewer) are spawned
  *at runtime* from role profile YAMLs, not pre-registered.
- **Workflow = SPEC-first triad**: `/moai plan` (EARS-format SPEC
  generation) → `/moai run SPEC-XXX` → `/moai sync` (docs + PR).
  Run mode auto-selects: **TDD** (`RED → GREEN → REFACTOR`) when
  coverage ≥10%, **DDD** (`ANALYZE → PRESERVE → IMPROVE`) when
  <10%.
- **Memory = `progress.md` cross-session continuity + Context Map
  + `.moai/logs/task-metrics.jsonl`**. *No per-agent memory, no
  vector retrieval, no per-role isolation.* Single shared turn log.
- **Install scope = user-global Go binary** (zero deps); `moai
  init` writes project-level `.moai/config/sections/*.yaml`,
  `.moai/specs/SPEC-XXX`, `.claude/skills/`, `.claude/commands/`,
  `.claude/rules/`, `progress.md`.
- **Multi-model Agent Teams** (the novel mechanism):
  - **CC**: Claude leader + Claude workers.
  - **GLM**: GLM 5 everywhere (~70 % savings).
  - **CG** (default): **Claude leader + GLM workers via tmux pane
    isolation** (~60 % savings). "Session-level env-var isolation"
    keeps the two providers from contaminating each other's
    context.
  - Triggered automatically for "complex projects" (≥3 domains,
    ≥10 files, complexity score ≥7).
- **Worktree-based parallelism** for batch jobs (`/batch`).
- **Korean-first community** — many issues filed in Korean, deep
  Windows compatibility focus (probably more Windows users than
  US-based projects see).

## 3. Panoramic snapshot — opencode-mem

- **Pitch**: persistent memory layer for OpenCode coding agents.
- **Storage = SQLite (source of truth) + USearch in-memory vector
  index** (rebuilt from SQLite on restart). Fallback to exact
  vector scan if USearch breaks.
- **Memory tagging = canonical project hash**
  (`opencode_project_<sha16>` derived from git remote URL or
  filesystem path) + canonical user hash
  (`opencode_user_<sha16>` from `git config user.email`).
  Per-project isolation is automatic — no manual `init` required
  per project.
- **Capture paths**:
  - **Auto-capture** via configured AI provider (extracts semantic
    snippets from agent activity when `autoCaptureEnabled: true`).
  - **Manual**: `memory({ mode: "add", ... })` tool call.
  - **Chat injection**: configurable, `maxMemories: 3` per turn,
    `injectOn: first`.
- **Retrieval = vector RAG**: 12+ local embedding models (default
  `Xenova/nomic-embed-text-v1`); selective per-turn injection
  (cap = 3) rather than full dump.
- **Install scope = user-global plugin** registered in
  `~/.config/opencode/opencode.json`. Storage at
  `~/.opencode-mem/data`, config at
  `~/.config/opencode/opencode-mem.jsonc`.
- **Web UI** at `http://127.0.0.1:4747` for browsing + manual
  retrieval.
- **No role / persona / personality concept.** Identity granularity
  = (project hash) × (user hash). Profile learning aggregates
  behavioral patterns across projects, not per-role contexts.
- **Maintainer succession risk** (#79). Future velocity uncertain;
  citing as design comparison, not as dependency.

## 4. Four-axis comparison

### a) Design philosophy

| | All-Might | moai-adk | opencode-mem |
|---|---|---|---|
| Central abstraction | **Role** (user-defined personality) × capability | **SPEC document** + dynamic teammate role profile | **Memory shard** keyed by `(project_hash, user_hash)` |
| What it organizes | *Identity* — "who is acting" | *Process* — "how the work flows: plan → run → sync" | *Recall* — "what was said before" |
| Who defines the roles | The user (`stdcell_owner`, `pll_owner`) | Framework ships managers; teammates spawned dynamically from role profile YAMLs | n/a |
| Per-role memory isolation | **✓** | ✗ (`progress.md` global) | ✗ (project-scoped, identity-blind) |
| Install scope | Project | **User-global binary + per-project init** | **User-global plugin** |
| Multi-model orchestration | ✗ (defers to host agent) | **✓ (CC/GLM/CG modes via tmux)** | ✗ |
| Memory medium | markdown/YAML + git mirror | `progress.md` + jsonl logs | SQLite + vector index |

**Two divergences worth naming:**

1. **moai-adk solves an orthogonal problem.** It codifies *what to
   do first* (SPEC), *how to execute* (TDD/DDD), *how to wrap up*
   (sync). All-Might codifies *who is doing it*. These compose
   cleanly — an All-Might personality could ship a `/plan-spec`
   skill — but moai is **not** a competitor on the role axis.
2. **opencode-mem stops at project-level isolation.** Per-role
   memory isolation remains structurally absent in every mainstream
   competitor: SuperClaude (stateless, delegates), Serena
   (project/topic, not identity), moai (`progress.md` global),
   opencode-mem (project hash, not role). The All-Might moat keeps
   widening — same observation as 2026-05-25 §4d, third independent
   confirmation.

### c) Mechanism design — genuinely distinct ideas

Four worth naming:

1. **moai-adk's EARS-format SPEC workflow + auto TDD/DDD** —
   codifies a workflow that capable models already do informally.
   For weak/air-gap models the explicit scaffold *might* matter.
   But: this is the same "mechanise into noise" risk CLAUDE.md
   already calls out for SuperClaude triggers (Q3, 2026-05-25). A
   SPEC document is heavier than a Triggers field.
2. **moai-adk's CG mode (Claude leader + GLM workers via tmux)** —
   cost-aware model routing inside a *single* project. The tmux
   pane env-isolation trick is concrete and clever. But it lives
   at a layer *below* the host agent: All-Might is consumed *by*
   Claude Code / OpenCode; routing the underlying provider is the
   host's job. Building this into All-Might means duplicating the
   host's model-routing logic, breaking the dual-platform
   invariant, and inheriting tmux as a runtime dep on the user's
   machine. The cost win is real (60-70%) but goes to projects
   that don't already have a host agent — a different product
   shape.
3. **moai-adk's dynamic teammates (role-profile-on-demand)** —
   manager agents are shipped specialists; teammates
   (researcher/architect/implementer) are spawned per task from
   role profile YAMLs. This sidesteps both the SuperClaude
   "global static persona pool" problem *and* the All-Might "user
   must register the personality upfront" problem. But: only
   useful when the role pool is generic SWE — moai's roles are
   generic-software-engineering specialists. All-Might's roles
   are *domain* roles (`stdcell_owner`, `pll_owner`) where dynamic
   instantiation makes no sense — you can't conjure a domain
   expert from a YAML profile.
4. **opencode-mem's project-tag identity (`opencode_project_<sha16>`
   from git remote)** — derives project identity automatically, no
   manual init. Lower friction than All-Might's explicit
   `personalities/` directory. But it collapses identity to
   *project*, losing the per-role-within-project granularity that
   *is* All-Might's whole point.

### d) Implementation details — two regressions worth checking

- **moai-adk #474: Claude Code hook input JSON format mismatch.**
  Claude Code (in moai's report, version 2.1.71, March 2026) was
  observed sending nested camelCase: `eventType`, `toolName`,
  `toolInput`, `session.id`. moai-adk's parser expected flat
  snake_case (`hook_event_name`, `tool_name`, `tool_input`,
  `session_id`). **All hooks silently failed** ("missing required
  field session_id"). **All-Might reads
  `payload.get("hook_event_name")` in
  `claude_bridge.py:113/178/248`** — the same flat-snake-case
  contract. If moai's report is accurate for the current Claude
  Code release, All-Might hooks are silently no-op'ing today.
  Verification cost: ≤30 min — run one Python hook with a sample
  Claude-Code input payload, see whether `hook_event_name`
  resolves or defaults to `"UserPromptSubmit"`. This is exactly the
  silent-drift class `tests/test_claude_bridge.py::TestHooksRunCleanly`
  is supposed to catch.
- **moai-adk #25: Windows `process.stdin.on('end')` hang.** Their
  `.cjs` hooks waited on stdin EOF; on Windows the `'end'` event
  never fires and the hook hangs forever. All-Might's Python hooks
  use `json.load(sys.stdin)` (Python's stdin EOF detection works
  on Windows) — verified locally in
  `core/claude_bridge.py:110, 175, 245` and
  `capabilities/memory/initializer.py:1916, 2161`. **Not affected.**
  No-action: log so a future audit doesn't re-investigate.

## 6. Investment verdict

**No new feature clears the simplicity bar today.** Four of five
candidate features are rejected outright; SPEC-first scores **2/5**
and is deferred behind a future trigger (first observed
weak-model failure-to-plan). The single actionable item is **Q6**,
a 30-minute hook-contract audit that is *defensive*, not a feature.

The two deep-dive competitors reinforce, not threaten, All-Might's
existing thesis:

1. **moai-adk operates on an orthogonal axis** (process discipline,
   not knowledge organization). It validates that a structured
   workflow is in demand, but doesn't compete with the role × capability
   shape — and the per-role memory gap is still unsolved at scale.
2. **opencode-mem proves vector RAG is a legitimate memory primitive
   at scale** (956 stars, 79 substantive issue reports) — but it
   *also* proves the SQLite-vector stack pays in compatibility cost
   (3 distinct vendor breakages: Zod schema, hnsw-wasm IDBFS,
   sqlite-vec dlopen — exactly the maintenance tail the All-Might
   CLAUDE.md "No SQLite memory store" non-goal protected against).
   The All-Might tier choice — L1 markdown / L2 markdown TOC / L3
   SMAK vector — keeps vector cost paid only where vector is the
   right primitive.

The widening moat continues to be **per-role memory isolation**.
Third independent competitor (moai joins SuperClaude + Serena +
opencode-mem) confirmed that *no published competitor* expresses
identity-scoped memory. Each one collapses identity to either
"global / stateless" or "per-project." This is now the property to
name in marketing copy.

## 8. Lessons captured for next reconnaissance

- **Three independent competitors converging on the same gap is a
  marketing signal, not just a design signal.** Per-role memory
  isolation as the moat now has SuperClaude + Serena +
  opencode-mem + moai all confirming the absence on their side.
  Worth a one-paragraph README section that *names* this gap so
  prospective users can immediately tell whether they need it.
- **"Different mechanism" check still pays.** moai's SPEC-first is
  the third 2026 example (after Serena's LSP and SuperClaude's
  Triggers) of a competitor mechanism that *looks* like it should
  be adopted but solves an adjacent problem. Default-reject and
  require evidence-of-fit.
- **Bug reports in competitor repos are a free contract test.** moai
  #474 surfaced a Claude Code hook input shape that *might* be
  affecting All-Might too — and the discovery cost was zero (we
  were already reading their issue list for real-signal
  filtering). Worth keeping the "scan top-3-real-signal repos'
  bug lists for shape regressions affecting us" pattern explicit
  in the next routine.
- **SQLite-vector stacks pay a compatibility tax.** opencode-mem's
  79 issues include 3 distinct vendor-stack breakages in 6 months
  (Zod, hnsw-wasm IDBFS/MEMFS, sqlite-vec dlopen). All-Might
  punting that tax onto SMAK (which lives in its own repo with its
  own test surface) and keeping the framework-side stack to
  markdown + YAML continues to pay off.
- **Maintainer-stepdown signal is information.** opencode-mem's
  #79 ("Maintainer Wanted") is a future-velocity flag. When citing
  a competitor as design reference, log who maintains it and
  whether they're still active. Cite from a snapshot, not from
  a project you're forecasting will stay alive.
