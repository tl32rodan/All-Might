## 2. Panoramic snapshot — github/spec-kit

- **Pitch**: SDD "inverts traditional software creation: rather
  than treating specifications as throwaway scaffolding, it makes
  them executable artifacts that directly generate working
  implementations." Intent precedes mechanism; specs are the
  *primary* artifact, code is downstream.
- **Workflow = 5 mandatory + 3 optional slash commands**:
  `/speckit.constitution` → `/speckit.specify` → `/speckit.plan`
  → `/speckit.tasks` → `/speckit.implement`. Optional:
  `/speckit.clarify`, `/speckit.analyze`, `/speckit.checklist`.
- **Disk layout**: `.specify/memory/constitution.md`,
  `.specify/templates/`, `.specify/scripts/`, and per-feature
  `specs/<NNN-feature-id>/{spec.md, plan.md, tasks.md, data-model.md,
  contracts/, research.md, quickstart.md}`.
- **Constitution = 9 articles** as "immutable architectural DNA".
  Examples: Article III "no implementation code shall be written
  before unit tests are written, validated, confirmed to FAIL";
  Article I "every feature MUST begin as a standalone library";
  Article VII "max 3 projects initially, more requires justified
  rationale".
- **Multi-agent**: works with ~30 agents via `--integration` flag.
  Slash commands or skills mode where the agent supports it.
- **Identity unit = feature spec.** No persistent per-role memory.
  No cross-feature memory layer. Constitution is the only
  cross-feature persistent doc.

## 3. Panoramic snapshot — gotalab/cc-sdd

- **Pitch**: specs are "**contracts between system parts, not
  master commands**. Code remains the source of truth; specs make
  boundaries explicit so humans and agents work in parallel."
  Strictly weaker claim than spec-kit's "specs are executable
  artifacts that generate code."
- **Workflow = complexity-routed entry**: `/kiro-discovery` picks
  one of 5 paths (extend existing spec / implement directly without
  spec / single spec / decompose into multiple / mixed
  decomposition). Then: `discovery` → `spec-init` → `requirements`
  (EARS format) → `design` (Mermaid + file-structure plan) →
  `tasks` → `/kiro-impl` (autonomous, per-task review, auto-debug).
- **Disk**: `{{KIRO_DIR}}/brief.md`, optional `roadmap.md`,
  `requirements.md`, `design.md`, `tasks.md`, `settings/`.
- **Agent surface = 17 skills per platform**, loaded on-demand;
  subagents spawn via each platform's native primitive. Legacy
  command modes deprecated.
- **Steering**: separate `kiro-steering` step precedes discovery.
  How it stores state is not documented; mismatch with #162
  (unreliable phase tracking) suggests this is a soft layer.
- **Identity unit = feature spec.** Same as spec-kit.

## 4. Comparison axes

### a. Slicing — the orthogonality insight

| | All-Might slices by | spec-kit / cc-sdd slice by |
|---|---|---|
| Primary unit | **role** (personality) | **feature** (spec) |
| Persistent state | per-role memory (L1/L2/L3) | per-feature artifacts (`specs/NNN-*/`) |
| Cross-cut | "which personality is active" | "which feature is in progress" |
| Re-use across projects | `/one-for-all` bundle a personality | constitution.md replicated, specs not |

These axes are **orthogonal, not competing.** A `code_reviewer`
personality and a `stdcell_owner` personality might *each* run a
spec-kit-style flow on different features, and the right place for
SDD artifacts is *inside the coding agent's current turn*, not in
All-Might's persistent layer. SDD describes a *transient design
session*; All-Might describes *durable identity*.

### b. Doc-layer architecture — vindication of a non-goal

spec-kit issue #609 ("Claude.md vs constitution.md") has 24
reactions and is unresolved. Users genuinely cannot tell which
context layer (`CLAUDE.md` vs `.specify/memory/constitution.md`)
governs which decisions. This is the **exact failure mode** that
All-Might's CLAUDE.md non-goal explicitly avoids:

> One root `AGENTS.md`, composed from per-personality `ROLE.md`.
> [...] No emitted body may contain a literal personality name.

All-Might has *three* doc layers (`MEMORY.md` = project map +
default callout, `AGENTS.md` = high-level WHAT, `ROLE.md` =
per-personality), each with a single, mutually-exclusive job.
spec-kit adds `constitution.md` as a *fourth* layer alongside
the host's `CLAUDE.md`, and the boundary is unclear to users.
**Adopting `constitution.md` would re-create the bug.**

### c. Idempotency — already won

spec-kit #916 ("specify init --here overwrites user files") is
**precisely the marker-discipline regression** All-Might
already pinned with `ALLMIGHT_MARKER_*` + the
`if staging: return` rule in `_install_globals` (CLAUDE.md
*Interface Isolation* §). Users land on this every time they
re-run init. We are not at risk of regressing here; the test
`test_reinit_staging_path_still_writes_claude_hooks` and
related already gate it.

### d. Context-budget convergence

spec-kit #752 (25 reactions): users explicitly ask for subagent
isolation to fight context bloat from spec workflows. This is the
same pressure All-Might's *Skill / Command Body Style Guide*
addresses with its hard size budget (`tests/test_skill_body_size.py`).
Independent ecosystems converging on "agent context budget is a
real, recurring pain point" is supporting evidence that the budget
discipline was worth its complexity.

### e. Phase tracking — a regression we're spared

cc-sdd #162 shows the "phase: tasks-generated → implementation-
complete" state machine flips inconsistently because agents
update it free-form. All-Might deliberately avoids state machines
on top of agent output: `read_onboard`/`write_onboard` are
*deterministic* (Python writes, agent doesn't), and `/remember`/
`/reflect` are stateless command bodies — there is no phase to
mis-update. The decision to keep state in YAML written by Python,
not in prose-flag-fields, holds up here.

## 8. Lessons captured for next reconnaissance

- **Orthogonal trends are easy to mis-read as competing.** When a
  loud new wave (SDD, 111k stars) appears, the reflex is to ask "do
  we need to adopt this?" The sharper question is "does it slice the
  problem space along the same axis we do?" SDD slices by *feature*,
  All-Might by *role*. They compose; they don't compete. Naming the
  axis is the first move every time.
- **Three independent vindications in one pass is a strong signal
  to *not* change anything.** Doc-layer non-goal (issue #609),
  idempotency discipline (#916), state-machine avoidance (#162),
  and context-budget discipline (#752) all surfaced in other
  projects' issue trackers as live, unresolved pain. When the
  ecosystem keeps re-hitting the bugs you already pinned with
  tests, the right move is to *not* touch the gates that pin them.
- **The "is X our competitor?" question deserves a written answer
  in the README.** Saves time on every future reconnaissance, and
  reduces the chance of someone (a contributor, a future agent
  thinking it's helpful) bolting on a spec-kit-shaped vertical
  inside All-Might. The Serena scope-boundary note (2026-05-25 §6)
  was the same shape; bundle them.
- **Skip-list grows faster than the candidate-set.** Maintaining
  the explicit skip list at the top of each entry is paying off —
  half the trending list was already triaged. Keep it.
