## 3. What Anthropic shipped (the new baseline)

Three tiers, picked by the user per task:

| Tier | Where it runs | Persistence | Min interval | State on disk |
|---|---|---|---|---|
| **`/loop` (in-session)** | Local, in CC session | Restored via `--resume` ≤7d | 1 min | Session JSON (not file-system) |
| **Desktop scheduled tasks** | Local, app daemon | Survives restarts; 7-day catch-up window | 1 min | `~/.claude/scheduled-tasks/<task>/SKILL.md` |
| **Routines (cloud)** | Anthropic infra | Always-on | 1 h | Cloud config |

Underlying mechanism for `/loop`: 5-field cron via three tools —
`CronCreate` / `CronList` / `CronDelete`. Cron expressions are
standard vixie-cron. Disable flag exists
(`CLAUDE_CODE_DISABLE_CRON=1`). Jitter, expiry, and 50-task cap are
all built in. **User's hypothesis confirmed verbatim: it's
cron + tools.**

**Critical isomorphism**: a Desktop scheduled task is stored as a
`SKILL.md` with YAML frontmatter (`name`, `description`) and the
prompt as body. Anthropic's own design declares: *a scheduled task
is a skill*. The runtime is cron; the *content* is a skill.

## 4. Four-axis comparison

### a) Design philosophy

| | All-Might (status quo) | Anthropic `/loop` + Desktop | `background-agents` |
|---|---|---|---|
| Abstraction | "out of scope — weekly cron is enough" | task = SKILL.md + cron expr | hosted UI form + cron |
| Trigger taxonomy | none | cron only | cron + webhook + Sentry + GitHub + Linear |
| Failure handling | none | jitter + 7-day expiry + missed-run catch-up | 90-min timeout + auto-pause after 3 failures |
| Granularity | external | per-prompt task | per-prompt task |

### b) Feature coverage gap

| Concern | Currently handled how | Could `/loop` cover it? |
|---|---|---|
| L3 incremental ingest | Marker file + `session.created` async spawn (no cron) | Already solved — irrelevant. |
| Curator audit of shared memory | "out of scope" / external cron suggested | `/loop 1w /curator-audit` — yes. |
| Plugin observability roll-up | `allmight plugin status` on demand | `/loop 168h /allmight-plugin-status` — yes. |
| Sidecar drift sweep after refactor | Manual via skill | Manual is correct here — not periodic. |
| L3 size threshold warning (`plan.md`) | None | One-shot self-reschedule possible via Desktop's `update_scheduled_task`. |

**No periodic need that exceeds `/loop` capabilities.** All
candidates are single-prompt, low-frequency, non-cross-cutting.

### c) Mechanism

- `/loop`'s `CronCreate`/`CronList`/`CronDelete` is exactly what
  the user described: "cron + utils". Reproducing it is pure
  duplication.
- Desktop scheduled tasks even reuse the SKILL.md format
  All-Might already emits. No translation needed.

### d) Implementation

- Each capability *can* declare its own recommended cadence in
  prose, but the run-time delivery is owned by Claude Code, not
  All-Might. No `core/` change required.

## 6. Proposal — "scheduling" as a single skill, not a capability

### 6.1 What to ship

A *single* SKILL.md at project-globals scope, dropped during
`allmight init`. No new Python code, no new CLI command, no new
plugin, no new template directory hierarchy.

**Path**: `.opencode/skills/scheduling/SKILL.md`
(mirrored to `.claude/skills/scheduling/` by the existing symlink).

**Frontmatter** (matches the 2-field contract used by
`anthropics/skills` and All-Might's existing skills):

```yaml
---
name: scheduling
description: >-
  How to set up periodic All-Might tasks (curator audit, plugin
  status roll-up, L3 sanity checks) using Claude Code's /loop or
  Desktop scheduled tasks. Defers runtime to Claude Code; this
  skill only encodes which cadences are appropriate.
---
```

**Body sections** (≤120 lines total):

1. *When to invoke* — user says "automate", "schedule", "every
   week", "remind me to audit", etc.
2. *Decision tree* — pick `/loop` (in-session, ≤7d), Desktop
   (persistent), or external cron (out of allmight scope) based
   on persistence + machine-on requirements.
3. *Catalogue of recommended cadences* — table mapping each
   All-Might concern to a suggested prompt + cron expression +
   chosen tier, e.g.:

   | Concern | Suggested prompt | Cadence | Tier |
   |---|---|---|---|
   | Curator audit | `/curator-audit` (once it exists) or "review `_inbox/`" prose | weekly | `/loop 1w` while present, Desktop for unattended |
   | Plugin observability | "`allmight plugin status` and summarize" | weekly | `/loop 168h` |
   | L3 size sanity | "warn if `personalities/*/memory/journal/` >50MB" | monthly | `/loop 30d` or Desktop |

4. *Anti-patterns* — don't schedule things that already have a
   reactive closure (L3 ingest); don't ship cron from
   `allmight init` (always user-opt-in).

### 6.2 Where the skill lives in the repo

Two minimal touch points, matching the existing hub-skill pattern:

- `src/allmight/hub/templates/skills/scheduling/SKILL.md` (new
  file, the only "real" deliverable)
- one of the existing initializers' `_install_globals` paths
  emits it. The memory capability is the natural home because the
  highest-value cadences (curator audit, L3 size) are
  memory-domain. **Alternative**: drop it under hub level if a
  hub install is required to see it across workspaces — decide at
  implementation time based on where the existing
  `self-improving` skill lives.

No `core/` change. No new CLI command. No new plugin. No mirror
work (the existing `.claude/skills` symlink handles dual-platform
visibility for free).

### 6.3 Estimated cost

| Item | LOC |
|---|---|
| `SKILL.md` body | ~120 |
| 4-line wrapper (`build_scheduling_skill_md()`) | 4 |
| One install-call wiring | 1 |
| Test (smoke: skill file present after init) | ~10 |
| **Total** | **~135 LOC, no infrastructure** |

Compare with hypothetical *capability*: ~500–1000 LOC, new
Python module, new `/schedule list/add/rm` commands, new plugin,
mirror to Claude Code, cron-expression validation, persistence
file under `.allmight/scheduled.yaml`, GC logic. All to wrap
something Anthropic already shipped.

### 6.4 Why **not** capability

A capability has install code, a status function, a template,
optional CLI flags, and a lifecycle. Scheduling has none of these
needs — there is no per-personality scheduling state, no
infrastructure to install, no template to instantiate, and no
lifecycle distinct from "the user added a `/loop` command".
Calling it a capability would mean the name is doing work the
implementation doesn't. **Skill is the honest abstraction.**

### 6.5 What we *don't* do today

- No `/schedule` slash command (would re-export `/loop`).
- No `.allmight/scheduled.yaml` (Anthropic owns task state).
- No allmight-side cron parser.
- No background daemon (already
  `CLAUDE_CODE_DISABLE_CRON=1`-respecting `/loop`).
- No tie-in to `update_scheduled_task` MCP tool until a concrete
  use case shows up.

## 7. OpenCode-aware revision — the primary surface

**Reframe**: All-Might's canonical surface is `.opencode/`, not
`.claude/`. The CC three-tier analysis above is necessary
background but is not the deciding axis. OpenCode is where the
gap actually lives.

### 7.1 What OpenCode currently provides

| | OpenCode | Claude Code |
|---|---|---|
| Native scheduling | **None.** [Issue #11232](https://github.com/anomalyco/opencode/issues/11232) open, assigned to jlongster, no ETA, no comments yet. | `/loop` + Desktop scheduled tasks + Routines (v2.1.72+). |
| Community solution | [`different-ai/opencode-scheduler`](https://github.com/different-ai/opencode-scheduler) (361★, single `src/index.ts`, MIT, TypeScript). | n/a (Anthropic ships it). |
| Workaround | OS cron / launchd / systemd. | OS cron / launchd / systemd. |

For an All-Might project run under OpenCode, **the runtime gap is
real**. Yesterday's "skill, not capability" decision was correct
for Claude Code but incomplete for the OpenCode reality.

### 7.2 `opencode-scheduler` — what it actually is

- One TypeScript file (`src/index.ts`) that registers 11 MCP tools
  via `@opencode-ai/plugin`: `schedule_job`, `list_jobs`, `get_job`,
  `update_job`, `delete_job`, `cleanup_global`, `run_job_now`,
  `get_logs`, `get_skill`, `install_skill`, `get_version`.
- Backends: launchd (macOS), `systemd --user` (Linux),
  `schtasks` (Windows), `crontab` (Linux fallback).
- State: `~/.config/opencode/scheduler/scopes/<scopeId>/jobs/*.json`
  where **`<scopeId>` is the normalised absolute workdir path**.
- Supervision: per-OS scheduler → Perl supervisor → `opencode run`
  with `OPENCODE_PERMISSION=deny` so the job doesn't hang on a
  permission prompt.
- Critical gap for All-Might: **scope is workdir, not personality.**
  All personalities under one project share a scope.

### 7.3 Three implementation tiers — per-personality scheduling

| Tier | What All-Might owns | Runtime | Difficulty | Est. LOC |
|---|---|---|---|---|
| **T1 — skill only** | A SKILL.md teaching the agent to use `opencode-scheduler`'s MCP tools with a `<personality>-<task>` slug convention | 100% deferred to `opencode-scheduler` | ★☆☆☆☆ | ~150 |
| **T2 — declarative state + apply** | `personalities/<p>/scheduled/<task>.md` (declarative source of truth) + `allmight schedule apply` (materialises to scheduler state) + skill | `opencode-scheduler` invoked via subprocess/MCP | ★★★☆☆ | ~400–600 |
| **T3 — own plugin** | Full opencode plugin, fork or rebuild | All-Might-owned launchd/systemd/cron wrapper | ★★★★★ | ~1500–2500 + ongoing per-OS maintenance |

### 7.4 Why **not** T3

- Re-implements what `opencode-scheduler` already does (361★,
  maintained, MIT). Net contribution: zero.
- Fails CLAUDE.md's bar: *"A new capability is justified only when
  it has its own data dir, its own skills/commands, and a meaningful
  uninstall semantics."* T3 invents a runtime we already have via
  third-party plugin — no novel infrastructure justified.
- OpenCode native scheduling (#11232) may land within months;
  we'd be writing throwaway code.
- Per-OS maintenance (launchd plist XML, systemd timer units,
  `schtasks` quoting) is exactly the
  *"Discipline When Generating Third-Party Integrations"* trap.
- Claude Code mirror burden: every additional plugin becomes a
  hand-mirrored Python hook tracked by `PLUGIN_MANIFEST`. Cost
  permanent.

### 7.5 Why **T2 over T1**

T1 alone solves the *runtime* gap but leaves three All-Might-shaped
gaps:

1. **Per-personality identity.** Schedules are part of a
   personality's behaviour — "stdcell_owner audits sidecars every
   Monday" is identity, not infrastructure. Slug convention alone
   (T1) keeps schedules in `~/.config/opencode/scheduler/` outside
   the project, divorced from the personality dir.
2. **Travel with `/one-for-all`.** If schedules are out-of-project,
   they don't ship with bundles. Exporting `stdcell_owner` to
   another machine loses its weekly cadence.
3. **Discoverability via `allmight list`.** A user opening a
   project should see the scheduled work declared in-tree, not
   buried in `~/.config`.

T2 solves all three by **owning the declaration** in
`personalities/<p>/scheduled/<task>.md` and **delegating the
runtime** to `opencode-scheduler` (or its eventual replacement).
This is isomorphic to the `database/` capability pattern: declare
in `personalities/<p>/database/<ws>/config.yaml`, delegate to
SMAK runtime.

### 7.6 Concrete T2 design

```
personalities/<p>/scheduled/<task>.md           ← declaration (SKILL.md shape)
                                                  ALLMIGHT_MARKER_MD; frontmatter
                                                  carries cron + permission +
                                                  optional timeout

src/allmight/capabilities/schedule/
├── __init__.py                                 ← TEMPLATE; _install_globals
│                                                 emits skill, _install creates
│                                                 empty personalities/<p>/scheduled/
├── initializer.py                              ← writes SKILL.md template +
│                                                 per-personality dir
├── templates/skills/scheduling/SKILL.md        ← agent's how-to
└── apply.py                                    ← diff: walk all
                                                  personalities/*/scheduled/*.md,
                                                  compare with
                                                  ~/.config/opencode/scheduler/
                                                  scopes/<project-scope>/jobs/*.json,
                                                  reconcile via opencode-scheduler
                                                  MCP tools or its files directly
                                                  (slug = "<personality>-<task>")

CLI additions:
  allmight schedule apply              ← reconcile repo → runtime
  allmight schedule status             ← read-only view; pretty-print state
  (no add/rm/edit — declarative edits happen in-tree)

Plugin additions:
  .opencode/plugins/schedule-sync.ts   ← session.created marker check; if
                                         any personalities/*/scheduled/*.md
                                         is newer than .allmight/schedule.pending,
                                         touch the marker; main session prints
                                         "schedules need apply" once.
                                         Same closure pattern as L3 ingest.
  Claude Code mirror: NONE — CC users have Anthropic Desktop tasks
  for persistent scheduling; PLUGIN_MANIFEST entry has
  claude_code_mirror: None.
```

Heartbeats, markers, ALLMIGHT_MARKER_MD, `_install_globals` /
`_install` split — all reuse existing patterns. No new conceptual
machinery.

### 7.7 Implementation difficulty — concrete breakdown

| Item | LOC | Risk |
|---|---|---|
| `__init__.py` TEMPLATE wiring (mirror memory's shape) | ~80 | Low — pattern duplicated three times already. |
| `initializer.py` (writes globals + per-personality `scheduled/` dir + SKILL.md) | ~120 | Low. |
| `apply.py` (diff + reconcile) | ~200 | **Medium** — has to call `opencode-scheduler` either via its MCP tools or by writing to its state files. Both are unstable contracts. |
| `templates/skills/scheduling/SKILL.md` (agent how-to) | ~140 | Low. |
| `.opencode/plugins/schedule-sync.ts` (marker-file closure) | ~80 | Low — direct lift of memory-history pattern. |
| CLI commands `apply`, `status` in `cli.py` (NOTE: violates "cli.py is closed against templates" — must register via `cli_options` instead, or land them as `allmight memory`-style subcommands owned by the schedule capability) | ~60 | Low. |
| Tests (smoke + apply idempotency + marker behaviour) | ~150 | Medium — apply tests need to mock `opencode-scheduler` state. |
| `/one-for-all` integration (auto-include `scheduled/`) | ~20 | Low — OFA already iterates per-personality dirs. |
| **Total** | **~850 LOC** | **Medium overall.** |

### 7.8 Hardest specific question

**How does `apply.py` reconcile?** Two options:

- **(a) Call `opencode-scheduler` MCP tools** via subprocess
  spawning an `opencode run` session. Heavy, slow, requires
  OpenCode runtime context.
- **(b) Write `~/.config/opencode/scheduler/scopes/.../jobs/*.json`
  directly** and trigger `opencode-scheduler` to pick them up.
  Faster, but couples All-Might to a third-party state format
  with no stability guarantee.

Recommendation: **(b) with an explicit "schema version check"** —
on first run, `apply` reads `opencode-scheduler`'s declared
package version (via `get_version`-equivalent file or
`package.json`) and refuses to write unless the version is within
a known-good range. If unsupported, print a clear error and abort.

This is the *"verify the API on one file before propagating"*
discipline applied to a third-party state format.

## 8. Investment verdict (revised, OpenCode-aware)

**Approved: T2 — schedule capability.** Total cost ~850 LOC,
spread across the existing capability template pattern. Justifies
the "new capability" bar because:

1. **Own data dir**: `personalities/<p>/scheduled/`.
2. **Own skill**: `scheduling` teaches the declarative-first flow.
3. **Meaningful uninstall**: `allmight remove <personality>` (when
   it ships) cleans cron entries via `schedule apply`.
4. **Travels with `/one-for-all` bundles** for free.
5. **Aligns with existing patterns**: declarative file + apply +
   marker-file closure all reused from L3 ingest / SMAK workspace.

**Rejected: T3 — own plugin.** Reimplements `opencode-scheduler`.
Fails the new-capability bar (no novel infrastructure). Owes
per-OS maintenance forever.

**Conditional alternative: T1 — skill only**, if the user
prefers the lighter footprint and is willing to:
(a) accept schedules don't travel with bundles,
(b) let agent manage slug discipline without Python enforcement,
(c) drop the `/one-for-all` schedule-portability story.

**Sequencing recommendation**:
1. *Today*: implement T1 (~150 LOC) — gets agent productive on
   `opencode-scheduler` immediately. The SKILL.md from T2 is the
   same content; T1 is a strict subset.
2. *After 1–2 weeks of T1 usage*: if users actually schedule work
   and want it portable, upgrade to T2 (~700 additional LOC).
3. *Watch issue #11232*: if OpenCode lands native scheduling,
   re-evaluate the entire stack. T2's `apply.py` becomes an
   adapter to native instead of `opencode-scheduler`.

**Deferred** (unchanged from earlier):
- `/curator-audit` slash command — wait for user demand.
- Self-rescheduling memory ingest — current marker-file closure
  is sufficient.

## 9. Lessons captured for next reconnaissance

- When the official platform ships a primitive (here `/loop` +
  `CronCreate`/`CronList`/`CronDelete`), the All-Might question
  collapses from "how do we build X?" to "what do we tell the
  agent to *use* X for?". The answer to the latter is almost
  always a skill.
- The Anthropic SKILL.md format is now the de-facto storage shape
  for task definitions (their own Desktop scheduled tasks store
  prompts that way). Continuing to emit SKILL.md keeps All-Might
  inside the mainstream.
- The user's "cron + utils" intuition was exactly right; it lets
  us skip 90% of the design work that a wrong-framing
  conversation would have produced.
