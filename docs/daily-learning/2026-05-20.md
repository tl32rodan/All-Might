## 3. What Anthropic shipped (the new baseline)

Three tiers, picked by the user per task:

| Tier | Where it runs | Persistence | Min interval | State on disk |
|---|---|---|---|---|
| **`/loop` (in-session)** | Local, in CC session | Restored via `--resume` ≤7d | 1 min | Session JSON (not file-system) |
| **Desktop scheduled tasks** | Local, app daemon | Survives restarts; 7-day catch-up window | 1 min | `~/.claude/scheduled-tasks/<task>/SKILL.md` |
| **Routines (cloud)** | Anthropic infra | Always-on | 1 h | Cloud config |

Underlying mechanism for `/loop`: 5-field cron via three tools —
`CronCreate` / `CronList` / `CronDelete`. Cron expressions are
standard vixie-cron. Disable flag exists
(`CLAUDE_CODE_DISABLE_CRON=1`). Jitter, expiry, and 50-task cap are
all built in. **User's hypothesis confirmed verbatim: it's
cron + tools.**

**Critical isomorphism**: a Desktop scheduled task is stored as a
`SKILL.md` with YAML frontmatter (`name`, `description`) and the
prompt as body. Anthropic's own design declares: *a scheduled task
is a skill*. The runtime is cron; the *content* is a skill.

## 4. Four-axis comparison

### a) Design philosophy

| | All-Might (status quo) | Anthropic `/loop` + Desktop | `background-agents` |
|---|---|---|---|
| Abstraction | "out of scope — weekly cron is enough" | task = SKILL.md + cron expr | hosted UI form + cron |
| Trigger taxonomy | none | cron only | cron + webhook + Sentry + GitHub + Linear |
| Failure handling | none | jitter + 7-day expiry + missed-run catch-up | 90-min timeout + auto-pause after 3 failures |
| Granularity | external | per-prompt task | per-prompt task |

### b) Feature coverage gap

| Concern | Currently handled how | Could `/loop` cover it? |
|---|---|---|
| L3 incremental ingest | Marker file + `session.created` async spawn (no cron) | Already solved — irrelevant. |
| Curator audit of shared memory | "out of scope" / external cron suggested | `/loop 1w /curator-audit` — yes. |
| Plugin observability roll-up | `allmight plugin status` on demand | `/loop 168h /allmight-plugin-status` — yes. |
| Sidecar drift sweep after refactor | Manual via skill | Manual is correct here — not periodic. |
| L3 size threshold warning (`plan.md`) | None | One-shot self-reschedule possible via Desktop's `update_scheduled_task`. |

**No periodic need that exceeds `/loop` capabilities.** All
candidates are single-prompt, low-frequency, non-cross-cutting.

### c) Mechanism

- `/loop`'s `CronCreate`/`CronList`/`CronDelete` is exactly what
  the user described: "cron + utils". Reproducing it is pure
  duplication.
- Desktop scheduled tasks even reuse the SKILL.md format
  All-Might already emits. No translation needed.

### d) Implementation

- Each capability *can* declare its own recommended cadence in
  prose, but the run-time delivery is owned by Claude Code, not
  All-Might. No `core/` change required.

|---|---|---|---|
| Q1 | Should `schedule` be a new All-Might **capability**? | **Reject.** | Would mean shipping cron infra (or wrapping it) + commands + a plugin. Anthropic's `/loop`+Desktop already provides cron + tools + state + observability. Reimplementing violates the simplicity principle outright. |

## 6. Proposal — "scheduling" as a single skill, not a capability

### 6.1 What to ship

A *single* SKILL.md at project-globals scope, dropped during
`allmight init`. No new Python code, no new CLI command, no new
plugin, no new template directory hierarchy.

**Path**: `.opencode/skills/scheduling/SKILL.md`
(mirrored to `.claude/skills/scheduling/` by the existing symlink).

**Frontmatter** (matches the 2-field contract used by
`anthropics/skills` and All-Might's existing skills):

```yaml
---
name: scheduling
description: >-
  How to set up periodic All-Might tasks (curator audit, plugin
  status roll-up, L3 sanity checks) using Claude Code's /loop or
  Desktop scheduled tasks. Defers runtime to Claude Code; this
  skill only encodes which cadences are appropriate.
---
```

**Body sections** (≤120 lines total):

1. *When to invoke* — user says "automate", "schedule", "every
   week", "remind me to audit", etc.
2. *Decision tree* — pick `/loop` (in-session, ≤7d), Desktop
   (persistent), or external cron (out of allmight scope) based
   on persistence + machine-on requirements.
3. *Catalogue of recommended cadences* — table mapping each
   All-Might concern to a suggested prompt + cron expression +
   chosen tier, e.g.:

   | Concern | Suggested prompt | Cadence | Tier |
   |---|---|---|---|
   | Curator audit | `/curator-audit` (once it exists) or "review `_inbox/`" prose | weekly | `/loop 1w` while present, Desktop for unattended |
   | Plugin observability | "`allmight plugin status` and summarize" | weekly | `/loop 168h` |
   | L3 size sanity | "warn if `personalities/*/memory/journal/` >50MB" | monthly | `/loop 30d` or Desktop |

4. *Anti-patterns* — don't schedule things that already have a
   reactive closure (L3 ingest); don't ship cron from
   `allmight init` (always user-opt-in).

### 6.2 Where the skill lives in the repo

Two minimal touch points, matching the existing hub-skill pattern:

- `src/allmight/hub/templates/skills/scheduling/SKILL.md` (new
  file, the only "real" deliverable)
- one of the existing initializers' `_install_globals` paths
  emits it. The memory capability is the natural home because the
  highest-value cadences (curator audit, L3 size) are
  memory-domain. **Alternative**: drop it under hub level if a
  hub install is required to see it across workspaces — decide at
  implementation time based on where the existing
  `self-improving` skill lives.

No `core/` change. No new CLI command. No new plugin. No mirror
work (the existing `.claude/skills` symlink handles dual-platform
visibility for free).

### 6.3 Estimated cost

| Item | LOC |
|---|---|
| `SKILL.md` body | ~120 |
| 4-line wrapper (`build_scheduling_skill_md()`) | 4 |
| One install-call wiring | 1 |
| Test (smoke: skill file present after init) | ~10 |
| **Total** | **~135 LOC, no infrastructure** |

Compare with hypothetical *capability*: ~500–1000 LOC, new
Python module, new `/schedule list/add/rm` commands, new plugin,
mirror to Claude Code, cron-expression validation, persistence
file under `.allmight/scheduled.yaml`, GC logic. All to wrap
something Anthropic already shipped.

### 6.4 Why **not** capability

A capability has install code, a status function, a template,
optional CLI flags, and a lifecycle. Scheduling has none of these
needs — there is no per-personality scheduling state, no
infrastructure to install, no template to instantiate, and no
lifecycle distinct from "the user added a `/loop` command".
Calling it a capability would mean the name is doing work the
implementation doesn't. **Skill is the honest abstraction.**

### 6.5 What we *don't* do today

- No `/schedule` slash command (would re-export `/loop`).
- No `.allmight/scheduled.yaml` (Anthropic owns task state).
- No allmight-side cron parser.
- No background daemon (already
  `CLAUDE_CODE_DISABLE_CRON=1`-respecting `/loop`).
- No tie-in to `update_scheduled_task` MCP tool until a concrete
  use case shows up.

## 7. Investment verdict

**Approved (lightweight): one SKILL.md.** Total cost ~135 LOC.
Adds: (a) discoverability — agent knows what All-Might-specific
work is worth scheduling, (b) sensible-default cadences captured
in-tree, (c) explicit alignment with the Anthropic three-tier
matrix. Subtracts: nothing from existing surface; no new tool, no
new command, no infrastructure.

**Rejected (heavyweight): scheduling capability.** Reimplements
what Anthropic ships in v2.1.72+. Adds zero coverage. Violates
simplicity in every measurable dimension (LOC, plugin count,
command count, mirror work).

**Deferred**:
- `/curator-audit` slash command — wait for a user request; the
  skill can reference it in the abstract for now and we backfill
  the command when there's actual demand.
- Self-rescheduling memory ingest via `update_scheduled_task` —
  not until the existing marker-file closure proves insufficient.

## 8. Lessons captured for next reconnaissance

- When the official platform ships a primitive (here `/loop` +
  `CronCreate`/`CronList`/`CronDelete`), the All-Might question
  collapses from "how do we build X?" to "what do we tell the
  agent to *use* X for?". The answer to the latter is almost
  always a skill.
- The Anthropic SKILL.md format is now the de-facto storage shape
  for task definitions (their own Desktop scheduled tasks store
  prompts that way). Continuing to emit SKILL.md keeps All-Might
  inside the mainstream.
- The user's "cron + utils" intuition was exactly right; it lets
  us skip 90% of the design work that a wrong-framing
  conversation would have produced.
