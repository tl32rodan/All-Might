## 2. Panoramic snapshot — Ponytail

- **Pitch**: "He says nothing. He writes one line. It works." A
  plugin that constrains the agent's code-generation toward minimal,
  reuse-first solutions.
- **The Ladder (7 rungs, evaluated in order at write-time)**:
  1. Does it need to exist? (YAGNI)
  2. Already in codebase? (reuse)
  3. Standard library?
  4. Native platform feature?
  5. Installed dependency?
  6. One line?
  7. Minimum viable implementation
- **Safety carve-outs (prose-only today)** — *validation, error
  handling, security, accessibility* are exempt from minimisation.
  Issue #271 flagged that these live in prose and aren't enforced
  at runtime; a contributor offered to draft a PreToolUse-hook PR
  that *checks proposed edits against the carve-outs*.
- **Surfaces shipped**: `/ponytail [lite|full|ultra|off]`,
  `/ponytail-review` (diff), `/ponytail-audit` (full repo),
  `/ponytail-debt`, `/ponytail-gain`, `/ponytail-help`.
- **Install matrix**: plugin-installable on Claude Code / Codex /
  Devin / Pi; rule-file on Cursor / Windsurf / Cline; extension on
  Gemini / Antigravity; instruction-only fallback for Copilot CLI.
  Per-project *or* user-global config.
- **No memory, no persona/identity.** The "lazy senior dev" framing
  is *stylistic*; there's no data-bearing state beyond mode flags.
- **Benchmark claims**: 54% less code (agentic baseline, post-walk-
  back), 20% cheaper tokens, 27% faster, 100% safety vs 95% for a
  naive "YAGNI" prompt. `benchmarks/robustness-audit.js --selftest`
  validates the harness with known-good + known-lazy-wrong cases
  *before* spending API tokens.

## 3. Panoramic snapshot — agentic-stack

- **Pitch**: portable `.agent/` folder that preserves memory + skills
  when switching coding harnesses (Claude Code / Cursor / Windsurf /
  OpenCode / OpenClaw / Hermes / DIY Python).
- **`.agent/` layout**:
  - **Memory tiers** (by *retention*, not access): working,
    episodic, semantic, personal.
  - **9 seed skills**: skillforge, memory-manager, git-proxy,
    debug-investigator, deploy-checklist, design-md, data-layer,
    data-flywheel, tldraw.
  - **Protocols**: typed tool schemas, permissions enforcement,
    delegation contracts.
  - **Tools**: `graduate.py`, `reject.py` (human-in-the-loop
    candidate review), reflection / search / export CLIs.
  - **Harness conductor**: PostToolUse logging, permissions,
    failure handling.
- **Cross-harness via per-harness adapters** (`/adapters/<name>/`)
  with `adapter.json` manifest declaring file-merge policy.
  Each harness reads its own config (`CLAUDE.md`,
  `.cursor/rules/*.mdc`, `.github/instructions/`); the brain logic
  lives once in `.agent/`.
- **No persona/identity concept** — explicitly *single brain* per
  project. Per-user style lives in `PREFERENCES.md` (first-read each
  session) — the same surface All-Might's MEMORY.md callout fills.
- **`auto_dream.py` nightly clustering** — offline pattern-detection
  over journal/episodic memory; stages "candidates" that the host
  agent must `graduate.py`-with-rationale or `reject.py`. Explicit
  non-goal: no unattended graduation.
- **Explicit non-goals listed**: no unattended reasoning, no provider
  coupling, no secret export, no telemetry, no training without
  approval.

## 4. Four-axis comparison

### a) Design philosophy

| | All-Might | Ponytail | agentic-stack |
|---|---|---|---|
| Central thesis | **role × capability**, project-scoped, per-role memory | code-minimalism constraint via "lazy senior dev" archetype | portable **single-brain** memory across harnesses |
| Identity model | user-defined personalities (≥1 per project) | one stylistic archetype, not data-bearing | **none** — single brain |
| Anti-pattern enforcement | written rules in CLAUDE.md (developer-facing) | shipped as runtime hooks + skills (end-user-facing) | shipped as protocols + review CLI |
| Cross-harness | OpenCode + Claude Code | 10+ (incl. instruction-only) | many via per-harness adapters |
| Memory tiers | L1/L2/L3 by **access pattern** | none | working/episodic/semantic/personal by **retention** |

Convergences this pass:
1. **Multi-tier memory** is now confirmed across 4 frameworks:
   All-Might (L1/L2/L3), OpenViking (L0/L1/L2 per-directory),
   claude-mem (index/timeline/details), agentic-stack
   (working/episodic/semantic/personal). Three different axes
   (content / access pattern / chronology / retention) — none is
   universally right, all four solve different problems.
2. **First-read session-start file** is the common pattern:
   All-Might's `MEMORY.md` callout, agentic-stack's
   `PREFERENCES.md`, Serena's onboarding-memory. Same primitive,
   minor naming variation.

### c) Mechanism design — three genuinely distinct ideas

1. **Ponytail's per-write decision ladder is operationalised prose.**
   The 7-rung structure is just a structured prompt that the agent
   walks before emitting code. All-Might already uses the same
   *technique* internally — CLAUDE.md's "Discipline When Generating
   Third-Party Integrations" is a 6-rule structured prompt for the
   framework-dev agent. **Different audience**: ponytail aims at the
   end-user agent writing user code; All-Might's rules govern the
   framework-dev agent editing All-Might itself. No merger; see §5
   Q1/Q2.

2. **Ponytail's `--selftest` for paid-API benchmark harnesses.**
   The cross-audit issue #271 called this "the cleanest 'verify the
   instrument before any API spend' I've seen in the wild." A
   universal engineering pattern — but applies when a test harness
   makes paid calls. All-Might tests run locally on free CPU; the
   equivalent invariant is **`TestMirrorCoherence` +
   `TestHooksRunCleanly`** (already in place). Pattern absorbed via
   the lesson "check the instrument" — no code change.

3. **Agentic-stack's user-extensible adapter API
   (`/adapters/<name>/adapter.json`).** This is the structurally
   distinct mechanism this pass. All-Might's two-editor support is
   compiled into `core/claude_bridge.py`; agentic-stack lets the
   user *author* a new adapter declaratively. Trade-off:
   extensibility vs maintained-surface complexity. See §5 Q6.

### d) Implementation details

- **Auto-clustering on the write path comes back, with a damage
  mitigant attached.** Agentic-stack ships `auto_dream.py`
  (nightly clusters episodic memory) but explicitly *requires*
  `graduate.py`-with-rationale to accept any candidate. They added
  human-in-the-loop because the unattended version has the
  failure-mode tail All-Might flagged against OpenViking (2026-05-22
  §4d, #2132 / #2154). Their patch is "add a review step";
  All-Might's choice is "don't auto-cluster at all, fold pattern
  detection into `/remember`'s Pattern Check." Both work; All-Might's
  is simpler.
- **Agentic-stack's per-harness adapter pattern** decomposes by
  *tool*. All-Might's marker + `.allmight/templates/conflicts.yaml`
  + `/sync` decomposes by *file*. The All-Might design is more
  general — it works without knowing the harness list in advance —
  but agentic-stack's is more *extensible* by users without core
  changes. See §5 Q6.
- **Ponytail's carve-out gap is in their court.** They acknowledge
  via #271 that safety carve-outs (validation / error handling /
  security / a11y) live in prose and aren't runtime-checked. A
  contributor offered a PreToolUse-hook PR. All-Might doesn't ship
  user-code constraints, so the gap doesn't apply here.

## 6. Investment verdict

**No new feature clears the simplicity bar today.** The pass
delivers four validations of existing design and one deferred
candidate with a watch trigger:

1. **The "constraint-via-structured-prompt" technique** that
   All-Might uses internally (CLAUDE.md anti-patterns, Part-D
   non-negotiables) is independently validated by Ponytail's
   58k-star adoption of the same technique for end-user code.
   Different audience, same method.
2. **Multi-tier memory** is now a 4-project convergence
   (All-Might / OpenViking / claude-mem / agentic-stack). The axis
   each project picks (access / content / chronology / retention)
   is the design choice; none is universally right.
3. **Auto-clustering on the write path requires human review to be
   safe** — agentic-stack's `graduate.py` step proves the
   failure-tail All-Might cited against OpenViking is real. Our
   non-goal stands.
4. **Per-role memory isolation remains a moat** — now confirmed
   absent across **four consecutive direct comparables**
   (SuperClaude, Serena, OpenViking, claude-mem) **plus
   agentic-stack** (which is explicitly *single-brain* by design).
   That is five-in-a-row. Beyond two, this stops being noise and
   becomes a structural absence worth naming in README. See §9
   open follow-up.

## 7. Lessons captured for next reconnaissance

- **Stars-per-day high + outside-contributor PR offers with
  concrete code = real adoption, not SEO pump.** Ponytail (58k in
  14 days) triggered the SEO heuristic but failed the SEO
  hypothesis — issues #271 / #289 carry real engineering content.
  Add to playbook: triage star velocity *and* contributor-PR
  content quality before discarding.

- **"Different audience" is a clean rejection clause.** Ponytail
  and All-Might share a philosophy (anti-over-engineering as
  first-class) but apply it to disjoint audiences (end-user code vs
  framework code). When two projects look similar but target
  different audiences, the answer is composition, not merger.

- **N-in-a-row absence of a feature is structural, not noise.**
  Five frameworks in a row lacking per-role memory isolation
  (SuperClaude / Serena / OpenViking / claude-mem / agentic-stack)
  closes the "is this an oversight?" question. It is a deliberate
  structural choice that the All-Might design alone rejects. Name
  it in user-facing docs.

- **A competitor's mitigation patch is a non-goal validator.**
  Agentic-stack ships auto-clustering with a *mandatory* human
  review step. They wouldn't need the review step if the
  auto-cluster were safe. The mitigant is the bug report for the
  feature itself — and All-Might's "don't auto-cluster" non-goal
  gets the safety without the review-step infrastructure.
