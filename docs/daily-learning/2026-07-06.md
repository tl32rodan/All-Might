## 2. Verification of the three "Verify" items

### 2a. Claude Code v2.1.196 exact-match hook matchers — **not a hit**

The release note says wildcards are no longer implicit: `mcp__brave-search`
won't match sub-tools; `mcp__brave-search__.*` is required. That
targets **tool matchers inside `PreToolUse` / `PostToolUse` blocks.**

Grep of `src/allmight/` for `"matcher"`, `PreToolUse`, `PostToolUse`,
`Notification`, `mcp__`: **zero matches** across the entire tree.
`core/claude_bridge.py::_settings_payload` writes hooks under four
event names — `SessionStart`, `PreCompact`, `Stop`,
`UserPromptSubmit` — none of which use a matcher field. The
sub-tool matcher change cannot touch our scaffold.

Conclusion: no action. Documenting the null result so the next
pass doesn't re-open the question.

### 2b. Claude Code v2.1.198 Notification hook + subagents background — **also not a hit**

Two orthogonal changes:

- Notification hook types (`agent_needs_input`, `agent_completed`)
  are additive; nothing in All-Might listens for them today, and
  our existing `UserPromptSubmit` / `SessionStart` wiring keeps
  working.
- Subagents run in background by default. Per `CLAUDE.md` §
  *Editor Compatibility*, All-Might's per-personality subagent
  projection is **OpenCode-only**: `compose_role_agents` writes
  `.opencode/agents/<name>.md`; there is no Claude Code
  counterpart. The row explicitly says "Claude Code's subagent
  format differs; not mirrored yet." So Claude Code's
  background-by-default flip has no All-Might scaffold to
  regress.

Conclusion: no action. If the Claude Code subagent mirror ever
gets built (out-of-scope today), that work will have to consider
foreground vs background — but that's a decision for the mirror
PR, not a live regression.

### 2c. OpenCode v1.17.11 session snapshots — **overlap, but memory-history stays**

OpenCode now ships a built-in session-snapshot mechanism keyed to
messages, with revert of both DB state and worktree files. That
looks superficially like `.allmight/memory-history/.git`, which
already snapshots memory writes on every turn. Three reasons the
overlap is only skin-deep:

1. **Scope.** OpenCode's snapshots cover the entire worktree of
   the *session*. All-Might's mirror scopes to memory data
   (`MEMORY.md`, `personalities/<p>/{ROLE.md, memory/, database/config.yaml}`)
   — not project source. Reverting session state does not restore
   an accidentally-overwritten `understanding/stdcell.md`.
2. **Auditability.** Our mirror is a real git repo the user can
   `log`, `diff`, `restore` via `allmight memory log / restore`.
   OpenCode's is opaque to the user's git tools.
3. **Correctness.** Two open outside-contributor issues on
   `sst/opencode` — `Snapshot.nothrow() causes silent data loss
   on /undo` (#10589) and `Snapshot.track() silently swallows git
   add failures` (#12719) — say the built-in mechanism has known
   correctness bugs. Deferring to it would be regression via
   convenience.

Conclusion: keep memory-history as-is. Do not add a docs note yet
— the overlap is not user-facing (they see two different verbs:
OpenCode `/undo` vs. `allmight memory restore`), so raising it
in docs before a user asks is over-anticipation. Re-open only if
a user files an issue confused by the coexistence.

## 3. Design-philosophy read of the OpenCode session-snapshot ship

This is the second time in three passes an OpenCode release
has partially landed on a moat All-Might currently owns:

- 2026-06-29: opencode-mem shipped opaque-DB memory ↔ our
  markdown+git-mirror wins on recoverability.
- 2026-07-06: OpenCode built-in session snapshots ↔ our
  memory-history mirror stays because it's scoped, auditable,
  and correctness-tested.

The pattern is worth naming: **when upstream ships the same
verb with different scope**, the moat becomes the *scope choice*,
not the mechanism. Our mirror snapshots memory writes, not the
whole workspace; our vector search targets domain corpora, not
conversation logs. If a future pass finds an upstream mechanism
that matches *both* the verb and the scope of one of our
capabilities, that's the moment to actually re-evaluate — not
sooner.

## 6. Lessons captured for next reconnaissance

- **When upstream ships the same verb, look at the scope.**
  Both opencode-mem (last pass) and OpenCode session snapshots
  (this pass) partially overlap All-Might. Neither dislodges us
  because the scope is different (memory-only, per-role vs.
  whole-project). Future overlaps should be evaluated on the same
  scope axis before triggering a moat panic.
- **A null-result verification is worth writing down.** §2a's
  five-minute grep proved v2.1.196 doesn't touch us. If it hadn't
  been persisted, the next pass would re-open the same question.
  Null results carry as much learning value as positive ones,
  provided they get logged.
- **Additive-upstream weeks are the healthy case, not the boring
  one.** Two months into daily reconnaissance, three of the last
  four passes have found "nothing to build; here's what didn't
  break." That's exactly what a small stable framework should
  produce. The value of the routine is the *audit trail* — proof
  the ecosystem got scanned and nothing forced our hand.
- **Repo signal is thinning at this cadence.** Weekly scans are
  now finding mostly noise; the substantive delta lives in
  upstream release notes. Consider stretching to a 10-day cadence
  after one more pass to check whether the noise-to-signal ratio
  improves — no change today, just a watchpoint.
