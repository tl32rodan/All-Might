# Daily Learning Log — 2026-05-18

> Reconnaissance & benchmarking notes against contemporary AI-agent
> projects. The goal is to validate (not pioneer) trends, and to
> challenge All-Might's design only when there is a clear, concrete
> upside that does not violate the simplicity principle.

## 0. Method

1. Search GitHub for recent popular AI-agent / Claude-Code-adjacent
   repos.
2. Filter out junk: discount repos whose star count is implausible
   (e.g. created weeks ago, 100k+ stars, fork count rivaling star
   count, "single CLAUDE.md file" repos). Prefer signals of real
   developer demand — substantive issues, real PRs from outside
   contributors.
3. Pick 1–2 repos, build a panoramic understanding via README /
   CLAUDE.md / docs.
4. Compare against All-Might along **design philosophy → feature
   coverage → mechanism → implementation**, in that order.
5. Challenge All-Might; rate every candidate idea against the
   simplicity principle.
6. Persist findings to avoid re-doing the work tomorrow.

## 3. Four-axis comparison

### a) Design philosophy

| | All-Might | Superpowers | Anthropic Skills |
|---|---|---|---|
| Central abstraction | role (personality) × capability | workflow skill | one SKILL.md folder |
| Core metaphor | multi-role collaboration (chip team) | single agent + linearized workflow | dynamic, on-demand load |
| Triggering | conversation context + `Default personality` hint + plugin inject | bootstrap skill + automatic in-workflow trigger | model discovery from `description` |
| Complexity | high (two-tier + downward compose + bridge mirror) | medium (flat 14 skills + multi-IDE plugins) | low (one file + frontmatter) |

**Insight**: Superpowers' "Skills are not prose — they are code that
shapes agent behavior" is isomorphic to All-Might's stance: ROLE.md
and SKILL.md *look* like prose but are mechanically injected by
`role-load` / `memory-load` plugins. Both projects treat textual
content as behavioural code under the hood; only the invocation path
differs.

### b) Feature coverage

| | All-Might | Superpowers | Gap analysis |
|---|---|---|---|
| Memory tiers | L1 (MEMORY.md) / L2 (understanding TOC + on-demand) / L3 (journal + SMAK RAG) + auto snapshot to local git mirror | none public | **All-Might leads.** |
| Knowledge graph | SMAK + sidecar annotations | none | **All-Might leads.** |
| Workflow skills (TDD / debug / brainstorm / plan / review) | none | comprehensive | **All-Might absent, but out of scope** — chip-domain knowledge keeper, not coding agent. |
| Multi-IDE | OpenCode + Claude Code (mirror) | OpenCode + Claude + Codex + Cursor + Gemini | **Superpowers leads.** |
| Share / distribute | `/one-for-all` + `/all-for-one` bundle + git remote + PII review | none | **All-Might leads.** |
| Subagent | per-personality `@<name>` (one ROLE.md, two surfaces) | dispatcher pattern across subagents | Isomorphic, different framing. |

### c) Mechanism design

- **Plugin observability** (touch-file heartbeats at
  `.allmight/plugins/heartbeats/{oc,cc}/`, surfaced by
  `allmight plugin status`) — All-Might's design here is simpler
  than anything visible in Superpowers' `hooks/` directory.
  Zero-dependency, file-system-only telemetry. **Distinctive
  strength.**
- **Anthropic skills issue #556 ("claude -p never triggers
  skills/commands, 0% trigger rate")** corroborates All-Might's
  decision to use *explicit plugin inject* (`memory-load`,
  `role-load`) instead of relying on the model's `description`-based
  discovery alone. The official path is fragile in practice;
  All-Might's heavier path is justified.
- **Superpowers `using-superpowers` bootstrap skill** is a single
  explicit entry-point that re-primes the agent each session.
  All-Might's equivalent is split across `role-load.ts` +
  `memory-load.ts` + `reflection.ts`. Equivalent effect, heavier
  mechanism. No action required — the heaviness is a known item on
  `docs/plugin-observability.md`'s plugin-reduction plan.

### d) Implementation details

- Directory layout (`.opencode/`, `AGENTS.md`, `CLAUDE.md`,
  per-IDE plugin dirs) is isomorphic between All-Might and
  Superpowers. All-Might is on the mainstream path.
- Architectural discipline isomorphism:
  - Superpowers: "general-purpose core only, domain-specific
    features must be standalone plugins".
  - All-Might (CLAUDE.md): "`cli.py` is closed against templates",
    "`core/` is closed against templates", "a capability template
    owns its directory, nothing else".
- Anthropic skill contract (SKILL.md + 2-field frontmatter) is
  already satisfied by All-Might's
  `.opencode/skills/<name>/SKILL.md`.

## 5. Investment verdict

**No new feature is worth investing in today.** Reasoning:

1. All-Might *leads* the comparators on memory + knowledge
   management (L1/L2/L3 + auto-snapshot + PII-aware share workflow
   are unmatched by Superpowers and Anthropic skills).
2. Anthropic's discovery-based skill triggering is *less* reliable
   than All-Might's explicit-inject path (issue #556 evidence).
   All-Might should *not* downgrade to that model.
3. Areas where Superpowers leads (multi-IDE breadth, coding
   workflow skills) are out of All-Might's scope. Pursuing them
   dilutes focus.
4. Known internal problem (plugin count) already has a plan; no
   new proposal needed.

**Only lightweight follow-up**: verify alignment with the
[agents.md](https://agents.md/) cross-tool standard (Q4). Budget
~10 minutes next pass.

## 6. Lessons captured for next reconnaissance

- Discount stars; weight issues + outside-contributor PRs.
- When two projects converge on the same directory layout
  (`.opencode/`, `AGENTS.md`), that's the mainstream path —
  All-Might is on it.
- When official mechanisms have reliability bugs (#556), a heavier
  in-house mechanism (explicit plugin inject) is justified, not
  legacy debt.
- The simplicity principle is sharp: every "should we add X?"
  question must clear "does it solve a real, experienced problem,
  and is no existing mechanism close enough?" Most candidates
  fail the second clause.
