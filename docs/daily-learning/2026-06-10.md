## 2. Panoramic snapshot — agentskills.io open standard

- **Origin**: Anthropic released the SKILL.md format as an open
  standard in late 2025; published spec at `agentskills.io`.
- **Required frontmatter — only two fields**: `name` (lowercase +
  hyphens, must match parent directory) and `description` (what the
  skill does + when to use it; this string drives activation).
- **Body convention**: plain Markdown. **Progressive disclosure** —
  body holds the minimum the agent needs to decide to act; deeper
  references (other files, URLs) are read on demand.
- **Cross-tool compatibility rule**: "If your skill uses only `name`,
  `description`, and plain markdown instructions, it works
  everywhere; agent-specific fields are safely ignored by agents
  that don't support them."
- **Official adopters as of 2026-06**: Claude Code, **OpenCode**,
  Cursor, Codex CLI, Gemini CLI, GitHub Copilot (VS Code agent
  skills), Cline, Windsurf, plus 20+ community tools.
- **Discovery mechanism**: agent matches user intent against
  `description` text — the *what + when* sentence is the index. No
  registry, no embedding (at the SKILL.md layer; tools can layer
  their own discovery on top).

## 3. Panoramic snapshot — anthropics/claude-plugins-official

- **Pitch**: Anthropic-curated marketplace for Claude Code plugins
  with two trees: `/plugins` (Anthropic-built) and
  `/external_plugins` (vetted third-party).
- **Plugin layout**:
  ```
  plugin-name/
  ├── .claude-plugin/plugin.json   ← required manifest
  ├── .mcp.json                    ← optional MCP servers
  ├── commands/                    ← optional slash commands
  ├── agents/                      ← optional agent defs
  ├── skills/                      ← optional skills (SKILL.md format)
  └── README.md
  ```
- **`strict: false` mode** — a plugin can reference existing
  skill-only repositories without rewriting them to manifest shape:
  ```json
  {
    "name": "example-bundle",
    "skills": ["./skill-a", "./skill-b"],
    "strict": false,
    "source": {"source": "git-subdir", "url": "...", "path": "..."}
  }
  ```
  Skills register as `<plugin-name>:<skill-name>`.
- **Install path**: `/plugin install <name>@claude-plugins-official`.
- **Approval**: manual via submission form; quality + security
  review before inclusion in the directory. No automated
  sandboxing/permission documentation.
- **Activity**: 820 commits, 29.7k stars, 3.2k forks, 714 open
  issues, 40 open PRs. **Real-signal: high open-issue count is
  feature requests for specific plugins + format clarifications,
  not bots.**

## 4. Four-axis comparison

### a) Design philosophy

| | All-Might | agentskills.io | claude-plugins-official |
|---|---|---|---|
| Unit of distribution | **personality** (role) × capability | **skill** (single procedure) | **plugin** (bundle of skills + commands + agents + MCP) |
| Scope | project-local | per-skill, cross-tool | per-plugin, cross-project install |
| Discovery | conversation + `Default personality` hint | `description` matching | `/plugin install` from curated registry |
| State | per-role memory (L1/L2/L3) | **stateless** | **stateless** |
| Generation | scaffold a project | author content | distribute & install |
| Primary audience | one team building domain expertise (chip flows, EDA, …) | every agent tool | every Claude Code user |

**Three things this confirms about All-Might's bets:**

1. **SKILL.md is the right surface for skills.** All-Might's
   `core/skill_io.py` already writes `name` + `description` +
   optional `disable-model-invocation` + marker + body
   (`skill_io.py:34-50`). That **already complies** with the open
   standard's minimum and works on any compliant tool, including
   OpenCode.
2. **The `.opencode/skills/` → `.claude/skills` dir-symlink mirror
   is the right shape** for what the standard covers. One set of
   SKILL.md files on disk, two filenames, zero drift — exactly
   what the standard makes possible.
3. **Hooks/plugins remain hand-mirrored.** SKILL.md is *content*;
   the standard does *not* cover runtime lifecycle hooks (Stop,
   SessionStart, PostToolUse). All-Might's
   "Dual-platform invariant" (TS plugin + Python hook) in
   `CLAUDE.md` stays valid — the convergence is below the hook
   layer, not above it.

### b) Feature coverage

| Concern | All-Might | agentskills.io | claude-plugins-official |
|---|---|---|---|
| Per-role identity | ✓ (user-defined) | n/a (skills are global) | ✗ |
| Cross-tool skill portability | ✓ (symlink mirror) | ✓ (single SKILL.md) | ⚠ Claude Code primarily |
| Plugin install from registry | ✗ (own `share publish/pull` git path) | n/a (just the format) | ✓ (`/plugin install`) |
| Bundle multiple skills | ✓ (`/one-for-all` dir) | ✗ (one skill = one dir) | ✓ (plugin = bundle) |
| Per-role isolation | ✓ | n/a | ✗ (plugins are flat) |
| MCP wiring | ✓ (`opencode.json` + `.mcp.json`) | n/a | ✓ (`.mcp.json` inside plugin) |
| Approval / vetting | ✗ (peer-to-peer) | n/a | ✓ (manual review) |
| Domain-knowledge memory | ✓ (SMAK L3 + sidecars) | ✗ | ✗ |

### c) Mechanism design — three distinct ideas

1. **SKILL.md convergence on a 2-field frontmatter** is the lowest
   useful common denominator. All-Might already meets it. The risk
   is *additive drift*: if All-Might later adds custom frontmatter
   (e.g. a per-personality routing field), it must stay
   **agent-ignored** — i.e. agent-specific fields a non-compliant
   tool can drop silently. This matches the standard's "safely
   ignored" rule and is consistent with the existing CLAUDE.md
   anti-pattern against meta-cognition fields. No change needed
   *today*; flag as a watch item for any future field proposal.

2. **`strict: false` plugin manifest** lets existing skill repos be
   wrapped as plugins without restructure. If All-Might ever wants
   `/one-for-all` bundles to be installable as Claude Code plugins,
   `strict: false` is the cheap path: emit a single
   `.claude-plugin/plugin.json` that lists the bundle's skills,
   keep everything else identical. **Plausible, deferred** —
   see Q3 below.

3. **`/plugin install` from a curated registry** vs **`allmight
   share pull <git-url>`** are different distribution patterns,
   not competitors. The official directory targets generic,
   publicly useful plugins under manual approval. All-Might
   targets *domain-specific* personalities (e.g. `stdcell_owner`)
   that are neither publicly useful nor approval-appropriate.
   **They serve different audiences.** No need to merge.

### d) Implementation details

- **All-Might's SKILL.md is already cross-tool-portable.**
  `core/skill_io.py::write_skill_md` emits the 2-required-fields
  frontmatter + plain markdown body. A user with only Cursor or
  Codex CLI can drop the generated skill dir into the right place
  and it works. **Validate by README pointer, not by code change.**
- The Python-vs-TS plugin mirror cost remains real because the
  standard does not extend to lifecycle hooks. Heartbeats + the
  `PLUGIN_MANIFEST` single source of truth (CLAUDE.md
  *Dual-platform invariant*) is the right primitive for that layer
  and the standard does not displace it.
- **Memory-research repos (MAGMA / HiMeS / SimpleMem / Memoria)**
  are the *opposite* convergence — every team is rolling its own
  memory architecture. No standard, no shared format, no shared
  tool. That's a signal that memory is still in research mode and
  All-Might's L1/L2/L3 + per-role-isolation choice is fine for now.
  Reopen only when one of these crosses into the production-tool
  tier with real users.

## 8. Lessons captured for next reconnaissance

- **An open standard you already comply with is the highest-value
  ecosystem event.** It costs nothing (you wrote the right thing
  already) and gives a free cross-tool guarantee. Look for the
  README pointer first — code change second.
- **Standards typically cover *content*, not *runtime*.**
  agentskills.io covers SKILL.md; it does *not* cover lifecycle
  hooks. When weighing "can the standard let me delete the
  hand-mirrored layer?", check whether the standard's scope
  actually overlaps the layer you'd drop. Usually it doesn't.
- **A registry-curated install path (`/plugin install`) is not a
  competitor to a peer-to-peer share path (`git pull`).** They
  serve different cohorts (public reuse vs domain-specific
  transfer). Don't fold one into the other to feel "modern."
- **Research-stage memory architectures (multi-graph, hippocampus,
  semantic-compression) proliferating is itself a signal: the
  field hasn't settled.** Adopting one now would be backing a
  horse before the race. Hold the current per-role L1/L2/L3
  position; revisit when one of them becomes a tool with real
  users, not a paper with a GitHub repo.
