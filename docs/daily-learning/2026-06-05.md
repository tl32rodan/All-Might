## 2. Panoramic snapshot — `agentic-stack`

- **Pitch**: *"One brain, many harnesses. Portable `.agent/` folder
  (memory + skills + protocols) that plugs into Claude Code, Cursor,
  Windsurf, OpenCode, OpenClaw, Hermes, or DIY Python — and keeps its
  knowledge when you switch."*
- **`.agent/` layout (5 layers)**:
  1. **memory/** — four retention policies: `working` (session),
     `episodic` (event logs), `semantic` (graduated lessons in
     `lessons.jsonl`), `personal` (user prefs).
  2. **skills/** — progressive-disclosure manifest;
     `SKILL.md` files load only when triggers match.
  3. **protocols/** — typed tool schemas + `permissions.md`.
  4. **harness/** — conductor logic, `pre_tool_call.py`,
     `post_execution.py` hooks.
  5. **tools/** — CLI utilities (review, reflection, memory mgmt).
- **Cross-harness portability via "manifest-driven adapter"
  (v0.9.0+)**: each harness has an `adapters/<name>/` dir with an
  `adapter.json`; a shared Python backend reads `.agent/` and writes
  harness-specific config (`CLAUDE.md` + `.claude/settings.json`,
  `.cursor/rules/*.mdc`, `.windsurfrules`, `.github/instructions/`).
  A `transfer` wizard moves memory between harnesses.
- **Memory model = global, not role-isolated.** "The design assumes
  a single agent persona per project." Memory layering is by
  **recency + salience**, not by identity.
- **Review gate + retraction**: lessons require explicit host-agent
  approval with rationale; `retract_lesson.py` audits removals
  append-only.

## 3. Panoramic snapshot — `claude-memory-compiler`

- **Pitch**: "Give Claude Code a memory that evolves with your
  codebase. Hooks automatically capture sessions, the Claude Agent
  SDK extracts key decisions and lessons, and an LLM compiler
  organizes everything into structured, cross-referenced knowledge
  articles — inspired by Karpathy's LLM Knowledge Base architecture."
- **Pipeline**: conversation → `SessionEnd` hook (capture) +
  `PreCompact` hook (safety net) → daily logs → after 18:00 local
  `flush.py` runs → Claude Agent SDK compiles into markdown
  knowledge articles in `knowledge/` → `index.md` is the entry point.
- **Storage = markdown only.** No vectors, no SQLite. Rationale
  quoted directly: *"At personal scale (50–500 articles), the LLM
  reading a structured index outperforms vector similarity."*
- **Distillation cadence = batch, off-hot-path.** Daily logs
  accumulate; compilation triggers nightly (or on `uv run compile`).
  Per-turn writes do **not** invoke the Agent SDK.

## 4. Four-axis comparison

### a) Design philosophy

| | All-Might | agentic-stack | claude-memory-compiler |
|---|---|---|---|
| Central abstraction | **role** (user-defined personality) × capability | **harness-portable brain** | **distilled markdown KB** + capture hooks |
| Identity scope | per-personality (project, multi-persona) | single persona per project | single persona per project |
| Memory layering axis | **invocation strategy** (L1 always-loaded / L2 TOC / L3 RAG) | **retention policy** (working / episodic / semantic / personal) | **temporal stage** (daily log → compiled article) |
| Surface count | dual (OC + CC) | many (CC, Cursor, Windsurf, Copilot, OC, Hermes…) | single (CC) |
| Distillation | **non-goal until needed**; CLI shape pre-described in `CLAUDE.md` | review-gated lesson promotion (sync, per-write) | **batch, off-hot-path, Agent-SDK-driven** |

**Two convergences keep accumulating:**
1. **Markdown-not-database** for human-readable memory: Serena
   (2026-05-25), agentic-stack semantic+personal tiers, and
   claude-memory-compiler all explicit. claude-mem / engram
   (SQLite+FTS5+vector) are now the clear outliers.
2. **Per-role isolation is structurally rare.** agentic-stack
   *literally states* "single agent persona per project" — same
   structural ceiling as SuperClaude+Serena (2026-05-25 §4d).
   All-Might's per-role memory isolation continues to be a property
   that *only the vertically-integrated design* expresses.

### c) Mechanism design — the two genuinely distinct ideas

1. **Manifest-driven adapter (agentic-stack) vs hand-coded dual
   bridge (All-Might).** All-Might's `claude_bridge.py` plus
   per-capability hook content hard-codes "OC = source of truth, CC
   = mirror" with a *single* Python generator per shared string
   (`PLUGIN_MANIFEST` + `_reminder_nudge_text` etc.). agentic-stack
   abstracts the bridge: every new harness is an `adapter.json` +
   small Python adapter. The price: a config-driven indirection
   that has to handle every harness's quirks (Cursor's `.mdc`
   format, Windsurf's plain-text rules, Copilot's instructions
   dir). The win: adding harness #6 is a directory drop.
   For All-Might, harness #3 is **not on any user's roadmap**
   (no issue, no PR, no inbound request) and Cursor/Windsurf
   handle the same "AI coding agent in a project" niche
   redundantly with OC+CC, not complementarily. See §5 Q1.

2. **claude-memory-compiler's nightly Agent-SDK distillation is
   *exactly* the shape `CLAUDE.md` non-goal predicted.** Quoting
   the locked decision verbatim: *"No `/distill` slash command.
   Incremental pattern detection is folded into `/remember`'s
   Pattern Check step. Batch distillation, if ever needed, becomes
   a CLI (`allmight memory distill`) that calls the Claude API
   directly — out of the agent's context window entirely."*
   claude-memory-compiler implements:
   - SessionEnd + PreCompact hooks → daily logs (capture, no
     model call) ← **All-Might's `memory-history.ts` Stop hook is
     identical in spirit, but only mirrors files; it does not
     queue for distillation.**
   - Nightly `flush.py` → Agent-SDK compile → markdown KB ←
     **this is `allmight memory distill` in another wrapper.**
   - `index.md` → entry point ← **this is L2's `_index.md`.**

   The convergence is striking: the same architecture is being
   independently re-derived. The trigger condition CLAUDE.md set
   ("if ever needed") is what to scrutinise — see §5 Q2.

### d) Implementation details

- **agentic-stack's `lessons.jsonl` semantic tier is an
  interesting analogue to All-Might L3 — but with a different
  governance.** L3 journals append on `/remember`; agentic-stack
  promotes a candidate lesson into `lessons.jsonl` only on
  explicit review with rationale. The review-gate eats friction;
  All-Might's user-triggered `/remember` already gates writes by
  human invocation, so the additional rationale field would be
  *meta-cognition tax* (style-guide anti-pattern).
- **claude-memory-compiler's "no vectors below 500 articles"
  threshold matches All-Might's own `docs/plan.md` threshold for L2
  (~200 files).** Independent confirmation that markdown+index
  outperforms RAG at personal scale — direct evidence supporting
  the locked "No L2 RAG until L2 demonstrably crosses ~200 files"
  non-goal.

## 7. Feature rating — the one candidate worth rating (Q2)

The candidate `CLAUDE.md` itself anticipated: `allmight memory distill`
as a batch CLI off the agent's context window.

| Dimension | Assessment |
|---|---|
| Problem reality | **Unproven within All-Might usage.** claude-memory-compiler proves the *shape* is reasonable. No All-Might user has reported L2 reorganisation pain, no L2 has crossed the threshold the *Non-goals* section identifies (~200 files / personality). Pattern Check inside `/remember` covers per-write detection; `/reflect` covers periodic audit. The unmet need is *L2 reorganisation* (consolidating overlapping understanding files), not *L3 distillation* (that's what RAG handles). |
| Simplicity cost | **Moderate.** A CLI subcommand calling the Anthropic API directly is well-contained (`cli.py` would need a new subcommand group, but `cli.py is closed against templates` is preserved if it lives under `allmight memory distill` and shells through `core/`). The real risk: incoming proposers will want to fold this back into a `/distill` slash command, re-litigating the 2026-05-25 anti-pattern decision. |
| Upside if real | High once the trigger condition fires. Distillation off-hot-path is genuinely the only sane way to reorganise dozens of L2 files. |
| Reversibility | Good — additive CLI subcommand; users opt in. |
| **Verdict** | **Defer, do not build now.** Rating **2/5** for investment — same as 2026-05-25 Q3. Re-open on the empirical trigger: any user reports L2 ≥ 50 files in a single personality, OR `/reflect` audits start systematically suggesting consolidation across files. Until then, `/remember`'s Pattern Check + `/reflect`'s periodic audit is the simpler mechanism and wins. |

## 8. Lessons captured for next reconnaissance

- **The non-goal section is the most predictive doc in `CLAUDE.md`.**
  Two passes in a row (2026-05-25 markdown convergence,
  2026-06-05 distillation CLI shape) — locked decisions are
  *predicting* the ecosystem, not reacting to it. Treat the
  non-goal list as a *forward-looking design map*, not a stop-sign
  list. The trigger conditions written into each non-goal
  ("until L2 crosses ~200 files", "if ever needed") are the
  empirical bar; external prior art existing is *not* one of them.
- **Adapter frameworks pay rent.** agentic-stack's manifest-driven
  adapter system is elegant *only* because it ships ≥4 harnesses.
  At 2 surfaces (OC+CC), the hand-coded bridge with a single
  Python generator per shared string (current All-Might design)
  beats it on debuggability and test-suite coverage. Generalisation
  cost is proportional to *how many actual instances* exist, not to
  *how many are conceivable*. CLAUDE.md's *"When in doubt: add a
  flag, not a capability"* generalises: *"add a hard-coded mirror,
  not an adapter framework."*
- **Renames are still silent doc-rot — but mature renames bridge.**
  OMO's "dual-published, both names recognised, legacy loads with
  warning" pattern is the gold-standard rename behaviour from a
  downstream-dependency-doc perspective. When All-Might has its own
  renames (the `corpus_keeper`/`memory_keeper` → `database`/`memory`
  rename was clean-break), consider whether dual-recognition is
  worth the small amount of extra code on a future analogous change.
