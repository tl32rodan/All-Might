## 2. Panoramic snapshot — agentic-stack

- **Pitch**: "One brain, many harnesses." A portable `.agent/`
  directory that travels with the project; per-harness adapters
  re-emit it as `.claude/`, `.cursor/`, `.github/instructions/`,
  etc.
- **On-disk layout**:
  ```
  .agent/
  ├── AGENTS.md
  ├── harness/hooks/
  ├── memory/                     ← 4 layers
  ├── skills/{SKILL.md, _manifest.jsonl}
  ├── protocols/{permissions.md, tool schemas}
  └── tools/{graduate.py, reject.py, learn.py, ...}
  adapters/<harness>/             ← per-harness re-emit
  ```
- **Memory model — 4 layers, all local files**:
  - **working** — session-scoped, ephemeral.
  - **episodic** — timestamped `.jsonl` event logs.
  - **semantic** — `lessons.jsonl`, lessons promoted from episodic.
  - **personal** — `PREFERENCES.md` (user prefs).
  - Retrieval: "salience × relevance" over local files. **No
    external vector DB.**
  - Memory history is just `git log .agent/memory/` — single git
    repo, in-project.
- **Multi-role isolation**: **none.** One identity per `.agent/`
  folder. The model is a single brain.
- **Cross-harness sync mechanism**: **adapter-based copy/mirror**,
  not symlinks. Each adapter has `adapter.json` declaring which
  `.agent/` files map to which harness path. The `install.py`
  applies merge policies with ownership tracking to prevent
  orphans.
- **Memory graduation**: `auto_dream.py` is a **scheduled**
  nightly cycle that clusters episodic events into candidate
  lessons; `graduate.py` / `reject.py` are the human-in-the-loop
  CLIs that promote / drop a candidate into `lessons.jsonl`.
- **Transfer wizard** (`agentic-stack transfer`): export with
  staging plan → SHA-256 bundle → curl command for transport;
  import verifies digest and merges. Scoped exports ("move my
  memory", "move my preferences and lessons").
- **Explicit non-goals**: no model training on artifacts, no
  unattended reasoning (staging is "mechanical file operations"
  with required human review), no cloud sync, **no role/persona
  isolation**, no automatic harness detection.

## 3. Panoramic snapshot — letta-code

- **Pitch**: "Memory-first coding agent. Designed for long-lived
  agents that can learn from experience and maintain a cohesive
  identity across models."
- **Memory storage**: **git repos** at
  `~/.letta/lc-local-backend/memfs/<agent-id>/memory`. All context
  (including memory blocks) tracked via git; can be `git push`-ed
  to a user-supplied GitHub repo via `/memory-repository set`.
  Scope is **per-agent**, not per-project, not per-role.
- **Personality / role / persona abstraction**: **none.** Agents
  are identity-agnostic; no role-switching mechanism.
- **Per-role memory isolation**: not addressed.
- **Update mechanism**: three paths — explicit (`/palace`,
  `/doctor`), autonomous (the agent "programmatically rewrites its
  context"), and hooks.
- **Integration with Claude Code / OpenCode / Cursor**: **none.**
  Letta Code is a **replacement CLI** (`letta`), with its own
  Desktop app + browser at `chat.letta.com` + messenger
  integrations. It does not plug into Claude Code or OpenCode.
- **What a user would lose by replacing All-Might + a host agent
  with letta-code**: per-role memory isolation, the host agent
  itself (Claude Code's tools, plan-mode, Bash sandboxing,
  ecosystem of MCP servers), per-project scoping. Effectively
  they'd be switching vendors, not adopting a complementary tool.

## 4. Panoramic snapshot — iwe

- **Pitch**: "Markdown knowledge graph for you and your AI agent.
  No cloud, no database, no lock-in."
- **On-disk layout**: plain `.md` files in a workspace
  (`iwe init`). Two link types: **nesting** (hierarchy) and
  **cross-references** (web).
- **LSP** (`iwes` server) — does what a standard markdown LSP
  doesn't: graph-aware find-references, backlinks, link-update on
  rename, extract/inline-by-graph, parent-reference inlay hints.
- **Agent API**: dual surface — `iwe` CLI (`find`, `retrieve`,
  `tree`, `new`, `extract`, `inline`, `rename`, `delete`) and
  `iwec` **MCP server** with the same operations. Watches files
  for live updates.
- **Per-role / persona abstraction**: **none.**
- **Explicit non-goals**: no built-in AI, no cloud, not a
  full-featured PKM (no daily-notes, etc.).

### a) Design philosophy

| | All-Might | agentic-stack | letta-code | iwe |
|---|---|---|---|---|
| Central abstraction | **role × capability** (user-defined, per-project) | **one brain** (per-project, single identity) | **per-agent identity** (long-lived agent) | markdown graph (no identity layer) |

| Memory store | markdown/YAML + per-project git mirror | markdown/JSONL + in-project git | git repos in `~/.letta/...` | markdown + LSP graph |
| Memory promotion | agent-triggered `/reflect` | **scheduled** `auto_dream.py` + human-in-the-loop `graduate.py`/`reject.py` | autonomous (agent rewrites context) | n/a |

**Three convergences worth logging:**

1. **Portable bundle across harnesses is mainstream now.**
   agentic-stack independently re-derived (a) one-brain-many-harnesses,
   (b) git-backed memory history, (c) markdown-not-database, (d)
   human-in-the-loop memory hygiene. Four All-Might bets, four
   independent re-derivations. The 2026-05-25 pass logged two
   (markdown memory via Serena; persona via SuperClaude); add two
   more today (portable bundle + git history). Convergence is now
   overwhelming.
2. **letta-code's per-agent isolation is still not per-role.**
   Same gap as SuperClaude+Serena (2026-05-25 §4d). The "persona
   framework + memory MCP" decomposition still cannot express
   per-role memory isolation. The vertical-integration moat
   continues to widen.
3. **iwe's MCP-only integration surface vs All-Might's
   skills/commands.** iwe ships only an MCP server; the agent
   discovers operations through MCP tool descriptions. All-Might
   ships slash commands + skill bodies. **iwe's choice scales to
   any MCP-capable host; All-Might's choice is more explicit and
   token-cheap per-invocation but requires per-harness emission.**
   Both are valid; neither is dominant yet.

### c) Mechanism design — the genuinely novel ideas

Three from this pass; one of them has a non-zero case.

1. **agentic-stack's `auto_dream.py` scheduled memory-graduation
   cycle.** Runs nightly via cron, clusters episodic events into
   candidate lessons. The output is *staged*, not committed —
   human runs `graduate.py <id>` or `reject.py <id>` later. This
   decouples *detection* (machine, off-hours, no token cost during
   work) from *acceptance* (human, fast). All-Might's `/reflect`
   does both at once during the agent's turn. **See §7 Q1.**

2. **agentic-stack's adapter-mirror vs All-Might's dir symlink.**
   adapter-mirror translates per harness (`.claude/settings.json`
   vs `.cursor/rules/*.mdc` vs `.github/instructions/*`); All-Might
   currently mirrors only Claude Code and OpenCode. **Implication:
   if All-Might ever wants Cursor / Copilot CLI / Windsurf parity,
   the question is whether to add per-harness translators (the
   agentic-stack way) or stop at "the two editors that share a
   convention".** This is a scope question, not a mechanism
   question. See §7 Q2.

3. **iwe's MCP-only surface.** Single MCP server gives every
   MCP-capable host instant access. All-Might's skills+commands are
   more explicit and per-invocation cheaper but require
   per-harness emission. If MCP becomes ubiquitous across CLIs (it
   already is on Claude Desktop / Cursor / Windsurf / Claude Code /
   OpenCode), an MCP surface for All-Might *might* eventually
   replace the dual emission. **Not now** — the dir-symlink
   approach is shipping and works. See §7 Q3.

### d) Implementation details

- **agentic-stack's adapter manifests vs All-Might's
  `PLUGIN_MANIFEST`.** Both centralise per-target mapping in a
  single declarative structure. All-Might's manifest also
  enforces "every plugin in `requires:` is a real capability" via
  test coverage; the agentic-stack manifest equivalent is less
  visible from the README. **Validation only.**
- **agentic-stack uses in-project `.git`** for memory history.
  All-Might uses a **separate `.git`** at
  `.allmight/memory-history/.git` precisely so the project's main
  `git status` doesn't see memory churn. **All-Might's choice is
  better** — same argument we made when the user originally rejected
  intertwining. Log the convergence-with-difference: agentic-stack
  hit the simpler shape, All-Might already iterated past it.

## 8. Investment verdict

**No new feature clears the simplicity bar today.**

The closest architectural competitor (agentic-stack, 2k★) is
**convergent on four core All-Might bets** — portable bundle,
git-backed memory history, markdown-not-database, human-in-the-loop
graduation — and **diverges on exactly the property that is
All-Might's moat**: per-role memory isolation. The 2026-05-25 pass
identified persona-framework + memory-MCP as the decomposition that
can't express per-role isolation; this pass adds *one-brain-per-project*
to the list of structurally-can't-express variants. Three independent
projects have now picked a memory architecture that lacks per-role
isolation. That is the differentiated property; protect it.

Claude Code's W19–W22 ships **distribution and observability
upgrades** (zip plugins, `/usage`, `/reload-skills`, dynamic
workflows) that are mostly orthogonal to All-Might's architecture.
The one item that *could* matter is whether `.claude/skills/`
auto-load follows symlinks — that's a verification, not a build.

## 10. Lessons captured for next reconnaissance

- **Convergence count is now the headline metric.** Two passes ago
  the framing was "two independent projects validate the markdown
  memory choice." This pass adds two more (portable bundle, git
  history). When N ≥ 4 independent projects re-derive your
  contrarian bets and diverge on the *same* missing property, that
  property is your moat. Stop re-validating it; protect it.
- **The "two-axis grid" reads cleanly now**: convergent on
  *substrate* (markdown + git + bundle), divergent on *identity
  layer* (per-role vs per-brain vs per-agent vs none). Future
  passes should triage candidates on the identity-layer axis first
  — that's where genuine differentiation lives.
- **Anthropic's platform release cadence (weekly) means a "did
  Anthropic ship anything that breaks us" pass is cheap insurance.**
  Today's `/whats-new` read took ~5 min and surfaced the
  `.claude/skills/` auto-load question (Q4) that could otherwise
  bit-rot silently for weeks. Worth doing every pass.
- **Star-farm pattern is stable enough to template.** Recurring
  signals: created < 8 weeks ago + ≥ 50k stars + fork count > 1/8
  of stars + topic list contains every keyword + "claude-code +
  cursor + codex + copilot" in description. Future passes can skip
  these in seconds.
