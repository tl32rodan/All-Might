## 2. Panoramic snapshot ‚Äî CodeGraph

- **Pitch**: "Pre-indexed code knowledge graph for [every harness] ‚Äî
  fewer tokens, fewer tool calls, 100% local."
- **Mechanism**: tree-sitter grammars per language (17 supported:
  TS/JS, Python, Go, Rust, Java, Kotlin, C/C++, C#, Swift, PHP, Ruby,
  Objective-C, Lua/Luau, Svelte, Vue, Liquid, Delphi) ‚Üí per-language
  extractors emit nodes (symbols, routes, plugins) + edges (calls,
  references, decorates) ‚Üí SQLite-backed graph ‚Üí MCP server exposing
  `codegraph_explore`, `codegraph_search`, `codegraph_trace`.
- **Framework-aware**: Spring (`@RestController`, `@FeignClient`),
  MyBatis (XML mapper ‚Üî Java DAO), Drupal (procedural + OOP hooks +
  plugin attributes + Symfony event subscribers), Vue/Svelte
  templates. *Static dispatch only* ‚Äî annotations / configuration
  files / decorators that the host LSP can't resolve cross-language.
- **Install flow** (`codegraph install -t claude -l local`): writes
  three files into the project ‚Äî `.mcp.json` (MCP entry),
  `.claude/settings.json` (permissions if `--auto-allow`), and
  `.claude/CLAUDE.md` (instructions block). Issue #243's critique:
  two of the three should default to the gitignored *personal*
  variant (`.claude/settings.local.json`, `CLAUDE.local.md`); they
  currently default to team-shared and silently co-mingle a
  *personal trust grant* with the team's hook config.
- **Distinct from Serena**: Serena's LSP is *online* (queries the
  language server at call time); CodeGraph is *offline* (pre-builds
  the graph once, queries SQLite). And Serena's LSP can't see
  framework-conventions (Feign methods ‚Üí Spring routes, MyBatis
  statement IDs ‚Üí XML), which is exactly the gap CodeGraph fills.
- **Distinct from SMAK**: pure structural (edge = "X calls Y", not
  "X is semantically about Y"). No embeddings, no vectors.
- **Issue traffic shape**: framework-extractor extensions (#300
  Drupal, #489 MyBatis, #490 Feign), language-support requests
  (#648 ‚Äî 17 supported, 22 requested), env-specific bugs (#208 NTFS,
  #535 opencode config path on Windows), monorepo limitations (#514).
  An architectural proposal (#687) for opt-in **LLM-assisted dynamic
  dispatch edge synthesis** for the long tail of patterns regex
  extractors can't catch ‚Äî gated behind env vars, deterministic
  output unchanged when unset.

## 3. Panoramic snapshot ‚Äî AgentMemory

- **Pitch**: "#1 Persistent memory for AI coding agents based on
  real-world benchmarks."
- **Mechanism**: A Node.js daemon (`iii` engine on port 3111) +
  KV store + durable trigger system + REST API + Electron viewer UI
  (port 3113) + Claude Code plugin shipping `session-start.mjs` /
  `session-end.mjs` hooks. Each user observation flows through
  `mem::compress` ‚Üí `mem::summarize` ‚Üí optional `mem::graph-extract`
  ‚Üí optional `mem::reflect` / `mem::consolidate`.
- **Pipeline complexity**: knowledge-graph extraction (LLM call to
  pull entity/relation triples from compressed observations),
  consolidation, semantic decay, "crystals" (curated highlights),
  "lessons", flags-driven feature gates (`GRAPH_EXTRACTION_ENABLED`,
  `CONSOLIDATION_ENABLED`, `AGENTMEMORY_AUTO_COMPRESS`,
  `AGENTMEMORY_INJECT_CONTEXT`).
- **Operational shape**: the engine is foreground-only by default ‚Äî
  no shipped LaunchAgent / systemd unit / Windows service. After
  reboot or terminal close, the engine dies and MCP calls fail
  silently until the user re-runs `agentmemory`. Issue #524 (9
  reactions) ships a hand-rolled macOS LaunchAgent and asks the
  project to absorb the OS-service wrapping.
- **Failure mode that matters here (#600)**: graph-extract was
  *never invoked automatically* ‚Äî `event::session::stopped`
  subscribes to topic `agentmemory.session.stopped`, but **nothing
  in the codebase publishes to that topic**. The user discovered
  this by grepping the repo after a dashboard showed `Graph: 0
  nodes` for two weeks despite 9,977 successful `mem::compress`
  calls. The extractor itself worked when invoked via curl; only
  the wiring was broken. The repo's existing `functionMetrics`
  telemetry showed `mem::summarize` and `mem::compress` but
  *omitted* `mem::graph-extract` ‚Äî no signal because no fire.

## 4. Four-axis comparison

### a) Design philosophy

| | All-Might | CodeGraph | AgentMemory |
|---|---|---|---|
| Substrate | files + per-turn hooks | SQLite graph + MCP server | Node daemon + REST + KV + UI |
| Lifecycle | scaffold once; hooks fire per turn | re-index on push (`-f`) or watch | long-running daemon |
| State | markdown / YAML in tree | binary SQLite, project-scoped | KV store on disk, user-scoped |
| Human-readable | yes (markdown / git diff) | no (graph DB, view via tools) | partial (markdown export only) |
| Failure-recovery | git mirror of memory tree | rebuild from source | restore from backup (#523 wants this built in) |
| Cross-harness | `.opencode/` + `.claude/` mirrors | MCP server (every harness) | MCP + per-harness plugins |

**Two convergences vindicating All-Might from different angles:**

1. **The "daemon + MCP server" stack accumulates installer pain.**
   AgentMemory's most-reacted feature request (#524) is "ship me
   an OS service installer so the engine doesn't die on reboot."
   CodeGraph's #243 is "your installer writes personal trust grants
   into team-shared files." Both repos are paying ongoing cost for
   *process lifecycle* and *config-scope* that All-Might's
   files-and-hooks design doesn't have. Same shape as the prior
   pass's claude-mem(SQLite) vs Serena/All-Might (markdown)
   convergence.

2. **Telemetry without "did-it-fire" is opaque.** AgentMemory has
   rich `functionMetrics`, a viewer UI, REST `/health`, and STILL
   #600 took a user two weeks of grep to diagnose because the
   metrics only count *invocations*, not *expected-but-missing
   fires*. All-Might's heartbeat design (`docs/plugin-observability.md`)
   makes this explicit: every plugin/hook touches a marker on fire;
   `allmight plugin status` shows "never fired" as a flag. This is
   the second independent project (after `oh-my-opencode`'s
   `claude-code-hooks` debacle) where the missing-fire signal would
   have shortened diagnosis by weeks. The heartbeat decision is
   strongly vindicated.

### c) Mechanism design ‚Äî three genuinely distinct ideas

1. **CodeGraph extends the "structural code retrieval" frontier
   past where Serena's LSP stops.** Serena (LSP) handles
   `find_references` *within* a project but cannot follow
   `@FeignClient`-declared HTTP boundaries or MyBatis statement-ID
   strings across modules ‚Äî those resolve only at runtime via
   configuration. CodeGraph adds *framework-aware extractors*
   that resolve them statically. This is a third point in the
   code-retrieval design space: **LSP (Serena) vs vector-semantic
   (SMAK) vs structural-with-framework-conventions (CodeGraph)**.
   The three answer different questions and none replaces the
   others.

2. **CodeGraph's #687 LLM-assisted edge synthesis as a design
   pattern.** The proposal: an **opt-in** heuristic backend for
   unresolved dynamic-dispatch sites (computed member calls,
   `getattr`, decorator registration, string-keyed dispatch).
   Gated behind `CODEGRAPH_HEURISTIC_RESOLUTION=1`; deterministic
   output unchanged when unset; cached per file; conservative
   confidence threshold; output edges carry `provenance: 'heuristic'`.
   This is a clean *escape hatch* design for "the rule-based
   resolver can't scale to every framework, but the long tail is
   addressable via an LLM call you opt into". Worth filing under
   "patterns" for the day an All-Might capability has the same
   shape (e.g., if `/distill` ever ships as a CLI, this gating
   pattern is the reference).

3. **AgentMemory's automatic graph extraction over observations
   is a tempting feature with a costly tail.** The idea: at every
   session end, LLM-extract entity/relation triples from
   compressed observations into a knowledge graph; serve queries
   over the graph. The cost: an entire trigger-and-flag pipeline
   (`GRAPH_EXTRACTION_ENABLED`, `CONSOLIDATION_ENABLED`, durable
   subscribers, hot-path LLM calls, a viewer to visualise it) ‚Äî
   and when ONE topic publication is missing, the whole thing
   silently no-ops (#600). For All-Might this falls under the
   already-locked non-goal: "No SQLite memory store ‚Ä¶ YAML +
   git mirror keeps memory human-readable" and "auto-capture on
   the hot path is the same write-path failure-tail class flagged
   for OpenViking (2026-05-22)." `/remember`'s explicit Pattern
   Check + `/reflect`'s periodic audit are the equivalent intent
   without the failure surface.

### d) Implementation details

- **The "personal trust vs team policy" axis (CodeGraph #243) is
  worth auditing against All-Might's emitted files.** Quick audit:
  - `.claude/settings.json` ‚Äî written by `_write_settings_json`
    (`core/claude_bridge.py:357`). Contains hook registrations
    only (`SessionStart` / `PreCompact` / `Stop` /
    `UserPromptSubmit`). These reference `.claude/hooks/*.py`
    scripts that are also in-tree, so any cloner's session reaches
    a working file. The hooks themselves no-op if their state
    files don't exist. **Team-shared is the correct default for
    All-Might's case** ‚Äî every teammate using All-Might needs the
    hooks; this isn't a personal trust grant.
  - `.mcp.json` ‚Äî All-Might doesn't write one (no MCP server).
  - `.claude/CLAUDE.md` ‚Äî All-Might writes a *root* `CLAUDE.md`
    shim (`@`-import to `AGENTS.md` + `MEMORY.md`), not
    `.claude/CLAUDE.md`. The shim *is* team-shared by intent.
  - **Verdict: no action.** All-Might's writes are correctly
    team-shared. But the framing is useful ‚Äî if a future
    capability emits a *user-trust* artifact (e.g., an
    auto-allow permission), it must route to `.local.json`. Log
    the rule.

- **AgentMemory's #524 hand-rolled LaunchAgent** is the kind of
  user-pull All-Might avoids by not running a long-lived process.
  Whenever the temptation arises to make a memory-layer "smarter"
  by moving work off the hot path into a background daemon
  (e.g., async embedding via a sidecar), this issue is the
  cautionary tale: you inherit the entire "ship me
  systemd/launchd/Windows-service installers" backlog.

## 8. Lessons captured for next reconnaissance

- **"Wide-net SEO description" is not the same as "SEO-farmed
  project".** CodeGraph and AgentMemory both stuff every harness
  name into their topics, but the issue traffic is the tell ‚Äî
  outside contributors filing framework-extractor PRs with 54
  unit tests (CodeGraph #300), and users posting 200-line
  root-cause analyses with code references (AgentMemory #600).
  When the description looks SEO but the issues look real, trust
  the issues.

- **Telemetry that counts *invocations* misses *missing
  invocations*.** AgentMemory has invocation metrics, REST
  `/health`, and a viewer UI; none of them surfaced the broken
  `mem::graph-extract` trigger for two weeks. All-Might's
  heartbeats explicitly catch "never fired"; the file's *presence*
  is the signal, not its content. Re-state this in
  `docs/plugin-observability.md` on next edit: invocation metrics
  ‚âÝ liveness signals.

- **"Personal trust vs team policy" is a useful axis to audit
  emitted files against.** Especially for any new artifact that
  has a "trust the executable" or "trust the API key" component.
  Add it to the CLAUDE.md invariant list (see Q6 action).

- **The "structural / semantic / LSP" trichotomy keeps holding.**
  Each new "code retrieval" entrant lands on one of the three
  axes. Knowing the trichotomy in advance lets new entrants get
  dispositioned in one sentence (CodeGraph: structural + framework
  conventions ‚Üí complementary; Serena: LSP-symbolic ‚Üí
  complementary; SMAK: vector-semantic ‚Üí All-Might's lane).
